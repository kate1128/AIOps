# 竞品分析规范

> 定义竞品选择、分析维度、执行周期和输出格式，确保竞品分析系统化、可沉淀。

---

## 一、竞品选择标准

### 竞品分层

| 层级 | 定义 | 监控频率 | 当前代表产品 |
|------|------|---------|------------|
| **核心竞品** | 目标客户群高度重叠，直接争夺同一市场 | 月度深度 + 实时动态 | 待填写 |
| **参考竞品** | 部分功能重叠，模式可借鉴 | 季度 | 待填写 |
| **行业标杆** | 国际领先产品，代表技术/产品最高水位 | 半年 | 待填写 |

### 纳入/剔除原则

- 纳入：目标客户是同类企业、核心功能覆盖 AI 知识问答/文档处理、有公开定价或公开 Release Notes
- 剔除：产品已停止更新超过 6 个月、目标市场完全不同（如纯消费级 vs 纯企业级）

## 二、分析维度

### 2.1 产品功能维度

| 维度 | 分析要点 |
|------|----------|
| **核心功能** | 知识库管理、问答能力、文档处理、多模态支持 |
| **AI 能力** | 使用的模型、RAG 策略、Agent 能力 |
| **集成与 API** | 是否有开放 API、与第三方系统集成能力 |
| **权限与安全** | RBAC 粒度、数据隔离方式、合规支持 |
| **私有化** | 是否支持私有部署、部署复杂度、离线能力 |
| **用户体验** | 响应速度、UI 设计质量、移动端支持 |

### 2.2 商业维度

| 维度 | 分析要点 |
|------|----------|
| **定价模型** | 按用户数 / 按调用量 / 订阅制 / 一次性 |
| **客户案例** | 典型客户行业、规模、实际使用深度 |
| **销售模式** | 自助购买 / 直销 / 渠道合作 |
| **融资/背景** | 融资轮次、背后投资方、团队背景 |

### 2.3 技术维度

| 维度 | 分析要点 |
|------|----------|
| **技术栈** | 可从 GitHub、招聘 JD、技术博客推断 |
| **性能指标** | 响应延迟、并发能力（公开 Benchmark 或测试）|
| **安全合规** | SOC 2、ISO 27001、等保认证 |
| **模型支持** | 支持的 LLM 清单，是否支持私有模型 |

## 三、分析执行流程

### 日常监控（持续）

```
信息来源：
├── 官网 Blog / Changelog（订阅 RSS）
├── GitHub Release（star/watch 竞品仓库）
├── G2 / Capterra 用户评测（月度查看）
├── Twitter / LinkedIn 官号
└── 竞品广告投放（SimilarWeb / 广告库）

每周 10 分钟扫描，发现重大更新立即记录到竞品动态文档。
```

### 季度深度分析

```
步骤：
1. 确认本季度重点竞品（1-2 个）
2. 完整体验产品（注册试用或复用团队账号）
3. 填写竞品分析模板（见第四节）
4. 与上次分析对比，标记变化
5. 输出结论和对产品路线图的建议
6. 在季度规划会议上同步结论
```

## 五、信息管理

| 类型 | 存储位置 | 更新频率 |
|------|---------|---------|
| 竞品动态速记 | 飞书文档《竞品动态》 | 随时 |
| 季度深度分析 | 本目录，按 `{竞品名}-{YYYY-Q}` 命名 | 季度 |
| 竞品功能对比矩阵 | 飞书多维表格 | 每次深度分析后更新 |
| 竞品定价历史 | 飞书文档《竞品定价追踪》 | 发现变化时更新 |

---

## 六、输出与同步

| 场景 | 同步范围 | 形式 |
|------|---------|------|
| 重大竞品更新（新功能/发布新版本）| 产品+研发 | 飞书消息同步 |
| 季度竞品分析结论 | 全团队 | 季度规划会议 |
| 影响路线图的竞品动作 | 产品+商务+管理层 | 单独汇报 |
