# 产品数据分析规范

> 定义产品核心指标体系、埋点规范和分析报告标准，用数据驱动产品决策。

---

## 一、核心指标体系（北极星指标）

### 北极星指标

> **核心指标**：月活跃知识库查询次数（Monthly Active Queries）

这一指标同时反映：产品被真实使用（活跃度）+ 知识库内容有价值（内容质量）+ 用户找到了答案（功能价值）。

### 指标树

```
北极星：月活跃查询次数（MAQ）
├── 用户维度
│   ├── 月活用户数（MAU）
│   ├── 用户留存率（Day7 / Day30 / Day90）
│   └── 用户活跃度（人均查询次数/月）
├── 内容维度
│   ├── 知识库文档数
│   ├── 文档覆盖率（有答案率）
│   └── 内容更新频率
└── 质量维度
    ├── 答案满意率（用户点赞/踩比例）
    ├── 引用准确率（有正确来源引用的回答比例）
    └── 响应延迟 P50 / P99
```

### 指标定义台账

| 指标名 | 定义 | 计算方式 | 数据来源 | 负责人 |
|--------|------|---------|---------|--------|
| MAU | 月内有任意一次查询行为的用户数 | Count Distinct user_id WHERE event='query' | 埋点事件 | 产品 |
| 留存率 D7 | 新用户注册后第 7 天仍有活跃 | 注册后 D7 有 event 的用户 / 新用户总数 | 埋点事件 | 产品 |
| 答案满意率 | 有反馈的查询中满意（点赞）的比例 | 点赞次数 / (点赞+踩) 次数 | 埋点事件 | 产品 |
| 响应延迟 P99 | 99% 的查询在此时间内返回结果 | 服务端日志统计 | 监控系统 | 研发 |

## 二、埋点规范

### 2.1 埋点事件命名规则

```
格式：{模块}_{动作}_{对象}

示例：
- knowledge_query_submit       # 提交查询
- knowledge_answer_feedback    # 对答案进行反馈
- document_upload_complete     # 文档上传完成
- user_login_success           # 登录成功
- admin_setting_save           # 管理员保存设置
```

### 2.2 标准公共属性

每个埋点事件**必须携带**以下公共属性：

| 属性 | 类型 | 说明 |
|------|------|------|
| `user_id` | string | 用户唯一标识（内部 ID，非敏感信息）|
| `org_id` | string | 所属企业/租户 ID |
| `timestamp` | number | 事件发生时间戳（ms）|
| `session_id` | string | 会话 ID，用于拼接用户行为序列 |
| `client` | string | web / mobile / api |
| `page` | string | 当前所在页面路径 |

### 2.3 核心事件埋点清单

| 事件名 | 触发时机 | 关键属性 |
|--------|---------|---------|
| `knowledge_query_submit` | 用户点击发送/回车 | query_text, kb_id, model_name |
| `knowledge_answer_received` | 答案完整返回 | query_id, latency_ms, has_citation |
| `knowledge_answer_feedback` | 用户点赞/踩 | query_id, feedback_type(like/dislike), feedback_text |
| `document_upload_start` | 开始上传文档 | file_type, file_size_kb |
| `document_upload_complete` | 上传解析完成 | doc_id, chunk_count, duration_ms |
| `document_upload_fail` | 上传失败 | error_code, error_message |
| `user_login_success` | 登录成功 | login_method(sso/password) |
| `user_session_end` | 会话结束或超时 | session_duration_s, query_count |

### 2.4 埋点接入规范

```javascript
// 前端埋点示例（参考格式）
tracker.track('knowledge_query_submit', {
  query_text: query,      // 注意：不上报敏感内容时需脱敏
  kb_id: currentKbId,
  model_name: selectedModel,
});

// 注意事项：
// 1. 不上报用户的原始查询内容到第三方分析平台（隐私保护）
// 2. 内部分析系统可存储查询，但需做访问权限控制
// 3. 每个新功能上线前必须提交埋点方案 Review
```

## 五、数据访问与隐私

| 数据类型 | 访问权限 | 存储位置 | 保留期 |
|---------|---------|---------|--------|
| 用户行为事件（脱敏）| 产品团队 | 内部数据仓库 | 1 年 |
| 查询内容（含文本）| 仅数据分析师+隐私合规 | 加密存储 | 90 天 |
| 客户级聚合数据 | CSM + 产品 | BI 系统 | 长期 |

> 任何包含用户原始查询内容的数据访问需走审批流程，禁止直接导出到本地。
