# AI 提效计划 - 产品分析

> 将 AI 嵌入竞品分析、用户研究、市场洞察和数据分析四条主线，目标：每月产品分析人力投入减少 40%，洞察质量和响应速度显著提升。

---

## 一、效率基线（现状）

> 量化当前各环节的实际时耗，作为 AI 提效的对比基准。

| 环节 | 当前耗时（估算）| 参与人 | 主要痛点 |
|------|--------------|--------|---------|
| 月度竞品分析报告 | 2 人天/月 | 产品经理 | 手动收集信息，信息滞后 2-4 周 |
| 用户反馈分类归纳 | 3-5 小时/周 | 产品经理 | 大量文本人工读取，分类主观 |
| 季度市场洞察报告 | 3 人天/季度 | 产品负责人 | 依赖个人阅读面，覆盖不全 |
| PRD 初稿撰写 | 1-2 人天/需求 | 产品经理 | 重复性框架内容耗时多 |
| 数据埋点分析 | 4-8 小时/次 | 产品 + 数据 | SQL 查询门槛高，周期长 |

## 三、高价值机会点详细方案

### 机会1：竞品动态 AI 监控

**当前状态**：人工月度收集，信息经常滞后。  
**目标状态**：核心竞品（3-5家）实现周级自动更新，产品经理只需 Review 摘要。

**方案设计**：
```
信息源采集 → AI 摘要提炼 → 结构化入库 → 周报推送

信息源：
  - 竞品官网更新（通过 Change Detection 监控）
  - 竞品 App Store 评论（通过 AppFollow / 手动抓取）
  - 竞品技术博客 / GitHub Release Notes
  - 行业媒体（36Kr、少数派、InfoQ 等 RSS 订阅）

AI处理：
  - 用 Claude API 对每条变更生成 100 字摘要
  - 自动标注变动类型：功能更新 / 定价变化 / 战略调整
  - 与上期对比，高亮新变化

输出：
  - 每周一自动推送到飞书群（产品频道）
  - 月度汇总报告自动生成草稿
```

**工具栈**：n8n（调度） + Firecrawl/Jina（网页抓取） + Claude API（摘要） + 飞书 Webhook（推送）  
**前置条件**：确定核心竞品清单（≥3家），飞书 Bot 权限  
**实施周期**：3-5 天  
**ROI 估算**：节省 1 人天/月（竞品信息收集），信息时效从月更到周更  

---

### 机会2：用户反馈 AI 分类与趋势分析

**当前状态**：来自销售、客服、用户访谈的反馈散落在飞书文档/邮件，每周人工归类。  
**目标状态**：反馈自动流入统一台账，AI 自动分类并识别趋势，产品每周只需 30 分钟 Review。

**方案设计**：
```
反馈收集 → 统一入库 → AI分类打标 → 趋势聚合 → 产品决策

分类维度（AI打标）：
  - 功能模块（智能问答/文档处理/权限/性能/其他）
  - 反馈类型（Bug/功能缺失/体验优化/正向反馈）
  - 优先级信号（客户规模/频次/情绪强度）
  - 关联需求（如已有Jira issue则自动关联）

Prompt 示例：
  "你是一位产品分析师，请分析以下用户反馈：
   - 所属功能模块（从[智能问答/文档处理/权限管理/性能/其他]中选择）
   - 反馈类型（Bug/功能缺失/体验优化/正向反馈）
   - 核心诉求（一句话总结）
   - 优先级建议（高/中/低）及原因
   反馈内容：{feedback_text}"

月度趋势报告自动生成：
  - TOP 5 高频问题
  - 趋势变化（与上月对比）
  - 建议优先处理的 3 个问题
```

**工具栈**：飞书多维表格 + Claude API（分类） + n8n（自动化）  
**前置条件**：建立统一反馈收集入口（飞书表单 or 反馈邮箱）  
**实施周期**：1 周  
**ROI 估算**：节省 3-4 小时/周（反馈分类），分类一致性从 ~70% 提升到 >90%

### 机会3：PRD AI 辅助起草

**当前状态**：产品经理从零开始写 PRD，大量时间花在框架填充和背景描述上。  
**目标状态**：产品经理输入需求背景，AI 生成结构完整的 PRD 草稿，人工只需补充核心业务逻辑和细节。

**方案设计**：
```
产品经理输入：
  - 功能名称
  - 解决什么问题（用户痛点）
  - 目标用户
  - 核心流程描述（2-3句话）

AI 输出 PRD 草稿包含：
  - 背景与目标
  - 用户故事（User Stories格式）
  - 功能范围（In Scope / Out of Scope）
  - 交互流程（文字描述）
  - 验收标准（Acceptance Criteria）
  - 数据指标（建议埋点）
  - 风险与依赖
```

**工具选型**：  
- 方案A（推荐）：基于 Dify 搭建内部 PRD 助手，接入公司知识库（历史PRD参考）  
- 方案B（快速验证）：Claude Projects，上传历史 PRD 作为上下文参考

**前置条件**：整理 5-10 份历史 PRD 作为示例（质量基准）  
**实施周期**：方案A 1-2周，方案B 1天可用  
**ROI 估算**：PRD 初稿时间从 1-2 人天缩短到 2-4 小时，降低约 60%

### 机会4：自然语言产品数据查询

**当前状态**：产品查数据需要找数据工程师写 SQL，通常等待 1-3 天。  
**目标状态**：产品经理直接用中文提问，AI 生成 SQL 并返回可视化结果。

**方案设计**：
```
产品经理：「上周新注册用户中，有多少人在 7 天内使用了 AI 查询功能？」

AI（Chat2SQL）：
  1. 理解语义，识别相关表和字段
  2. 生成 SQL（展示给用户确认）
  3. 执行查询并可视化结果
  4. 提供简短的数据解读
```

**工具选型**：Metabase（已有 or 评估中）+ LLM SQL Generator  
**前置条件**：数据指标定义台账（关键表/字段有中文注释），需数据工程师配合  
**实施周期**：2-4 周（依赖数据基础建设）  
**ROI 估算**：70% 的临时数据需求从 1-3 天缩短到 15 分钟

## 四、实施路径

### Phase 0（第 1-2 周）：快速起效，验证价值

**目标**：在不引入新基础设施的情况下，立即体验 AI 提效，建立使用习惯。

| 任务 | 具体行动 | 验收标准 | Owner |
|------|---------|---------|-------|
| PRD 助手快速验证 | 在 Claude Projects 上传 5 份历史 PRD，起草 1 份新需求 | 产品经理评分：AI草稿节省时间 > 50% | 产品经理 |
| 用户反馈手动试跑 | 选取最近 1 个月反馈（≥50条），用 Claude 批量分类 | 分类结果人工抽检准确率 > 85% | 产品经理 |
| 确定竞品清单 | 列出需要监控的核心竞品（3-5家）及信息源 | 完成竞品信息源清单 Excel | 产品负责人 |

---

### Phase 1（第 3-4 周）：自动化核心工作流

**目标**：竞品监控和用户反馈分类实现自动化，减少重复人工操作。

| 任务 | 具体行动 | 验收标准 | Owner | 前置条件 |
|------|---------|---------|-------|---------|
| 竞品监控工作流 | 用 n8n 搭建：信息抓取 → Claude摘要 → 飞书推送 | 每周一自动推送竞品动态摘要 | 运维/产品 | 飞书 Bot 权限 + n8n 部署 |
| 反馈分类自动化 | 打通飞书表单 → n8n → Claude 分类 → 多维表格存档 | 新反馈提交后 5 分钟内完成自动分类 | 运维/产品 | 飞书多维表格权限 |
| 确立 Prompt 规范 | 为每个场景（PRD/分类/竞品）建立标准 Prompt，版本化管理 | Prompt 文档写入知识库，可复用 | 产品经理 | - |

---

### Phase 2（第 5-8 周）：提升质量，形成规范

**目标**：AI 分析结果纳入正式流程，产品决策有 AI 洞察支撑。

| 任务 | 具体行动 | 验收标准 | Owner | 前置条件 |
|------|---------|---------|-------|---------|
| 月度洞察报告 AI 化 | 基于 Phase 1 数据积累，用 AI 生成月度竞品+反馈洞察报告草稿 | 报告生成时间 < 2 小时，内容 Review 通过 | 产品负责人 | Phase 1 数据积累 ≥ 1 个月 |
| Dify PRD 助手上线 | 部署 Dify，接入公司知识库，替代 Claude Projects 方案 | PRD 助手可用，响应时间 < 10s | 运维 | 内网 Dify 部署 |
| AI 洞察纳入月度产品会议 | 每月产品会议中，AI 生成的竞品/反馈洞察作为固定议题 | 已执行 ≥ 1 次，产品团队认可价值 | 产品负责人 | Phase 1 完成 |

---

### Phase 3（第 9-12 周）：深度数据智能

**目标**：产品经理可自助查询数据，减少对数据工程师的依赖。

| 任务 | 具体行动 | 验收标准 | Owner | 前置条件 |
|------|---------|---------|-------|---------|
| 数据字典完善 | 与数据工程师协作，完成核心表/字段的中文注释 | 关键业务指标对应的表字段有注释覆盖率 > 80% | 数据工程师 | - |
| Chat2SQL 接入 | 在 Metabase 或 Dify 中接入 SQL 生成能力 | 产品经理可用中文提问，获得准确 SQL | 数据工程师 + 运维 | 数据字典完善 |
| 效果回顾 | 统计 AI 提效实际数据，对比 Phase 0 的基线 | 月度分析人力投入对比基线减少 ≥ 40% | 产品负责人 | Phase 1-2 数据 |

## 六、风险与回退

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| AI 分类准确率不达标（<80%）| 误导产品决策 | 低准确率的反馈标记为"待人工确认"，不直接进入看板 |
| 竞品信息源失效（网站改版）| 监控中断 | 每月人工检查一次信息源有效性，n8n 设置失败告警 |
| PRD 草稿质量差（偏离业务）| 产品经理仍需大量修改，提效不明显 | 建立草稿质量评分机制，持续迭代 Prompt |
| API 成本超预期 | 超出预算 | 设置 API 月度用量上限告警（Claude Console 中设置）|
| 团队对 AI 输出过度依赖 | 降低人工判断能力 | AI 输出标注为"参考"，重要决策必须有人工确认 |
