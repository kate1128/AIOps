# 测试策略规范

> 定义测试分层、各层的职责边界和覆盖要求，确保测试投入产出比最优。

---

## 一、测试分层

```
                    /\
                   /  \
                  /    \
                 / E2E  \        —— 5%：关键业务流程
                /  (API)  \
               /────────────\
              /              \
             /   集成测试    \  —— 15%：服务间交互
            /    (Service)    \
           /────────────────────\
          /                      \
         /      单元测试         \  —— 60%：函数/模块
        /        (Unit)           \
       /────────────────────────────\
      /                              \
     /        手动探索测试           \  —— 20%：新功能/UI
    /        (Exploratory)            \
   /────────────────────────────────────\
```

---

## 二、各层测试要求

### 单元测试（Unit Test）

| 维度 | 要求 |
|------|------|
| 范围 | 函数、方法、工具类 |
| 速度 | 毫秒级 |
| 运行 | CI 每次提交自动运行 |
| 覆盖目标 | 核心业务逻辑 > 80%，整体 > 60% |
| Mock | 外部依赖全部 Mock，不依赖真实服务或数据库 |

### 集成测试（Integration Test）

| 维度 | 要求 |
|------|------|
| 范围 | 服务间调用、数据库访问、外部 API |
| 速度 | 秒级到分钟级 |
| 运行 | MR 触发 + 每日定时 |
| 覆盖目标 | 核心调用链路、异常场景 |
| 环境 | 使用独立测试数据库或 Testcontainer |

### E2E 测试（End-to-End Test）

| 维度 | 要求 |
|------|------|
| 范围 | 关键业务流程（登录、创建、核心操作）|
| 速度 | 分钟级 |
| 运行 | 预发环境部署后触发 |
| 覆盖目标 | Top 10 核心用户旅程 |
| 环境 | 预发环境，独立测试账号 |

### 手动探索测试（Exploratory Test）

| 维度 | 要求 |
|------|------|
| 范围 | 新功能、复杂交互、边界场景 |
| 频率 | 每个迭代 |
| 重点 | 自动测试覆盖不到的视觉、交互、体验问题 |
| 记录 | 必须记录测试过程和发现的 Bug |

---

## 三、测试优先级

| 优先级 | 内容 | 强制要求 |
|--------|------|----------|
| **P0 - 关键** | 核心业务流程（登录、支付、核心 AI 推理） | 必须有单元 + 集成 + E2E |
| **P1 - 重要** | 主要功能模块 | 必须有单元 + 集成 |
| **P2 - 常规** | 辅助功能 | 建议有单元测试 |
| **P3 - 次要** | UI 展示、文案 | 手动测试即可 |

---

## 四、测试前置（Shift Left）

### 测试介入时机

| 阶段 | 测试活动 |
|------|----------|
| 需求评审 | QA 参与评审验收标准的可测试性 |
| 技术设计 | QA 参与评审测试方案 |
| 开发阶段 | 开发编写单元测试，QA 编写测试用例 |
| Code Review | 测试代码同步 Review |
| CI 阶段 | 自动运行单元 + 集成测试 |
| 预发部署 | 自动运行 E2E 测试 |

---

## 五、禁止规则

| ❌ 禁止 | 原因 |
|---------|------|
| 无测试合入主干代码 | 代码质量无保障 |
| 测试环境使用生产数据 | 数据安全风险 |
| 手动测试结果不做记录 | 不可追溯、不可复盘 |
| E2E 测试依赖测试人员本地环境 | 环境不一致导致误报 |
