# 产品缺陷管理 FAQ

---

**Q1：现在缺陷管理到底用飞书还是 Jira？**

当前建议是：**飞书负责提单，Jira 负责处理**。

- Jira 继续作为唯一缺陷系统，用来做分级、指派、状态流转、统计报表
- 飞书多维表格或飞书表单作为统一入口，方便产品、测试、客服提报问题
- 通过自动同步把飞书记录创建为 Jira Bug，避免人工搬运和漏单

这不是双系统并行处理，而是前台入口和后台处理分层。状态流转以 Jira 为准，飞书不直接处理缺陷状态。

---

**Q2：P0 Bug 需要中断当前迭代去修吗？**

是的，P0 定义为"核心功能不可用或数据安全风险"，必须立即中断当前工作处理。

流程：

1. 确认 P0 后，**当前迭代内容全部暂停**
2. 建立 hotfix 分支（`hotfix/fix-xxx`）
3. 修复 → QA 验证 → 紧急发版（走 CI/CD hotfix 流水线）
4. 发版后复盘（1 天内）

---

**Q3：开发说 Bug 无法复现，怎么处理？**

Bug 单必须包含复现步骤，如果开发无法复现：

1. 要求提报人提供更详细的复现信息（traceId、日志、操作录屏）
2. 开发和 QA 结对调试
3. 如果 3 个工作日仍无法复现，可以标记为 `status::cannot-reproduce`，但不关闭，继续观察

**不允许**直接以"无法复现"为由关闭 Bug。

---

**Q4：同一个 Bug 在不同版本都存在，需要开多个 Bug 单吗？**

不需要。在同一个 Bug 单中用 `影响版本` 字段记录所有受影响的版本。修复时在各版本分支分别 cherry-pick，commit 都关联同一个 Issue。

---

**Q5：客户反馈的问题和内部测试发现的 Bug 用同一套流程吗？**

是的，统一走同一套流程，只是来源字段不同。处理方式建议如下：

1. 客服、产品、测试统一在飞书表格提交问题
2. 飞书自动同步创建 Jira Bug，并带上来源字段，如 customer、internal-qa、alert
3. 在 Jira 按同一套分级、指派、修复、验证流程处理

客户反馈的 Bug 因为有外部影响，在同等严重程度下适当提高优先级。

---

**Q6：Bug 修复后需要回归测试哪些范围？**

| Bug 严重程度 | 回归范围 |
| --- | --- |
| P0/P1 | 全量冒烟测试 + 问题模块回归 |
| P2 | 问题模块回归 + 相关联模块 |
| P3 | 问题点验证即可 |
