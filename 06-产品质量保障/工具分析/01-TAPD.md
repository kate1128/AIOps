# 缺陷与项目管理工具选型：TAPD vs Linear vs Jira

> 三者功能高度重叠，核心是**三选一**，不是同时使用。本文说清楚什么情况选哪个。

---

## 一句话定位

| 工具 | 定位 |
|---|---|
| **Linear** | 为工程师设计的极简项目管理，键盘驱动，开发体验最好 |
| **TAPD** | 国内全功能研发协作平台，内置测试管理，数据在国内 |
| **Jira** | 企业级高度定制，大型组织和复杂流程的事实标准 |

---

## 核心差异（选型关键维度）

| 维度 | Linear | TAPD | Jira |
|---|---|---|---|
| **数据存储** | 境外（美国）| 国内（腾讯云）| 境外（可私有化部署）|
| **内置测试管理** | ❌ | ✅ | 需插件（Xray，付费）|
| **操作体验** | 极简，键盘优先，秒级响应 | 中等 | 复杂，点击多，配置量大 |
| **定制化程度** | 中等（够用）| 高 | 极高（容易过度配置）|
| **学习成本** | 低（30 分钟上手）| 低中 | 高（需要专门培训）|
| **适合规模** | 5-50 人 | 10-500 人 | 50+ 人 |
| **价格** | 适中 | 国内性价比高 | 贵（企业版）|
| **GitLab 集成** | ✅ 深度自动化 | ✅ | ✅ |
| **中文支持** | 有 | 原生 | 有但不完善 |

## 各工具适用场景

### Linear — 适合这种团队
- 纯研发驱动的小团队（< 50 人）
- 工程师文化强，大家愿意用键盘操作
- 不需要复杂审批流程
- 数据在境外没有合规风险

**最大优势**：提交代码时 `fix(LIN-234)` 自动关联 Issue，MR 合并自动完成 Issue，几乎零手动操作。

### TAPD — 适合这种团队
- 国内商业化产品团队，数据合规有要求
- 需要测试用例管理和测试计划
- 产品、开发、测试、运营都在同一平台协作
- 腾讯企业微信生态（通知集成好）

**最大优势**：内置测试管理模块，Bug ↔ 测试用例 ↔ 需求的关联是原生的，不需要额外工具。

### Jira — 适合这种团队
- 已有 Atlassian 全家桶（Confluence + Bitbucket）
- 大型企业，多部门协作，复杂审批流
- 需要高度定制的工作流（不同项目不同流程）
- 合规要求高，需要私有化部署

**最大优势**：几乎可以配置成任何流程，插件生态最大，与 Confluence 集成最深。

## 缺陷管理核心流程（三个工具通用）

```
Bug 来源：
  ├── QA 手动提交
  ├── Sentry 自动触发（生产报错）
  └── 用户反馈（客服转单）

状态流转：
  新建 → 确认 → 分配 → 修复中 → 待验证 → 关闭
                                    └→ 重新打开（验证不通过）

代码关联：
  git commit -m "fix: 修复推理超时 --bug=1234"  ← TAPD
  git commit -m "fix(LIN-567): 修复推理超时"    ← Linear
  git commit -m "fix: 修复推理超时 PROJ-567"    ← Jira
```

---

## 关键缺陷指标（工具无关）

| 指标 | 计算方式 | 健康值 |
|---|---|---|
| **修复及时率** | 按时修复数 / 总 Bug 数 | > 90% |
| **逃逸率** | 生产发现 / (测试 + 生产) | < 10% |
| **重开率** | 被重开 Bug / 已关闭 Bug | < 5% |
| **P0/P1 平均修复时长** | 提交到关闭平均时间 | P0 < 4h，P1 < 24h |

## 与 Sentry 联动（三个工具都支持）

Sentry 检测到生产报错 → Webhook → 自动在项目管理工具创建 Bug，附上堆栈和影响用户数，避免人工转单。

```python
# Sentry Webhook → 自动创建 Bug（以 TAPD 为例，其他工具换 API 即可）
def on_sentry_alert(issue: dict):
    priority = "高" if issue['userCount'] > 100 else "中"
    create_bug(
        title=f"[Sentry] {issue['title']}",
        description=f"影响用户: {issue['userCount']}\n链接: {issue['permalink']}",
        priority=priority
    )
```

## 本项目建议

> **结论：选 TAPD**
>
> 理由：国内商业化产品、数据合规、内置测试管理、性价比好。
> 如果后期团队 < 20 人且无合规要求，可以切换到 Linear 提升工程师体验。
> Jira 除非已有全家桶，否则没有必要引入。

```
Bug 提交（QA/用户/自动化）
    │
    ├── 状态：新建 ──→ 确认 ──→ 分配 ──→ 修复中
    │                                    │
    │                         ←──────────┘
    │                         验证中（QA）
    │                         │
    │                    ┌────┴────┐
    │                  通过       不通过
    │                    │           │
    │                  关闭      重新打开（升级优先级）
    │
    └── 关联：提交代码（Commit 引用 Bug ID）

```

## 缺陷数据关键指标

### 建议在 TAPD 报表中追踪的指标

| 指标 | 计算方式 | 健康值参考 |

| **P0/P1 平均修复时长** | 从提交到关闭的平均时间 | P0 < 4h，P1 < 24h |

## 与 Sentry 的联动

```
生产环境报错（Sentry）
    │
    ├── Sentry 告警通知
    └── 自动或手动 → 在 TAPD 创建 Bug
            │
            ├── 附上 Sentry Issue 链接（堆栈、影响用户数）
            ├── 优先级根据影响用户数自动设置
            └── 分配到对应服务负责人
```

可以用 Sentry 的 Webhook 集成，在 Issue 达到阈值时自动调用 TAPD API 创建 Bug：

```python
# Sentry Webhook → 自动创建 TAPD Bug
import requests

def create_tapd_bug(sentry_issue: dict):
    tapd_api = "https://api.tapd.cn/bugs"
    payload = {
        "workspace_id": "YOUR_WORKSPACE_ID",
        "title": f"[Sentry] {sentry_issue['title']}",
        "description": f"""
**Sentry Issue**: {sentry_issue['permalink']}
**影响用户**: {sentry_issue['userCount']}
**发生次数**: {sentry_issue['count']}
**首次发生**: {sentry_issue['firstSeen']}

**堆栈摘要**：
{sentry_issue['culprit']}
        """,
        "priority": "高" if sentry_issue['userCount'] > 100 else "中",
        "status": "新建",
    }
    resp = requests.post(
        tapd_api,
        json=payload,
        auth=("api_user", "api_token")
    )
    return resp.json()
```
