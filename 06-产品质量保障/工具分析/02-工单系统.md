# 客户工单系统选型：Zendesk vs Freshdesk vs 飞书服务台

> 客户反馈缺陷的入口，与内部 Bug 追踪系统（TAPD/Jira）是两层——工单系统面向客户，Bug 追踪系统面向研发。

---

## 分层架构

```
客户
  │ 提交工单（描述问题、截图、复现步骤）
  ▼
工单系统（Zendesk / Freshdesk / 飞书服务台）
  │ 客服分类处理
  ├── 可直接回答 ──→ 直接回复客户，关闭工单
  └── 确认是 Bug ──→ 转成内部 Bug（推送到 TAPD/Jira）
                            │
                            开发修复
                            │
                            Bug 关闭 ──→ 触发工单状态更新 ──→ 回复客户
```

---

## 工具对比

| 维度 | 飞书服务台 | Freshdesk | Zendesk |
|---|---|---|---|
| **数据存储** | 国内（飞书云）| 国际 | 国际 |
| **前提条件** | 需已用飞书 | 独立部署 | 独立部署 |
| **价格** | 飞书套餐内含 | 免费版可用 | 较贵 |
| **功能完整度** | 中 | 高 | 最高 |
| **与研发工具集成** | Jira / TAPD / GitLab 可集成 | Jira / TAPD 有集成 | Jira / Linear 有集成 |
| **客户接入渠道** | 飞书 / Web | Email / Web / 社交媒体 | Email / Web / 社交媒体 / 电话 |
| **适合场景** | 已用飞书的企业内部和外部支持 | 中小型 SaaS 产品客户支持 | 大型企业、多渠道客服 |

```
团队已用飞书 ──→ 飞书服务台（零边际成本，通知无缝）
       │
```

## 工单 → Bug 转化流程

### 飞书服务台 + Jira（自动同步）

```
1. 客户在飞书服务台提工单
2. 命中自动化规则（如标签=bug 或 分类=缺陷）
3. 自动创建 Jira Issue（项目=SUP，类型=Bug）
4. 飞书工单写回 Jira Key（如 SUP-123）
5. Jira 状态变更（Done/Resolved）自动回写飞书工单状态
```

#### 配置入口（飞书后台）

```
飞书服务台后台
    → 设置
    → 第三方集成
    → Jira
    → 绑定 Jira Cloud 域名 + API Token
```

#### 自动化规则建议

```
触发条件：
    - 工单分类 = 缺陷
    - 或工单标签包含 bug

执行动作：
    - 创建 Jira Issue
    - 回填字段：jira_key、jira_url
    - 通知责任群（飞书群）

状态回写：
    - Jira: To Do/In Progress/Done
    - 飞书工单: 待处理/处理中/已解决
```

#### 字段映射（推荐）

| 飞书服务台字段 | Jira 字段 |
|---|---|
| 工单标题 | Summary |
| 工单描述 | Description |
| 客户优先级（P1/P2/P3）| Priority |
| 问题分类（产品模块）| Component |
| 客户ID/公司名 | Custom Field: Customer |
| 附件（截图/录屏）| Attachment |

> 建议在 Jira 中新增 `来源渠道=Feishu` 字段，方便后续统计客户反馈问题占比。

### Freshdesk + TAPD（Webhook）

```python
# Freshdesk Webhook → 自动在 TAPD 创建 Bug
import requests

def freshdesk_to_tapd(ticket: dict):
    """Freshdesk 工单转 TAPD Bug"""
    # 只转化被标记为 Bug 的工单（客服打标签 "bug"）
    if "bug" not in ticket.get("tags", []):
        return

    tapd_bug = {
        "workspace_id": "YOUR_WORKSPACE_ID",
        "title": f"[客户工单#{ticket['id']}] {ticket['subject']}",
        "description": f"""
**来源**: Freshdesk 工单 #{ticket['id']}
**客户**: {ticket['requester']['name']}
**问题描述**:
{ticket['description_text']}

**Freshdesk 链接**: {ticket['url']}
        """,
        "priority": map_priority(ticket['priority']),
        "status": "新建",
        "label": "客户反馈",
    }

    resp = requests.post(
        "https://api.tapd.cn/bugs",
        json=tapd_bug,
        auth=("api_user", "api_token")
    )
    bug_id = resp.json()['data']['Bug']['id']

    # 把 TAPD Bug ID 写回 Freshdesk 工单备注
    requests.post(
        f"https://yourcompany.freshdesk.com/api/v2/tickets/{ticket['id']}/notes",
        json={"body": f"已创建内部 Bug，追踪编号：TAPD #{bug_id}"},
        auth=("api_key", "X")
    )

def map_priority(freshdesk_priority: int) -> str:
    return {1: "低", 2: "中", 3: "高", 4: "高"}[freshdesk_priority]
```

### Bug 修复后反向通知（TAPD Webhook → Freshdesk）

```python
# TAPD Bug 关闭时，触发 Freshdesk 工单回复
def tapd_bug_closed(bug: dict):
    ticket_id = extract_ticket_id(bug['description'])  # 从描述中提取
    if not ticket_id:
        return

    requests.put(
        f"https://yourcompany.freshdesk.com/api/v2/tickets/{ticket_id}",
        json={
            "status": 4,  # 4 = Resolved
            "reply": {
                "body": f"您好，您反馈的问题已在 {bug['fix_version']} 版本修复，感谢您的反馈！"
            }
        },
        auth=("api_key", "X")
    )
```

## 工单字段规范

客服接单时必须填写的字段，确保转 Bug 时有足够信息：

| 字段 | 必填 | 说明 |
|---|---|---|
| 问题描述 | ✅ | 客户的原始描述 |
| 复现步骤 | ✅ | 客服引导客户填写 |
| 发生频率 | ✅ | 必现 / 偶发 / 一次 |
| 影响功能 | ✅ | 下拉选择：推理/文档/账户/计费 |
| 客户类型 | ✅ | 付费用户 / 试用 / 免费 |
| 截图/录屏 | 建议 | 帮助开发定位 |
| 浏览器/版本 | 建议 | 前端 Bug 必填 |
