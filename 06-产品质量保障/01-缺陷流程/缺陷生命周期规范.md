# 缺陷生命周期规范

> 定义 Bug 从发现到关闭的完整状态流转、各角色职责边界和强制规则。

---

## 一、状态流转图

```
新建 (New)
  │
  ├─→ 已拒绝 (Rejected)        — 非 Bug（需求误解/预期行为），由开发说明原因
  │
  ↓
确认 (Confirmed)               — 可复现，属于 Bug，由 QA 或开发确认
  │
  ↓
已分配 (Assigned)              — 指派给具体开发，填写预计修复版本
  │
  ↓
修复中 (In Progress)           — 开发已开始处理
  │
  ↓
待验证 (Verifying)             — 开发完成修复，@QA 验证
  │
  ├─→ 重新打开 (Reopened)      — QA 验证不通过，退回开发
  │         │
  │         └─→ 修复中（重新进入流程）
  ↓
已关闭 (Closed)                — QA 验证通过，正式关闭
```

---

## 二、各角色职责

| 角色 | 职责 | 不能做 |
|---|---|---|
| **提报人**（QA/开发/客服）| 创建 Bug 单，填写完整复现步骤、环境、分级 | 不能自行关闭或修改优先级 |
| **QA** | 确认 Bug 可复现 → Confirmed；完成验证 → Closed 或 Reopened | 不能跳过验证直接关闭 |
| **开发** | 修复 Bug，提交时关联 Issue；完成后状态改为 Verifying | 不能自行将 Bug 改为 Closed |
| **技术 Leader** | 审核 P0/P1 分级；超期 Bug 升级处理 | — |

---

## 三、状态流转规则

| 规则 | 说明 |
|---|---|
| **开发不能自行关闭 Bug** | 必须经 QA 验证后才能关闭 |
| **Reopened 自动升级** | Bug 被重新打开一次，优先级升一级（P2 → P1）|
| **超期未处理自动通知** | P0 超 4h 未进入 In Progress，自动通知技术负责人 |
| **拒绝必须说明原因** | Rejected 状态必须填写拒绝原因，提报人有权申诉 |
| **不允许直接 Closed** | 不允许从 New/Confirmed/In Progress 直接跳到 Closed |

### Label 规范

```

P0-critical
P1-severe
P2-normal
P3-minor

# 状态（必填，用 scoped label 限制单选）
status::new
status::confirmed
status::in-progress
status::verifying

# 模块

module::ai

source::customer
```

## 五、禁止规则

| ❌ 禁止 | 原因 |
|---|---|
| 开发自行将 Bug 状态改为 Closed | 修复质量无保障，绕过 QA 验证 |
| Bug 单无复现步骤 | 开发无法复现，只能靠猜 |
| 以"无法复现"为由直接关闭 | 应标记 `cannot-reproduce` 并继续观察 |
| P0 Bug 超过 4 小时无响应 | 必须有人认领并开始处理，否则上报 |
| Bug 分级随意（全填 P1）| 导致优先级失去意义，P0 响应被稀释 |
