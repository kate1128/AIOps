# 产品质量保障 体系建设总览

> 质量不是测完才出来的，而是内建在开发流程中的——从需求到上线的每一个环节都应该有质量关卡。本体系的目标是让缺陷越来越少（防患于未然），让已有缺陷越来越快被发现、定位、修复。

---

## 一、现状与痛点

### 1.1 核心子域评估

| 子域 | 当前状态 | 主要痛点 | 影响 |
|------|---------|---------|------|
| **缺陷流程** | 🟡 Jira 跟踪，流程基本完善 | 缺陷指派依赖人工判断，新人不熟悉模块归属，指派不准 | 高优缺陷响应慢 |
| **测试策略** | 🔴 主要靠人工回归 | 自动化覆盖率低，每次发布需大量手工验证 | 发布节奏受限，发布频率 < 1次/周 |
| **自动化测试** | 🟡 部分接口有脚本 | 脚本维护成本高，接口变更后用例易失效 | 自动化效果打折 |
| **性能测试** | 🔴 无常态化机制 | 只有大版本才做性能测试，日常需求对性能影响无感知 | 性能问题到生产才暴露 |
| **度量体系** | 🟡 有数据但无报告 | 缺陷密度/逃逸率/修复时效散落各处，无统一看板 | 质量趋势不可见 |

### 1.2 典型问题场景

**场景一：缺陷指派来回转单**
研发提交 Bug，测试指派给研发 A，A 认为不是自己的模块，转给 B，B 再看一遍说是 C 的问题。一个缺陷在三个人之间流转 3 次，浪费 1-2 天。根本原因：没有结构化的模块归属映射，指派全凭经验。

**场景二：版本发布前手工回归 3 天**
每次大版本发布前需要 3-4 人花 2-3 天做全量手工回归。自动化覆盖率不足 20%，核心场景都需要人工点击。这直接成为发布节奏的瓶颈。

**场景三：性能问题到生产才暴露**
某需求上线后发现 DB 查询无索引，P99 延迟从 200ms 变为 5s。原因是日常测试没有性能基准对比，无法发现性能退化（regression）。

## 二、各子域建设方案

### 2.1 测试策略（测试金字塔）

**目标覆盖率**：

```
                  ┌───────────────┐
                  │  E2E / UI 测试 │  ~10%（关键用户旅程）
                 ─┴───────────────┴─
              ┌─────────────────────────┐
              │    集成测试 / API 测试   │  ~30%（核心接口契约）
             ─┴─────────────────────────┴─
          ┌─────────────────────────────────┐
          │          单元测试                │  ~60%（业务逻辑、工具函数）
         ─┴─────────────────────────────────┴─
```

**与 CI 流水线联动**（详见 03-CICD建设）：
- `feature` 分支：单元测试 + API 接口测试必须全绿才能合并
- `pre` 分支：自动触发集成测试套件 + 性能基准对比
- `main`/Tag：E2E 冒烟测试 + 全量回归

### 2.2 缺陷流程规范化

**AI 辅助缺陷分类指派流程**：

```
测试/用户提交 Bug（标题 + 复现步骤 + 截图）
  ↓
AI 分析：提取关键词 → 匹配模块归属映射表（从 GitLab 代码目录结构生成）
  ↓
自动推荐：模块 Owner + 优先级初判（P0/P1/P2）+ 相似历史缺陷
  ↓
Jira 自动预填写（模块/指派人/关联需求）
  ↓
Owner 确认 → 正式指派（节省人工判断时间）
```

**缺陷优先级定义**：

| 级别 | 定义 | 响应时效（工作时间）|
|------|------|----------------|
| P0 | 生产服务不可用 / 数据丢失 | 30 分钟内响应，4 小时内修复 |
| P1 | 核心功能异常 / 高频用户路径阻断 | 4 小时内响应，1 天内修复 |
| P2 | 非核心功能问题 / 有 workaround | 1 天内响应，当前迭代修复 |
| P3 | 体验优化 / 低频场景 | 按需排期 |

### 2.3 自动化测试体系

**接口测试（pytest + httpx）**：

```
tests/
├── api/
│   ├── test_user_api.py
│   ├── test_inference_api.py
│   └── conftest.py          # 环境配置、认证 fixture
├── integration/
│   └── test_e2e_workflow.py  # 核心用户旅程（5-10 个关键场景）
└── performance/
    └── test_baselines.json   # 性能基准（P99 延迟上限）
```

**测试用例 AI 辅助生成流程**：

```
接口文档（OpenAPI/已有代码）→ AI 生成 pytest 测试用例框架
  ↓
开发/测试 review 并补充边界测试
  ↓
提交 PR → CI 自动运行 → 测试报告到 MR 评论
```

**性能基准测试（k6）**：
- 触发时机：每次合并到 `pre` 分支时自动运行
- 指标：QPS / P99 延迟 / 错误率
- 回归告警：P99 延迟较上次增加 > 20% 则 CI 失败，附带性能差异报告

### 2.4 质量度量看板

**核心指标定义**：

```
质量门禁指标（每个版本必须达到）：
- 自动化测试通过率 > 99%
- 代码覆盖率 > 60%（核心业务模块 > 80%）
- 缺陷逃逸率（上线后 7 天内生产缺陷数 / 当前迭代缺陷总数）< 10%

过程质量指标（持续监控趋势）：
- 缺陷密度（每千行代码缺陷数）
- 缺陷修复时效（P0/P1 平均修复时间）
- 自动化覆盖率月度增长
```

## 五、建设路径

### Phase 0（第 1-2 周）：质量门禁基础

| 任务 | 具体行动 | 验收标准 | Owner |
|------|---------|---------|-------|
| CI 强制测试 | feature→dev 合并必须通过 pytest 单测 | MR 无法合并（测试不通过时）| 研发/DevOps |
| 缺陷优先级规范 | 在 Jira 配置 P0-P3 工作流和 SLA 提醒 | P0 缺陷未在 4h 内响应时自动飞书告警 | 测试 Lead |
| 质量看板初版 | Grafana/Jira Dashboard：缺陷密度/修复时效 | 每周一可查上周质量趋势报告 | 测试 Lead |

### Phase 1（第 3-4 周）：自动化提速

| 任务 | 具体行动 | 验收标准 | Owner |
|------|---------|---------|-------|
| 测试用例 AI 生成 | 对核心 5 个模块的 OpenAPI 生成 pytest 用例 | 自动化覆盖率从 < 20% → ≥ 40% | 测试 + 研发 |
| Bug 分类指派 | Claude API Webhook 接 Jira | 新建缺陷自动推荐 Owner 和优先级，人工确认率 < 30s | 测试 Lead |
| 性能基准建立 | 关键接口 k6 脚本 + 基准 JSON | pre 分支自动运行性能测试，基准建立完成 | 测试 |

### Phase 2（第 5-8 周）：体系化运营

| 任务 | 具体行动 | 验收标准 | Owner |
|------|---------|---------|-------|
| E2E 测试覆盖 | 5 条核心用户旅程的 E2E 自动化 | 每次 pre 发布前 E2E 自动运行 | 测试 |
| 质量周报自动化 | AI 根据 Jira + CI 数据生成周报 | 每周自动推送飞书，人工零操作 | 测试 Lead |
| 缺陷逃逸率追踪 | 在 Jira 标记"生产发现缺陷"，自动统计逃逸率 | 逃逸率 < 10% 并可追踪趋势 | 测试 |

## 六、关键指标

| 指标 | 当前值 | Phase 2 目标 | 采集方式 |
|------|--------|-------------|---------|
| 自动化覆盖率 | < 20% | > 60% | CI 测试报告 |
| 缺陷指派时效 | 15-30min | < 2min（AI 辅助）| Jira 时间戳 |
| 缺陷逃逸率 | 未统计 | < 10% | Jira 生产缺陷标签统计 |
| P0/P1 修复时效 | 未统计 | P0 < 4h, P1 < 1d | Jira SLA 报告 |
| 用例生成效率 | 2-4h/模块 | < 1h（AI 辅助）| 人工估算对比 |
| 发布回归时间 | 2-3 天（手工）| < 2 小时（自动化）| CI 执行时间 |
