# 性能测试规范

> 性能测试保障系统在高负载下仍能稳定运行，防止"上线即崩溃"。

---

## 一、性能测试类型

| 类型 | 目的 | 工具推荐 | 频率 |
|------|------|----------|------|
| **基准测试** | 建立性能基线 | wrk / ab / Locust | 每次发布前 |
| **压力测试** | 找到系统极限 | Locust / k6 / JMeter | 迭代末 |
| **稳定性测试** | 长时间运行无异常 | Locust / k6 | 月度 |
| **并发测试** | 验证资源竞争 | 自研脚本 / Gatling | 核心功能上线前 |
| **扩容测试** | 验证水平扩展能力 | Locust + K8s HPA | 季度 |

## 二、性能指标

### 响应时间

| 指标 | 核心服务 | 非核心服务 |
|------|----------|------------|
| P50 | < 200ms | < 500ms |
| P95 | < 500ms | < 2s |
| P99 | < 1s | < 5s |

| TPS（AI 推理）| 需测量 | 目标值 TBD |
| 并发连接数 | 需测量 | 目标值 TBD |

### 资源使用

| 指标 | 健康值 | 告警值 |
|------|--------|--------|
| CPU 使用率 | < 70% | > 85% |
| 内存使用率 | < 70% | > 85% |
| GPU 利用率 | > 60% | < 30% |
| 磁盘 IO | < 60% | > 80% |
| 网络带宽 | < 50% | > 70% |

## 三、性能测试流程

### 测试前准备

```
1. 确定性能目标（响应时间 / QPS / 并发数）
2. 搭建测试环境（独立资源或预发环境）
3. 准备测试数据（模拟真实场景）
4. 编写测试脚本
5. 准备监控看板（实时观察资源使用）
```

### 测试执行

```
1. 跑基准测试，记录基线
2. 逐步增加负载，观察拐点
3. 持续加压，找到系统极限
4. 释放负载，观察恢复情况
5. 记录测试结果
```

### 测试后分析

```
1. 对比基线，识别性能退化
2. 定位瓶颈（CPU / 内存 / IO / 锁）
3. 给出优化建议
4. 输出性能测试报告
```

## 五、性能门禁

| 门禁 | 阈值 | 处理 |
|------|------|------|
| P99 响应时间退化 | > 基线 20% | 阻断发布 |
| 错误率 | > 1% | 阻断发布 |
| CPU 峰值 | > 90% 持续 5 分钟 | 告警 |
| 内存泄漏 | 持续增长不释放 | 阻断发布 |

---

## 六、持续性能测试

- 核心接口在 CI/CD 中集成性能回归测试
- 每次发布自动对比基线
- 超过阈值自动阻断
- 每月输出性能趋势报告
