# 资源规格规范

> 统一各类服务的标准资源配置，避免随意申请导致的资源浪费或稳定性风险。

---

## 一、服务分类与资源模型

### 1.1 服务角色分类

| 分类 | 典型服务 | CPU 敏感度 | 内存敏感度 | GPU 需求 | 存储需求 |
|------|---------|-----------|-----------|---------|---------|
| AI 推理 | vLLM、Triton | 中 | 高 | 必须 | 低 |
| AI 训练 | PyTorch、TF | 高 | 高 | 必须 | 中 |
| API 网关 | Nginx/Kong、Ingress | 中 | 低 | 无 | 低 |
| 业务后端 | Go/Java/Python 服务 | 中 | 中 | 无 | 低 |
| 前端/静态 | Nginx、Node.js | 低 | 低 | 无 | 低 |
| 数据管道 | Kafka、Flink | 高 | 中 | 无 | 中 |
| 存储层 | PostgreSQL、Redis、Milvus | 中 | 高 | 无 | 高 |
| 可观测性 | Prometheus、Grafana、Loki | 中 | 高 | 无 | 高 |
| 任务型 | CronJob、批处理 | 低 | 中 | 可选 | 低 |

## 二、标准资源规格模板

### 2.1 AI 推理服务

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: vllm-example
  labels:
    app: vllm-qwen-7b
    team: ai-infra
    env: prod
spec:
  replicas: 2
  template:
    spec:
      containers:
      - name: vllm
        image: vllm/vllm-openai:latest
        resources:
          requests:
            cpu: "4"
            memory: "16Gi"
            nvidia.com/gpu: 1
          limits:
            cpu: "8"
            memory: "32Gi"
            nvidia.com/gpu: 1
```

| 规格档次 | CPU Request | CPU Limit | Memory Request | Memory Limit | GPU | 适用模型规模 |
|---------|------------|-----------|---------------|-------------|-----|------------|
| 小型 | 2 | 4 | 8Gi | 16Gi | 1 | 7B 以下 |
| 中型 | 4 | 8 | 16Gi | 32Gi | 1 | 7B-14B |
| 大型 | 8 | 16 | 32Gi | 64Gi | 1-2 | 14B-34B |
| 超大型 | 16 | 32 | 64Gi | 128Gi | 2-4 | 34B-72B |

### 2.2 业务后端服务

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: backend-example
  labels:
    app: ai-backend
    team: backend
    env: prod
spec:
  replicas: 3
  template:
    spec:
      containers:
      - name: backend
        image: smartvision/backend:latest
        resources:
          requests:
            cpu: "1"
            memory: "1Gi"
          limits:
            cpu: "2"
            memory: "2Gi"
        livenessProbe:
          httpGet:
            path: /healthz
            port: 8080
          initialDelaySeconds: 10
          periodSeconds: 15
        readinessProbe:
          httpGet:
            path: /ready
            port: 8080
          initialDelaySeconds: 5
          periodSeconds: 10
```

| 规格档次 | CPU Request | CPU Limit | Memory Request | Memory Limit | 适用场景 |
|---------|------------|-----------|---------------|-------------|---------|
| 微服务 | 0.5 | 1 | 512Mi | 1Gi | 边缘/辅助服务 |
| 小型 | 1 | 2 | 1Gi | 2Gi | 无状态业务服务 |
| 中型 | 2 | 4 | 2Gi | 4Gi | 核心业务服务 |
| 大型 | 4 | 8 | 4Gi | 8Gi | 高负载处理服务 |

| 规格档次 | CPU Request | CPU Limit | Memory Request | Memory Limit | 副本数 |
|---------|------------|-----------|---------------|-------------|-------|
| 低流量 | 0.5 | 1 | 256Mi | 512Mi | 1-2 |
| 中流量 | 1 | 2 | 512Mi | 1Gi | 2-4 |
| 高流量 | 2 | 4 | 1Gi | 2Gi | 3+ |

## 三、GPU 资源规格

### 3.1 GPU 类型与分配策略

| GPU 类型 | 显存 | 适用场景 | 分配策略 |
|---------|------|---------|---------|
| NVIDIA A100 80G | 80GB | 大模型推理/训练 | MIG 或整卡 |
| NVIDIA L20 48G | 48GB | 中型模型推理/精调 | 整卡 |
| NVIDIA RTX 4090 24G | 24GB | 开发调试/小模型 | 整卡或时隙共享 |

### 3.2 GPU 分配建议

```yaml
# 整卡分配（推荐）
resources:
  requests:
    nvidia.com/gpu: 1
  limits:
    nvidia.com/gpu: 1

# MIG 切分（A100 支持，生产环境优先）
# 将 1 张 A100 80G 切分为多个 GPU 实例
# 示例：1g.10gb（1 计算切片 + 10GB 显存）
resources:
  requests:
    nvidia.com/gpu: 1
    nvidia.com/mig.config: "1g.10gb"  # 需要 MIG 管理组件支持
  limits:
    nvidia.com/gpu: 1
    nvidia.com/mig.config: "1g.10gb"
```

### 3.3 GPU 使用规则

```
- 非推理服务不得抢占 GPU（通过 nodeSelector 隔离 GPU 节点）
- 开发环境允许 GPU 超卖，生产环境禁止
- 空闲 GPU Pod 超过 24h 自动告警通知 owner
- GPU 节点预留 10% 容量用于故障转移
```

## 四、Pod 资源最佳实践

### 4.1 Request 与 Limit 规则

```yaml
# ✅ 正确：同一比例，Limit >= Request
resources:
  requests:
    cpu: "1"
    memory: "1Gi"
  limits:
    cpu: "2"
    memory: "2Gi"

# ❌ 错误：不设 limits（Pod 可能耗尽节点资源）
resources:
  requests:
    cpu: "1"
    memory: "1Gi"
  # limits 未设置 - 禁止

# ❌ 错误：Limit 远大于 Request（资源超卖风险高）
resources:
  requests:
    cpu: "0.5"
    memory: "512Mi"
  limits:
    cpu: "8"      # Request 的 16 倍，不合理
    memory: "16Gi"
```

**推荐比例：**

| 服务类型 | CPU Limit/Request 比 | Memory Limit/Request 比 |
|---------|---------------------|------------------------|
| AI 推理 | 1:1 ~ 2:1 | 1:1 ~ 2:1 |
| 业务后端 | 2:1 ~ 4:1 | 2:1 ~ 3:1 |
| 网关代理 | 2:1 | 2:1 |
| 数据管道 | 1:1 ~ 2:1 | 1:1 |
| 存储中间件 | 2:1 | 2:1 |

### 4.2 亲和性调度

```yaml
# GPU 服务：调度到 GPU 节点
spec:
  template:
    spec:
      nodeSelector:
        nvidia.com/gpu: "true"
      tolerations:
      - key: nvidia.com/gpu
        operator: Exists
        effect: NoSchedule

# 有状态服务：避免同节点部署
spec:
  template:
    spec:
      affinity:
        podAntiAffinity:
          preferredDuringSchedulingIgnoredDuringExecution:
          - weight: 100
            podAffinityTerm:
              labelSelector:
                matchExpressions:
                - key: app
                  operator: In
                  values:
                  - postgresql
              topologyKey: kubernetes.io/hostname
```

### 4.3 存储资源规格

| 存储类型 | 访问模式 | 性能等级 | 推荐场景 | 备份策略 |
|---------|---------|---------|---------|---------|
| 云盘 SSD (阿里云) | ReadWriteOnce | 高 IOPS | 数据库、有状态服务 | 自动快照每日 |
| 云盘 ESSD | ReadWriteOnce | 极高 IOPS | PostgreSQL、Milvus | 自动快照每日 |
| NFS | ReadWriteMany | 低 IOPS | 模型文件、共享配置 | rsync + 快照 |
| MinIO | - | 中 IOPS | 备份、日志归档 | 跨站复制 |

## 五、资源规格审批流程

### 5.1 规格变更流程

```
申请（开发者/SRE）
  → 提交 ResourceQuota 或资源配置变更
  → 说明变更理由（性能数据支撑）

评估（SRE 负责人）
  → 评估集群剩余容量
  → 检查是否超出命名空间配额
  → 评估对现有服务的影响

审批
  - CPU/Memory 变更在配额范围内：SRE 自主审批
  - GPU 变更或超出配额：SRE 负责人 + 技术负责人审批
  - 新增存储超过 500Gi：技术负责人审批

落地
  - 通过 CI/CD 或 kubectl apply 执行
  - 变更后 24h 观察资源使用趋势
```

### 5.2 资源浪费治理

```bash
# 巡检：查找 Request 远高于实际使用的 Pod
# CPU 利用率持续低于 20% 且 Request > 1 的 Pod
kubectl top pods -A --sort-by=cpu

# 内存浪费：Memory Request > 4Gi 但实际使用 < 1Gi
kubectl top pods -A --sort-by=memory

# 空闲 GPU：GPU Pod 无需分配但长期占用
kubectl get pods -A -l nvidia.com/gpu

# 未设置 Request/Limit 的 Pod（违反规范）
kubectl get pods -A -o json | \
  jq '.items[] | select(.spec.containers[].resources.requests == null) | .metadata.name'
```

**告警阈值：**

| 指标 | Warning | Critical |
|------|---------|----------|
| CPU 利用率 < 20% 持续 7 天 | 通知 owner | 缩容建议 |
| Mem 利用率 < 30% 且 Request > 2Gi | 通知 owner | 降配建议 |
| GPU 空闲 > 24h | 通知 owner | 回收 GPU 资源 |
| Pod 未设置 Resource | 警告 | 阻止部署（门禁）|
