# 环境运维 体系建设总览

> 环境运维是保障各套环境（dev/pre/prod）持续健康、配置合规、资源合理使用的工程实践。运维不应该只是"救火"，而应该通过系统性巡检和自动化检查，把问题消灭在萌芽阶段。

---

## 一、现状与痛点

### 1.1 核心子域评估

| 子域 | 当前状态 | 主要痛点 | 影响 |
|------|---------|---------|------|
| **集群管理** | 🟡 K8s 3套环境（dev/pre/prod）在运行 | 无常态化巡检，节点资源使用率超标靠告警发现 | 资源问题发现滞后 |
| **命名空间治理** | 🔴 命名空间无 ResourceQuota | 单个 namespace 可能耗尽集群资源 | 资源争抢，影响其他服务 |
| **配置合规** | 🔴 无自动合规检查 | K8s 配置安全基线（readOnly/非 root/资源 limits）靠 review，漏检率高 | 配置漂移，安全风险 |
| **Pod 异常诊断** | 🟡 人工 kubectl describe | Pod OOMKilled/CrashLoopBackOff 排查靠运维经验，10-60 分钟 | 故障响应慢 |

### 1.2 典型问题场景

**场景一：dev 环境资源耗尽波及 pre**
开发在 dev namespace 跑了一个性能测试，没有 ResourceQuota 限制，CPU 请求打满，导致 pre namespace 调度失败，测试流水线卡住整整一天。

**场景二：配置漂移未被发现**
某服务为了快速调试，直接在 prod 执行了 `kubectl set env`，修改了环境变量，但没有同步到 GitLab 的 Helm values。下次部署覆盖后问题消失，但一周后再次部署该变量被还原，相同问题复现，排查花了 2 小时。

**场景三：CrashLoopBackOff 靠人工翻日志**
Pod 进入 CrashLoopBackOff 状态，运维执行 `kubectl describe` → `kubectl logs --previous` → 分析错误，耗时 10-60 分钟，还需要熟悉该服务的人介入。

## 二、各子域建设方案

### 2.1 集群巡检自动化

**Popeye 集群巡检**（每日定时执行）：

```bash
# Popeye：K8s 集群健康检查工具
popeye --context prod-cluster --namespace prod --save --output json \
  > /tmp/popeye-report.json

# 输出结果分类：
# - A/B 级：健康
# - C/D 级：有风险（如 Pod 无资源 limits、Liveness Probe 缺失）
# - E/F 级：严重问题（CrashLoop/镜像拉取失败等）
```

**巡检 + AI 摘要流程**：

```
每日 09:00 定时 CronJob 运行 Popeye
  ↓ 生成 JSON 报告
  ↓ 聚合 E/F/D 级问题（过滤噪音）
  ↓ Claude API：分析问题清单 + 输出优先级排序 + 处置建议
  ↓ 推送飞书（"今日集群健康摘要"，含 Top 5 需处理问题）
```

**目标效果**：运维每天只需 5 分钟 review 飞书摘要，替代 20-40 分钟人工巡检。

### 2.2 命名空间治理（ResourceQuota）

**标准资源配额模板**：

```yaml
# 按 namespace 类型分配配额
apiVersion: v1
kind: ResourceQuota
metadata:
  name: ns-quota
  namespace: dev-<service-name>
spec:
  hard:
    requests.cpu: "4"
    requests.memory: 8Gi
    limits.cpu: "8"
    limits.memory: 16Gi
    pods: "50"
    persistentvolumeclaims: "10"
```

| Namespace 类型 | CPU Request | Memory Request | 说明 |
|--------------|------------|--------------|------|
| dev-* | 4 核 | 8 Gi | 开发调试用，可弹性 |
| pre-* | 8 核 | 16 Gi | 等比缩小的生产环境 |
| prod-* | 按服务容量规划 | 按服务容量规划 | 由架构评审决定 |

### 2.3 配置合规（Kyverno）

**关键合规策略（Kyverno Policy）**：

```yaml
# Policy 1: 禁止以 root 用户运行容器
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: disallow-root-user
spec:
  validationFailureAction: enforce
  rules:
    - name: check-runAsNonRoot
      validate:
        message: "容器必须以非 root 用户运行，请设置 runAsNonRoot: true"
        pattern:
          spec:
            containers:
              - securityContext:
                  runAsNonRoot: true
```

**检查项清单（Phase 0 必须通过）**：

| 检查项 | Kyverno 策略 | 违规处置 |
|-------|------------|---------|
| 容器非 root 运行 | `disallow-root-user` | Block |
| 设置资源 Limits | `require-resource-limits` | Block |
| 镜像必须来自内部 Harbor | `restrict-image-registries` | Block |
| 禁止使用 latest 镜像 tag | `disallow-latest-tag` | Block |
| Liveness/Readiness Probe 必填 | `require-probes` | Warn |

### 2.4 Pod 异常诊断 AI 辅助

**自动化诊断流程**：

```
Prometheus 告警：pod_status{phase="CrashLoopBackOff"} > 0
  ↓ Alertmanager Webhook → 诊断脚本
  ↓ 自动采集：
     - kubectl describe pod <name> -n <ns>
     - kubectl logs <pod> --previous --tail=100
     - kubectl get events -n <ns> --field-selector involvedObject.name=<pod>
  ↓ Claude API 分析：
     - 识别错误类型（OOMKilled / 配置错误 / 依赖服务不可达 / 代码异常）
     - 给出处置建议（扩内存 / 检查 Secret 配置 / 回滚版本）
  ↓ 飞书通知 OnCall（含 AI 诊断摘要 + 建议操作）
```

## 五、建设路径

### Phase 0（第 1-2 周）：安全基线

| 任务 | 具体行动 | 验收标准 | Owner |
|------|---------|---------|-------|
| ResourceQuota 全覆盖 | 为所有 dev/pre/prod namespace 配置资源配额 | 无 namespace 缺少 ResourceQuota | 运维 |
| Kyverno 部署 + 4 条核心 Policy | Helm 安装 Kyverno，配置非 root/资源 limits/Harbor 镜像/非 latest 4 条 Block 策略 | 新 Deployment 不符合规范时自动拒绝 | 运维 |
| 现有服务合规修复 | 审查现有服务，修复违规配置 | Kyverno 无 Block 违规存量 | 运维 + 研发 |

### Phase 1（第 3-4 周）：巡检与诊断自动化

| 任务 | 具体行动 | 验收标准 | Owner |
|------|---------|---------|-------|
| Popeye 每日巡检 | K8s CronJob 运行 Popeye + 结果存储 | 每日 09:00 自动执行 | 运维 |
| AI 巡检摘要 | Popeye JSON → Claude API → 飞书推送 | 飞书可收到日报，Top5 问题准确识别 | 运维 |
| Pod 异常自动诊断 | 告警 Webhook → 上下文采集 → Claude → 飞书 | CrashLoop 告警触发后 3 分钟内收到 AI 诊断 | 运维 |

### Phase 2（第 5-8 周）：配置治理完善

| 任务 | 具体行动 | 验收标准 | Owner |
|------|---------|---------|-------|
| kube-score CI 集成 | 在 GitLab CI 中运行 kube-score，评分 < 8 时 Warning | CI 流水线可查到 kube-score 报告 | DevOps |
| 配置变更自动追踪 | 关键 ConfigMap/Secret 变更自动推送飞书 | 生产配置变更 30 分钟内有通知记录 | 运维 |
| 多环境配置 diff | 自动比对 dev/pre/prod 三环境关键配置差异 | 发布前自动生成差异报告 | 运维 |
