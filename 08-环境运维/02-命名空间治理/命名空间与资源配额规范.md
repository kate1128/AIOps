# 命名空间与资源配额规范

> 解决当前所有服务堆在同一 namespace 导致资源争用和可观测性差的问题。

---

## 一、标签规范

所有 K8s 资源（Deployment、Service、Pod 等）必须携带以下标签：

| 标签 Key | 含义 | 示例 |
|---|---|---|
| `app` | 服务名 | `vllm-qwen-7b` |
| `team` | 负责团队 | `ai-infra`, `backend` |
| `env` | 部署环境 | `prod`, `pre`, `dev` |
| `owner` | 直接负责人（RTM）| `zhangsan` |
| `version` | 服务版本 | `v1.2.0` |

```yaml
# 在所有 Deployment metadata.labels 中添加
labels:
  app: scheduler
  team: backend
  env: prod
  owner: zhangsan
  version: v2.1.0
```

## 二、命名空间规划

## 三、ResourceQuota 配置

### ai-infra

```yaml
apiVersion: v1
kind: ResourceQuota
metadata:
  name: ai-infra-quota
  namespace: ai-infra
spec:
  hard:
    requests.cpu: "32"
    requests.memory: "128Gi"
    limits.cpu: "64"
    limits.memory: "256Gi"
    pods: "50"
    persistentvolumeclaims: "20"
```

### observability

```yaml
apiVersion: v1
kind: ResourceQuota
metadata:
  name: observability-quota
  namespace: observability
spec:
  hard:
    requests.cpu: "8"
    requests.memory: "32Gi"
    limits.cpu: "16"
    limits.memory: "64Gi"
    pods: "30"
```

## 四、LimitRange 配置（防止 Pod 不设 limits）

```yaml
apiVersion: v1
kind: LimitRange
metadata:
  name: default-limits
  namespace: ai-infra
spec:
  limits:
  - type: Container
    default:
      cpu: "1"
      memory: "2Gi"
    defaultRequest:
      cpu: "0.5"
      memory: "512Mi"
    max:
      cpu: "32"
      memory: "128Gi"
```

## 五、迁移策略

**当前状态**：所有服务在同一 namespace（生产为 `default` 或业务 ns）。

**迁移步骤（低风险）**：

1. **先打标签，不移动服务**（零风险）
   ```bash
   kubectl label deployment <name> -n <current-ns> team=ai-infra env=prod --overwrite
   ```

2. **新建 namespace + quota**（不影响现有服务）

3. **新服务直接部署到新 namespace**（不动老服务）

4. **老服务在下次版本发布时迁移**（结合 CI/CD 变更，降低单次风险）

## 六、验证配额使用情况

```bash
# 查看 namespace 配额使用
kubectl describe resourcequota -n ai-infra

# 查看各 namespace 资源使用汇总
kubectl top pods -A --sort-by=memory

# 查看是否有 Pod 因配额不足无法创建
kubectl get events -n ai-infra | grep -i quota
```
