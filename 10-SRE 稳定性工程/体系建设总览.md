# SRE 稳定性工程 体系建设总览

> SRE 的本质是用工程化方法管理系统可靠性：用数据（SLO/错误预算）代替主观判断，用流程（OnCall/变更管理）代替救火文化，用复盘（故障分析）代替责任追究。目标不是零故障，而是在可控的不可靠性范围内交付业务价值。

---

## 一、现状与痛点

### 1.1 核心子域评估

| 子域 | 当前状态 | 主要痛点 | 影响 |
|------|---------|---------|------|
| **SLO 与错误预算** | 🔴 无 | 没有定义服务可用性目标，无法量化稳定性水平 | 稳定性改进无方向，客户 SLA 承诺无数据支撑 |
| **告警体系** | 🟡 Prometheus 基础告警 | 告警有效率低（< 30%），OnCall 疲劳 | 告警被忽视，真实问题漏报 |
| **OnCall 值班** | 🔴 无正式制度 | 靠个人微信/钉钉自发响应，无轮换机制 | 关键人依赖，新人无法参与 |
| **故障复盘** | 🟡 偶尔复盘 | 无标准模板，复盘质量参差，行动项未追踪 | 相同问题重复出现 |
| **变更管理** | 🔴 无 | 生产变更无审批记录，回滚方案缺失 | 变更引入问题时难以快速回滚 |
| **混沌工程** | 🔴 无 | 从未主动验证系统故障恢复能力 | 真实故障时暴露未知弱点 |

### 1.2 典型问题场景

**场景一：OnCall 靠喊人**
某次凌晨 2 点生产告警，因为没有 OnCall 轮值，通知到全员群，半小时才有人响应。靠经验最丰富的那个人登录排查，没有 Runbook 参考，全靠个人知识。

**场景二：同一个故障复现 3 次**
"Redis 连接数耗尽"这个问题在 6 个月内出现了 3 次。每次复盘不了了之，没有追踪行动项落地，下次同样情况下再次出现。

**场景三：变更后故障无法快速确认**
发布后出现问题，不确定是这次发布引起的还是其他原因。因为没有变更记录，排查方向不明，多花了 30-60 分钟才确认是发布引入的问题。

## 二、各子域建设方案

### 2.1 SLO 定义与错误预算

**核心服务 SLO 定义模板**：

```yaml
# 以 AI 推理服务为例
service: vllm-inference
slos:
  - name: availability
    target: 99.5%          # 月度可用率目标（允许 ~3.6h 故障/月）
    indicator:
      metric: "sum(rate(http_requests_total{job='vllm',status!~'5..'}[5m]))
               / sum(rate(http_requests_total{job='vllm'}[5m]))"

  - name: latency_p99
    target: 95%             # 95% 的请求 P99 < 5s
    indicator:
      metric: "histogram_quantile(0.99, rate(http_request_duration_seconds_bucket{job='vllm'}[5m]))"
      threshold: 5s

error_budget:
  window: 30d
  burn_rate_alert:
    - severity: critical   # 1 小时内燃烧 > 2% 的月度错误预算
      threshold: 14.4      # burn_rate > 14.4x 触发 P0 告警
    - severity: warning    # 6 小时内燃烧 > 5%
      threshold: 6.0
```

**错误预算看板（Grafana）**：
- 剩余错误预算（百分比 + 绝对时间）
- 最近 30 天错误预算消耗趋势
- 各 SLO 当前状态（绿/黄/红）

### 2.2 OnCall 值班制度

**值班周期**：每周轮换，周一 09:00 交接

**值班人员职责**：
1. 响应告警（P0 < 15min，P1 < 1h）
2. 执行 Runbook 处置
3. 超出处置能力时 escalation 到相关研发
4. 记录处置过程（写入故障票）

**飞书 OnCall 机器人接入**：

```
告警触发 → Alertmanager → 飞书 OnCall 机器人
  飞书消息格式：
  - 告警级别 + 服务名
  - 告警摘要 + 建议 Runbook 链接
  - 认领按钮（点击后记录响应人 + 时间）
  - 升级按钮（自动通知备份 OnCall）
```

### 2.3 Runbook 体系（AI 推荐）

**Runbook 目录结构**：

```
10-SRE 稳定性工程/流程模板/runbooks/
├── vllm-oom.md          # vLLM OOM 处置
├── db-connection-exhausted.md  # DB 连接耗尽
├── kafka-consumer-lag.md       # Kafka 堆积
├── pod-crashloop.md            # Pod 崩溃循环
└── high-error-rate.md          # 接口高错误率
```

**每个 Runbook 标准结构**：
1. 告警触发条件
2. 快速确认步骤（3-5 条命令）
3. 常见根因及处置方法
4. 升级条件
5. 恢复验证步骤

**AI 辅助 Runbook 推荐**：

```
告警触发 → Claude API 分析告警名称 + 当前指标
  → 自动匹配最相关的 Runbook
  → 飞书消息附带 Runbook 直链 + AI 快速摘要
```

### 2.4 故障复盘（Blameless Postmortem）

**标准复盘模板**（发生 P0/P1 故障后 48h 内完成）：

```markdown
## 故障摘要
- 故障时间：[开始] ~ [结束]（持续 XX 分钟）
- 影响范围：[受影响服务/用户数/业务影响]
- 严重程度：P0 / P1

## 时间线（Timeline）
| 时间 | 事件 |
|------|------|
| HH:MM | 告警触发 |
| HH:MM | OnCall 开始排查 |
| HH:MM | 根因确认 |
| HH:MM | 修复上线 |
| HH:MM | 服务恢复 |

## 根因分析（5 Why）
- Why 1: ...
- Why 2: ...
- Why 3: ...（根本原因）

## 行动项
| 行动 | Owner | 截止日期 | 状态 |
|------|-------|---------|------|
| 修复 X | @ | YYYY-MM-DD | 🔴 待开始 |

## 经验教训
...
```

**AI 辅助故障复盘**：
- 输入：时间线数据（告警日志 + 操作记录）
- 输出：结构化复盘初稿（Claude 生成），人工 review 并补充
- 效果：2-4h 手工撰写 → 30-60min AI 辅助

### 2.5 变更管理

**变更分类与审批矩阵**：

| 变更类型 | 审批级别 | 必须条件 | 回滚方案 |
|---------|---------|---------|---------|
| 配置调整（非关键参数）| 自审 | 有对应 GitLab Issue | 原配置有版本控制 |
| 服务部署（正常迭代）| Lead 审批 | CI 全绿 + 测试通过 | helm rollback |
| 数据库 DDL | DBA + Lead 审批 | pre 环境验证 + 可逆脚本 | 有回滚 SQL |
| 生产紧急修复（Hotfix）| 事后补审 | 同时通知 OnCall 值班人 | git revert |

## 五、建设路径

### Phase 0（第 1-2 周）：制度化

| 任务 | 具体行动 | 验收标准 | Owner |
|------|---------|---------|-------|
| SLO 定义 | 为核心 3 个服务定义 Availability + Latency SLO | Grafana SLO Dashboard 可查剩余错误预算 | SRE Lead |
| OnCall 制度 | 建立值班轮换表，配置飞书 OnCall 群 + 告警路由 | P0 告警 15 分钟内有人认领 | SRE Lead |
| Runbook 补全 | 为 Top 5 高频告警撰写 Runbook | 告警飞书卡片有 Runbook 链接 | SRE |

### Phase 1（第 3-4 周）：AI 辅助

| 任务 | 具体行动 | 验收标准 | Owner |
|------|---------|---------|-------|
| 告警 AI 研判 | 集成 Claude API，告警卡片附带 AI 分析 + Runbook 推荐 | 告警卡片含 AI 建议，值班满意度提升 | SRE |
| 故障复盘 AI 辅助 | 下次 P0/P1 故障使用 AI 生成复盘初稿 | 复盘耗时 < 1h（AI 辅助）| SRE Lead |
| 变更管理规范 | 建立变更审批流程，生产变更记录到飞书表格 | 所有生产变更有记录 | SRE + 研发 |

### Phase 2（第 5-8 周）：体系化运营

| 任务 | 具体行动 | 验收标准 | Owner |
|------|---------|---------|-------|
| 错误预算月度报告 | 每月自动生成错误预算消耗报告 + AI 解读 | 月度报告推送飞书 | SRE Lead |
| 混沌工程试点 | 在 pre 环境做第一次混沌实验（随机 Pod 删除）| 混沌实验有记录，系统自动恢复 | SRE |
| Runbook 质量提升 | 基于 OnCall 使用反馈迭代 Runbook | Top 5 Runbook 有人使用记录 + 反馈 | SRE |

## 六、关键指标

| 指标 | 当前值 | Phase 2 目标 | 采集方式 |
|------|--------|-------------|---------|
| 核心服务可用率 | 未统计 | > 99.5% | Prometheus SLO 指标 |
| P0 告警平均响应时间（MTTA）| > 30min | < 15min | 告警触发 → 认领时间差 |
| P0/P1 平均恢复时间（MTTR）| 未统计 | < 2h | 故障开始 → 恢复时间差 |
| 有效告警率 | < 30% | > 70% | 人工确认有效告警数 / 总告警数 |
| 复盘完成率 | < 50% | 100%（P0/P1 必做）| P0/P1 工单 → 复盘文档 |
| 错误预算月消耗 | 未统计 | < 50%/月（稳定期）| Grafana 错误预算面板 |
