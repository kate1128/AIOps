# AI 提效计划 - SRE 稳定性工程

> 将 AI 嵌入 SRE 核心工作流：智能告警研判、OnCall Runbook 自动推荐、故障复盘辅助、变更风险预测。目标：P0/P1 告警响应时间减少 50%，OnCall 新人上手周期从 1 个月缩短到 1 周。

---

## 一、效率基线（现状）

| 环节 | 当前耗时（估算）| 参与人 | 主要痛点 |
|------|--------------|--------|---------|
| 告警研判（确认是否真实故障）| 5-20 分钟/个 | OnCall 工程师 | 告警上下文不完整，需手动拼凑信息 |
| 故障定位（找到根因）| 30 分钟-4 小时 | SRE + 研发 | 跨系统关联分析效率低，新人不熟悉 |
| Runbook 查找与执行 | 5-15 分钟/次 | OnCall 工程师 | Runbook 分散，搜索效率低 |
| 故障复盘报告撰写 | 2-4 小时/次 | SRE | 时间线还原耗时，Action Items 容易遗漏 |
| 变更风险评估 | 0.5-1 小时/次 | 变更执行者 | 凭经验判断，评估不系统 |

## 三、高价值机会点详细方案

### 机会1：AI OnCall 研判助手

**当前状态**：OnCall 工程师收到告警后，需要手动打开 Grafana 看图、查日志、判断真实性，信息分散耗时长。  
**目标状态**：告警触发时，AI 自动聚合相关上下文（指标/日志/最近变更），给出研判结论和初步行动建议。

**方案设计**：
```
告警触发 → AI 研判助手 → 飞书推送（含诊断摘要）

AI 自动聚合（告警触发后 1 分钟内完成）：
  1. 告警指标的 Grafana 截图链接（最近 30 分钟）
  2. 相关服务的最近异常日志（Loki 查询）
  3. 最近 2 小时内的发布/变更记录（GitLab API）
  4. 该告警的历史记录（过去 30 天触发频率和处置结果）

AI 研判输出示例：
  "🚨 [P1] vllm-qwen 推理服务请求失败率超 10%
   
   📊 当前状态：
   - 失败率：15%（过去 5 分钟），正常时 < 1%
   - 受影响 Pod：vllm-qwen-0（Running），vllm-qwen-1（CrashLoopBackOff）
   
   🔍 可能根因（AI 推断）：
   1. [高概率] 14:32 有一次 Deployment 更新（commit abc123），
      时间与异常完全吻合
   2. [中概率] GPU 显存使用率 47.8GB/48GB，接近上限
   
   📋 建议操作：
   Step 1: kubectl rollout undo deployment/vllm-qwen -n prod
   Step 2: 确认回滚后失败率下降
   Step 3: 查看 abc123 变更内容，分析根因
   
   🔗 Grafana: [链接] | Runbook: [链接] | 相关变更: [链接]"
```

**工具栈**：Alertmanager Webhook + Prometheus API + Loki API + GitLab API + Claude API + 飞书  
**前置条件**：Alertmanager 已配置，Prometheus 指标覆盖核心服务；有 ≥ 2 周历史告警数据  
**实施周期**：2-3 周（需打通多个数据源）  
**ROI 估算**：告警研判时间从 5-20 分钟减少到 2-5 分钟；OnCall 新人也能快速行动

### 机会2：SRE 知识库 AI 助手（Runbook 查询）

**当前状态**：Runbook 和 SOP 分散在飞书文档/本仓库/个人脑子里，搜索效率低，新人不知道去哪找。  
**目标状态**：OnCall 工程师用自然语言描述问题，AI 立即推荐对应 Runbook 并给出操作指引。

**方案设计**：
```
用户（飞书消息）：
  "Redis 内存使用率超过 90% 怎么处理？"

AI 助手响应：
  "参考 Runbook：Redis 内存告警处置
   
   快速判断：
   1. 检查是否有 key 异常增长：redis-cli --bigkeys
   2. 查看 eviction 情况：INFO stats | grep evicted_keys
   
   处置方案（根据判断选择）：
   A. 临时缓解：调高 maxmemory（注意影响）
   B. 根本处理：找到大 key/无 TTL 的 key 并清理
   C. 紧急：检查是否有代码 bug 导致无限写入
   
   相关文档：[Redis 参数调优规范] [中间件告警 Runbook]"

实现方式：
  Dify 部署 RAG 知识库：
  - 知识库来源：本仓库所有 Runbook/SOP/FAQ 文档
  - 飞书 Bot 接入 Dify API
  - 支持上下文对话（追问细节）
```

**工具栈**：Dify（RAG）+ 飞书 Bot  
**前置条件**：核心 Runbook 已整理（至少覆盖 TOP 10 告警类型）  
**实施周期**：1-2 周（Dify 部署 + 知识库录入）  
**ROI 估算**：Runbook 查找时间从 5-15 分钟降至 1-2 分钟；新人 OnCall 能力大幅提升

### 机会3：故障复盘报告 AI 辅助生成

**当前状态**：P0/P1 故障后，复盘报告撰写需要 SRE 花 2-4 小时还原时间线、分析根因、制定 Action Items。  
**目标状态**：故障结束后，AI 自动从监控数据和操作记录中还原时间线，生成复盘草稿，人工聚焦根因分析和行动项确认。

**方案设计**：
```
故障结束后触发：

自动数据采集：
  1. 告警历史（Alertmanager：告警触发时间/处置时间）
  2. 关键指标时序（Prometheus：故障前后 2 小时）
  3. 日志异常时间线（Loki：第一个 ERROR 出现时间）
  4. 操作记录（飞书消息记录/GitLab 操作日志）

AI 生成复盘草稿：
  # 故障复盘 - {服务名} - {日期}
  
  ## 时间线（AI 自动还原）
  - HH:MM 首次异常日志出现（Loki）
  - HH:MM 告警触发（Alertmanager）
  - HH:MM OnCall 开始响应（飞书消息）
  - HH:MM 根因定位完成
  - HH:MM 恢复操作执行
  - HH:MM 服务恢复正常
  
  TTD（检测时间）：X 分钟
  TTR（恢复时间）：X 分钟
  
  ## 根因分析（待人工填写）
  AI 推断：...（附上证据链）
  
  ## Action Items（AI 建议，待人工确认 Owner 和截止日期）
  1. [ ] 增加该类故障的早期检测告警
  2. [ ] 更新 Runbook 增加该场景处理步骤
  3. [ ] ...
```

**工具栈**：Alertmanager API + Prometheus API + Loki API + Claude API  
**前置条件**：Alertmanager 有历史告警数据；日志已接入 Loki  
**实施周期**：2-3 周  
**ROI 估算**：复盘报告从 2-4 小时减少到 30-60 分钟（人工只需补充根因和确认 Action Items）

### 机会4：变更风险 AI 预测

**当前状态**：变更前风险评估靠工程师经验，容易遗漏关键风险点，变更引发的故障是生产故障的主要来源。  
**目标状态**：提交变更申请时，AI 自动评估风险等级并给出检查清单。

**方案设计**：
```
变更描述输入 → AI 风险评估 → 输出风险矩阵 + 建议

AI 评估维度：
  1. 变更范围（影响哪些服务、多少用户）
  2. 变更类型（代码/配置/基础设施/数据库 DDL）
  3. 变更时机（业务高峰/低谷）
  4. 历史相似变更的故障率（基于复盘记录）
  5. 回滚难度和时间

输出：
  风险等级：🟡 中风险
  主要风险点：
  - 数据库 DDL 变更（ALTER TABLE）在高并发时可能锁表
  - 变更在业务高峰期（14:00），建议推迟到低谷期（22:00）
  
  必须完成的检查项：
  [ ] pre 环境已验证 DDL 执行时间 < 5 分钟
  [ ] 已准备回滚 SQL（ALTER TABLE 反向操作）
  [ ] 告警已设置，变更后监控 30 分钟
  [ ] 有回滚决策时间点（变更后 15 分钟内无法恢复则回滚）
```

**工具栈**：飞书审批工作流 + Claude API  
**前置条件**：变更管理流程已建立；有历史故障复盘记录  
**实施周期**：1-2 周  
**ROI 估算**：变更引发故障减少 30%；变更评估时间减少 50%

## 四、实施路径

> ⚠️ **注意**：当前 OnCall 制度未建立、SLO 未定义、告警体系部分覆盖，AI 能力需在完成[体系建设总览](./体系建设总览.md) Phase 1 基础建设后才能有效发挥。

### Phase 0（第 1 周）：知识库助手快速上线

| 任务 | 具体行动 | 验收标准 | Owner |
|------|---------|---------|-------|
| 整理 TOP 10 Runbook | 选取最高频的告警类型，每个写一份 Runbook | 10 份 Runbook 完成，写入知识库 | SRE |
| Dify 知识库助手 | 部署 Dify，接入 Runbook 文档，飞书 Bot 接入 | 飞书可查询到 Runbook 内容 | 运维 + SRE |

### Phase 1（第 2-4 周）：告警智能化

| 任务 | 具体行动 | 验收标准 | Owner | 前置条件 |
|------|---------|---------|-------|---------|
| 告警上下文自动聚合 | Webhook 自动拉取 Grafana + Loki + 变更记录 | 每条 P1 告警附带上下文摘要 | SRE + 运维 | Alertmanager 已配置 ≥ 2 周数据 |
| AI 研判评分 | 告警自动输出真实性评分和建议操作 | 有效告警识别准确率 > 75% | SRE | 上下文聚合完成 |

### Phase 2（第 5-8 周）：复盘和变更自动化

| 任务 | 具体行动 | 验收标准 | Owner | 前置条件 |
|------|---------|---------|-------|---------|
| 复盘报告 AI 生成 | 故障结束后自动触发复盘草稿生成 | 复盘草稿撰写时间 < 1 小时 | SRE | Loki 日志 + 告警历史就绪 |
| 变更风险 AI 评估 | 接入变更审批流程，AI 自动评估风险 | 每次变更有 AI 风险评估结果 | SRE + 运维 | 变更管理流程已建立 |

## 六、风险与回退

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| AI 告警研判误将真实 P0 降级 | 故障响应延迟，影响 SLA | P0 类告警不允许 AI 降级；AI 只做"提升优先级"，不做"降低优先级" |
| 故障时 AI 服务不可用（Claude API 超时）| 无法获得 AI 辅助 | AI 辅助是加分项，不是必须依赖；Claude 超时时降级为人工研判 |
| Runbook 知识库内容过时 | AI 推荐已废弃的操作步骤 | 知识库文档有 last_updated 字段；超过 6 个月未更新的自动提醒 |
| AI 复盘报告时间线错误 | 错误的 TTD/TTR 影响 SLO 计算 | AI 时间线标注为"自动推断"，人工确认后才进入正式记录 |
