# License 授权 体系建设总览

> License 管理是产品商业化的重要控制点：它决定了哪些客户可以使用哪些功能、使用多久、有多少席位。缺乏自动化的 License 体系会导致运营效率低、管控漏洞多、客户体验差（超期无感知、续费提醒不及时）。

---

## 一、现状与痛点

### 1.1 核心子域评估

| 子域 | 当前状态 | 主要痛点 | 影响 |
|------|---------|---------|------|
| **授权设计** | 🟡 基本功能可用 | 授权方案设计复杂，变更时运营人员配置易出错 | License 错误导致客户功能受限 |
| **生成操作** | 🔴 手工流程 | License 生成靠运营人员手工操作，15-30 分钟/次 | 响应慢，客户等待 |
| **到期管理** | 🔴 无自动预警 | License 到期无提前通知，靠客户反馈才知道 | 客户突然断服，影响续费 |
| **使用量监控** | 🔴 无实时监控 | 不知道客户实际使用量，只有客户投诉才发现超量 | 超量使用无法管控 |
| **异常检测** | 🔴 无 | 无法检测 License 共享/跨环境使用等违规行为 | 授权被滥用 |

### 1.2 典型问题场景

**场景一：License 到期客户不知道**
客户 License 到期日到了，系统突然限制使用，客户打电话投诉。运营人员查看台账发现确实到期了，但没有提前通知客户。来回沟通 + 紧急续签花了 2 天，客户体验极差。

**场景二：运营手工生成 License 出错**
运营人员手工填写 License 参数（有效期/功能点/并发数），误填了有效期（2025 年填成 2024 年），导致客户部署后当天就过期。客户反馈后运营重新生成，全程 1 小时沟通。

**场景三：不知道哪些功能被实际使用**
产品经理想了解哪些功能最常用，运营人员无法提供数据，因为系统没有采集 License 使用量指标。

## 二、各子域建设方案

### 2.1 License 生成辅助（Dify 工作流）

**License 生成流程（AI 辅助）**：

```
运营人员填写 Dify 表单：
  - 客户名称
  - License 类型（Standard/Professional/Enterprise）
  - 有效期（从/到）
  - 功能模块（勾选）
  - 最大并发请求数
  - 部署节点数限制

  ↓ Dify 工作流处理：
    - AI 根据客户套餐自动预填大部分参数
    - 对异常参数（如有效期太短）发出警告
    - 生成 License JSON 并签名

  ↓ 生成物：
    - license.json（加密签名）
    - 配套激活说明（含安装步骤）
    - 同步写入 License 台账（飞书多维表格）

效率：15-30 分钟 → 3-5 分钟
```

**License 数据结构**：

```json
{
  "version": "2",
  "customer_id": "cust-20250101",
  "customer_name": "示例科技有限公司",
  "plan": "enterprise",
  "features": ["inference", "finetune", "rag", "workflow"],
  "limits": {
    "max_concurrent_requests": 50,
    "max_nodes": 3,
    "max_users": 100
  },
  "valid_from": "2025-01-01",
  "valid_until": "2026-01-01",
  "issued_at": "2025-01-01T00:00:00Z",
  "signature": "<RSA签名>"
}
```

### 2.2 到期自动预警

**预警层级**：

| 提前天数 | 通知对象 | 通知内容 |
|---------|---------|---------|
| 90 天 | 销售对接人 + CSM | 续约提醒，附客户使用量报告 |
| 30 天 | 销售 + 客户邮件 | 正式续约提醒，附报价 |
| 7 天 | 销售 + CSM + 运营 | 紧急提醒，需立即跟进 |
| 1 天 | 全员相关方 | 高优提醒，当日必须处理 |

**Prometheus + 飞书告警实现**：

```yaml
# License 到期预警告警规则
- alert: LicenseExpiringSoon
  expr: (license_valid_until_timestamp - time()) / 86400 < 30
  labels:
    severity: warning
  annotations:
    summary: "客户 {{ $labels.customer_name }} License 将于 {{ $value | humanizeDuration }} 后到期"
```

### 2.3 使用量监控

**需采集的 License 使用量指标**：

```
license_requests_total{customer_id, feature}  # 各功能调用次数
license_concurrent_requests{customer_id}       # 当前并发数
license_users_active{customer_id}              # 活跃用户数（近 7 天）
license_valid_until_timestamp{customer_id}     # 到期时间戳
```

**月度使用量报告（AI 生成）**：

```
每月 1 日自动：
  Prometheus 查询各客户上月使用数据
    ↓ Claude API 生成使用量分析：
      - 实际使用 vs License 上限（利用率）
      - 使用趋势（增长/下降）
      - 功能使用分布
      - 建议（升级套餐/续费/扩容）
    ↓ 飞书推送给 CSM + 销售（附件 PDF 报告）
```

### 2.4 异常使用检测

**异常行为检测规则**：

```
检测项 1：超量使用
  license_concurrent_requests > license.limits.max_concurrent_requests
  → 告警：超出并发限制，触发限流

检测项 2：跨 IP 激活
  license_activation_ip 与注册 IP 不匹配
  → 告警：可能存在 License 共享

检测项 3：License 文件篡改
  定期验证 RSA 签名
  → 签名验证失败立即吊销
```

## 五、建设路径

### Phase 0（第 1-2 周）：基础规范

| 任务 | 具体行动 | 验收标准 | Owner |
|------|---------|---------|-------|
| License 台账整理 | 将所有现有客户 License 录入飞书多维表格 | 台账完整，包含到期日/功能/联系人 | 运营 |
| 到期告警（基础版）| Prometheus 规则 + 飞书通知（30/7/1 天）| 告警正常触发，CSM 收到通知 | 运维 |
| 生成流程文档化 | 将现有手工流程文档化，减少出错 | 新运营人员可独立完成 License 生成 | 运营 Lead |

### Phase 1（第 3-4 周）：AI 工具化

| 任务 | 具体行动 | 验收标准 | Owner |
|------|---------|---------|-------|
| Dify License 工作流 | 开发 Dify 表单工作流，接入 License 生成逻辑 | 运营通过 Dify 完成一次完整 License 生成 | AI Infra + 运营 |
| 使用量指标采集 | 在应用侧埋点，暴露 Prometheus 指标 | Grafana 可查各客户使用量 | 研发 |
| 异常检测规则 | 超量/跨 IP 告警规则上线 | 测试场景可触发告警 | 运维 |

### Phase 2（第 5-8 周）：全链路自动化

| 任务 | 具体行动 | 验收标准 | Owner |
|------|---------|---------|-------|
| 月度报告自动化 | Prometheus 数据 → Claude 报告 → 飞书/邮件 | 连续 2 个月自动生成报告 | AI Infra + 运营 |
| 90 天预警 | 配置 90 天续约提醒，附使用量摘要 | CSM 反馈提醒有价值，续约率提升 | 运营 |

| 指标 | 当前值 | Phase 2 目标 | 采集方式 |
