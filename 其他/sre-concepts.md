# SRE（站点可靠性工程）概念与实践指南

> SRE = Site Reliability Engineering，源自 Google，用工程化方法保障系统稳定性。

---

## 一、SRE 是什么

### 1.1 定义

SRE（Site Reliability Engineering）是用**软件工程的方法**来保障系统稳定性和可靠性的工程实践。

> 传统运维靠人，SRE 靠代码。

### 1.2 SRE 的核心理念

| 理念 | 说明 |
|---|---|
| **稳定性是可量化的** | 用 SLO/SLI 定义"稳定"的标准 |
| **变更是故障的头号原因** | 通过变更管理降低故障率 |
| **自动化优于人工** | 用代码和平台替代人工操作 |
| **无指责文化** | 故障复盘是为了改进系统，不是找人背锅 |
| **可靠性是产品的属性** | 稳定性不是运维的事，是整个团队的事 |

### 1.3 SRE vs 传统运维

| 维度 | 传统运维 | SRE |
|---|---|---|
| **定位** | "救火队"，被动响应 | "防火体系"，主动预防 |
| **方法** | 人工 + 经验 | 工程化 + 自动化 + 数据驱动 |
| **目标** | 修复问题 | 预防问题 + 量化可靠性 |
| **度量** | 无统一标准 | SLO/SLI/Error Budget |
| **工具** | 脚本 + 命令行 | 平台 + 自动化 + 数据看板 |
| **文化** | 背锅文化 | 无指责复盘 |
| **发布** | 人工审批 + 手动部署 | 自动化 + 灰度 + 可回滚 |

## 二、SRE 的核心工作

### 2.1 SLO/SLI 体系（目标定义）

#### 术语解释

| 术语 | 全称 | 含义 | 例子 |
|---|---|---|---|
| **SLI** | Service Level Indicator | 量化指标 | API 成功率 99.9% |
| **SLO** | Service Level Objective | 目标值 | P99 延迟 < 500ms |
| **SLA** | Service Level Agreement | 对客户的承诺 | 可用性 99.95%，不达标赔偿 |
| **Error Budget** | Error Budget | 允许出错的空间 | 99.9% SLO → 每月允许 43 分钟不可用 |

#### Error Budget 策略

```
Error Budget 充足（> 50%）
  → 可以大胆发布、做实验

Error Budget 紧张（< 20%）
  → 冻结非关键变更，专注稳定性

Error Budget 耗尽（≤ 0%）
  → 停止发布，直到恢复
```

#### SLO 定义示例

| 服务类型 | SLI | SLO |
|---|---|---|
| Web API | 成功率 | ≥ 99.9% |
| Web API | P99 延迟 | ≤ 500ms |
| AI 推理服务 | 推理成功率 | ≥ 99.5% |
| AI 推理服务 | TTFT P99 | ≤ 2s |
| 平台可用性 | 用户可正常访问比例 | ≥ 99.95% |

### 2.2 On-Call 制度（值班响应）

#### 值班架构

```
主 On-Call（一线响应，5 分钟响应）
    ↑ 升级
备 On-Call（二线支持，15 分钟响应）
    ↑ 升级
技术负责人（三线，重大故障介入）
```

#### 值班规范

| 项目 | 要求 |
|---|---|
| 响应 SLA | P0 告警 5 分钟内响应，P1 告警 30 分钟内响应 |
| 升级机制 | 15 分钟内无法定位 → 升级备 On-Call |
| 交接制度 | 每次换班交接当前未解决问题和关注事项 |
| 补偿机制 | 夜班 / 节假日值班给予调休或补贴 |
| 工具 | PagerDuty / Grafana OnCall / 飞书日历 |

#### 告警分级

| 级别 | 响应时间 | 通知方式 |
|---|---|---|
| P0 | 5 分钟 | 电话 + 短信 + 飞书 → 5 分钟无响应升级 |
| P1 | 30 分钟 | 短信 + 飞书 → 30 分钟无响应升级 |
| P2 | 工作时间 | 飞书通知 |
| P3 | 下次迭代 | 邮件 / 工单 |

### 2.3 故障管理（MTTD → MTTR）

#### 故障生命周期

```
发现（MTTD） → 响应 → 定位 → 恢复（MTTR） → 复盘
```

| 阶段 | 目标 | 关键动作 |
|---|---|---|
| **发现（MTTD）** | < 5 分钟 | 监控告警 → On-Call 响应 |
| **响应** | 快速止损 | 确认影响面、启动应急预案 |
| **定位** | 找到根因 | 日志、指标、链路追踪 |
| **恢复** | < 30 分钟 | 修复或回滚 |
| **复盘** | 48 小时内 | 无指责复盘、Action Items |

#### 复盘模板

```markdown
# 故障复盘：[故障标题]

## 基本信息
- 日期：
- 影响范围：
- 影响时长：
- 严重程度：P0 / P1 / P2

## 时间线
- HH:MM 故障开始
- HH:MM 告警触发
- HH:MM On-Call 响应
- HH:MM 定位根因
- HH:MM 开始恢复
- HH:MM 完全恢复

## 根因分析
直接原因：
根本原因：（5-Why 分析法）

## 做得好的
- （应急响应中做得好的地方）

## 待改进的
- （可以改进的地方）

## Action Items
| 编号 | 改进项 | 负责人 | 截止日期 | 状态 |
|---|---|---|---|---|
| 1 | | | | |
```

#### 复盘原则

- **Blameless（无指责）**：目的是找到系统根因并改进，不是追究谁的责任
- **48 小时内完成**：故障恢复后 48 小时内完成复盘文档
- **Action Items 必须跟踪**：改进项录入缺陷管理系统，指定负责人和截止日期

### 2.4 变更管理（控制变更风险）

#### 变更分级

| 级别 | 影响 | 审批 | 例子 |
|---|---|---|---|
| **S（标准）** | 低风险 | 无需审批 | 配置微调、日志级别修改 |
| **N（普通）** | 中等风险 | 团队 Lead | 功能发布、中间件升级 |
| **H（高危）** | 高风险 | 技术负责人 | DB Schema 变更、核心链路改造 |
| **E（紧急）** | 紧急修复 | 事后补审 | 线上 Hotfix |

#### 变更规范

- **变更窗口**：普通变更限制在工作日 10:00-16:00
- **冻结期**：大促 / 重要活动期间冻结变更
- **三板斧**：
  1. **可灰度**：先小范围验证
  2. **可监控**：变更后立即观察指标
  3. **可回滚**：回滚方案提前准备
- **变更记录**：每次变更自动记录到变更日志（对接 CI/CD 和 GitOps）

---

### 2.5 容量管理（预测和预防）

| 工作 | 说明 | 工具 |
|---|---|---|
| **容量规划** | 预测未来 3 个月资源需求 | 历史趋势分析 |
| **自动扩容** | HPA（水平）、VPA（垂直） | K8s HPA / Karpenter |
| **资源优化** | Right-Sizing、闲置资源清理 | Goldilocks / Kubecost |
| **成本归因** | 按 Namespace / Service 统计成本 | Kubecost / OpenCost |

---

### 2.6 自动化（减少人工）

| 人工操作 | SRE 自动化替代 |
|---|---|
| 手动部署 | CI/CD 流水线（GitLab CI + ArgoCD） |
| 手动配置 | GitOps（ArgoCD 自动同步） |
| 手动扩缩容 | HPA / Karpenter 自动调度 |
| 手动故障恢复 | 自动重启 / 自动迁移 / 自动回滚 |
| 手动巡检 | 监控告警 + 自动报表 |
| 手动备份 | CronJob + Velero 自动备份 |

## 三、SRE 的核心目标

### 3.1 稳定性指标

| 指标 | 目标 | 说明 |
|---|---|---|
| **可用性** | ≥ 99.9% | 系统正常运行时间占比 |
| **MTTD** | < 5 分钟 | 平均发现时间 |
| **MTTR** | < 30 分钟 | 平均恢复时间（P0） |
| **变更失败率** | < 5% | 变更引发的故障占比 |
| **告警准确率** | > 80% | 真正需要处理的告警 / 总告警数 |
| **Error Budget** | > 20% | 每月剩余的错误预算 |

### 3.2 SRE 成熟度模型

| 级别 | 特征 | 标志 |
|---|---|---|
| **L1 初始** | 被动响应，无 SLO | 人工运维，故障后修复 |
| **L2 基础** | 有监控和告警 | 能发现故障，响应较慢 |
| **L3 规范** | 有 SLO、On-Call、变更管理 | 量化可靠性，主动预防 |
| **L4 成熟** | 自动化、混沌工程 | 自动恢复，主动注入故障验证 |
| **L5 领先** | AI 辅助、预测性维护 | 预测故障，自动优化 |

## 四、SRE 工具链

| 领域 | 工具 | 用途 |
|---|---|---|
| **监控** | Prometheus + Grafana | 指标采集和可视化 |

| **变更管理** | GitLab CI + ArgoCD | 自动化部署和回滚 |

## 五、落地建议

### 第一周：建立 SLO

1. 选择 3-5 个核心服务
2. 定义 SLI 和 SLO
3. 在 Grafana 建立 SLO Dashboard

### 第二周：建立 On-Call

1. 建立排班表（飞书日历即可）
2. 定义响应 SLA
3. 配置核心告警规则（5 条起步）

### 第三周：变更管理

1. 定义变更分级（S/N/H/E）
2. 建立变更审批流程
3. 写变更单模板

### 第四周：故障复盘

1. 写复盘模板
2. 组织第一次故障复盘（真实或演练）
3. 产出 Action Items 并跟踪

### 持续优化

- **每月**：Review SLO 达成率
- **每季度**：Review On-Call 体验
- **每半年**：做一次混沌工程演练
