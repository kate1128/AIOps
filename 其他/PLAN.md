## Phase 0：准备阶段（第 1-2 周）

> 目标：搞清楚现状、确定技术选型、准备好学习环境

### 要回答的关键问题

| # | 问题 | 为什么要问 |
|---|---|---|
| 1 | 团队有几个人？各自擅长什么？ | 决定哪些事自己做、哪些买服务 |
| 2 | 现在有没有现成的服务器/云账号？ | 决定从零开始还是基于现有资源 |
| 3 | 产品现在什么阶段？有代码了吗？ | 决定 CI/CD 和部署的紧迫程度 |
| 4 | 预算大概多少？ | 决定用托管服务还是自建 |
| 5 | 预计用户规模？ | 决定集群规模和架构复杂度 |

### 技术选型决策（本阶段必须确定）

| 决策项 | 推荐起步方案 | 备选 | 理由 |
|---|---|---|---|
| 云厂商 | 阿里云 | AWS / 华为云 / 私有 IDC | 国内生态好，文档中文 |
| K8s | 托管 ACK | 自建 K3s | 省运维精力 |
| CI/CD | GitLab CI | GitHub Actions | 可私有部署，流水线强 |
| 镜像仓库 | 阿里云 ACR | Harbor | 起步用托管，后期迁自建 |
| 数据库 | 阿里云 RDS PostgreSQL | 自建 | 起步省备份和高可用的心 |
| 缓存 | 阿里云 Redis | 自建 Sentinel | 同上 |
| 监控 | Prometheus + Grafana | 云厂商监控套件 | 开源标准，不锁定 |

### 学习资源（边做边学）

| 领域 | 推荐入门 |
|---|---|
| K8s | 《Kubernetes in Action》+ 官方教程 |
| Docker | Docker 官方 Get Started |
| GitLab CI | GitLab 官方 CI/CD 文档 |
| Prometheus | 《Prometheus: Up & Running》 |
| SRE | Google SRE Book（免费在线版） |

### 本阶段产出

- [ ] 云账号开通，RAM 权限体系设计好
- [ ] 团队分工确定（谁负责哪块）
- [ ] 技术选型确认，写入本文档
- [ ] 各工具的访问权限分配完毕

## Phase 3：可观测性（第 8-14 周）

> 目标：系统"看得见"——指标有图、日志能查、告警能响
> 涉及目录：03-observability、04-ai-service-monitoring
> 前置：Phase 1 的应用已经在跑

### 第 8-10 周：基础监控

| 步骤 | 具体动作 | 验收标准 |
|---|---|---|
| 1.1 | 部署 Prometheus（kube-prometheus-stack） | Prometheus UI 能访问 |
| 1.2 | 部署 Grafana | 能登录，数据源连上 Prometheus |
| 1.3 | 导入基础 Dashboard（K8s/Node/Pod） | 能看到集群资源使用图 |
| 1.4 | 部署 Loki + Promtail | 能查到 Pod 日志 |
| 1.5 | 配置核心告警规则（5 条起步） | 告警能发到你用的 IM 工具 |

**先配这 5 条告警（够了，别贪多）：**

1. 节点 NotReady
2. Pod CrashLoopBackOff 超过 5 分钟
3. 节点磁盘使用率 > 85%
4. API 请求错误率 > 5%
5. 推理服务不可用

### 第 11-12 周：AI 专项监控

| 步骤 | 具体动作 | 验收标准 |

| 2.3 | 建 Token 用量 Dashboard | 按模型/用户看用量趋势 |
| 2.4 | 建推理延迟 Dashboard | TTFT、TPOT、P99 延迟图表 |
| 2.5 | 配置 GPU 温度和 ECC 告警 | 硬件异常能及时通知 |

### 第 13-14 周：链路追踪（可选，但推荐）

## 风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| 团队没 K8s 经验 | Phase 1 延期 | 先用托管 ACK，降低学习曲线 |
| GPU 资源贵 / 难申请 | Phase 2 卡住 | 先用小规格 GPU 开发，生产再上大卡 |
| 监控配了没人看 | Phase 3 白做 | 告警必须接到 IM，Dashboard 每周回顾 |
| 流程太重拖慢研发 | Phase 4 被抵触 | 先轻后重，S 级变更免审批 |
| License 被破解 | 商业损失 | 接受"提高门槛"而非"绝对防破" |
