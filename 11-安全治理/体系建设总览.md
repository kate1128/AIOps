# 安全治理 体系建设总览

> 安全不是一道墙，而是贯穿整个研发和运维生命周期的纵深防御体系。从代码提交、镜像构建、K8s 部署到运行时，每个环节都应有安全检查，而不是仅靠最后一层网关来兜底。

---

## 一、现状与痛点

### 1.1 核心子域评估

| 子域 | 当前状态 | 主要痛点 | 影响 |
|------|---------|---------|------|
| **密钥与凭据安全** | 🟡 K8s Secret 管理 | 偶有凭据写入代码/配置文件，无自动检测 | 凭据泄露导致服务被入侵 |
| **代码与依赖安全** | 🟡 Trivy 镜像扫描已在 CI | 只扫镜像层，代码依赖 CVE 未单独扫；扫描结果无 AI 分析，误报率高 | 已知漏洞未修复 |
| **镜像与供应链安全** | 🟡 Harbor 内部镜像仓库 | 无镜像签名，无法验证镜像完整性；外部镜像直接 pull 未过审 | 供应链攻击风险 |
| **接口鉴权安全** | 🟡 各服务自行实现 | 鉴权标准不统一，部分接口缺少认证；无统一 API Gateway | 鉴权漏洞散落各处 |
| **K8s 配置安全** | 🔴 无自动检查 | 无 Kyverno/OPA，容器可能以 root 运行，无资源限制 | 权限提升/资源滥用风险 |
| **告警与响应** | 🔴 无运行时告警 | 无 Falco/异常行为检测，入侵后难以发现 | 安全事件发现滞后 |

### 1.2 典型风险场景

**风险一：API Key 被提交到代码库**
研发本地测试时将 API Key 写入 `.env` 文件，未添加 `.gitignore`，被提交到 GitLab。即便后来删除，仍在 Git 历史中可查。如果未及时吊销，该 Key 可能被恶意利用。

**风险二：依赖漏洞长期未修复**
Trivy 扫描结果有 200+ CVE，因为没有 AI 辅助分类，运维无法快速判断哪些是高危/可利用的，哪些是误报。结果是所有 CVE 都堆积，没有优先级，没有修复进展。

**风险三：K8s Pod 以 root 运行**
某服务容器以 `root` 身份运行（默认行为，未设置 `runAsNonRoot: true`），如果容器被攻破，攻击者能以 root 权限操作节点文件系统，影响范围极大。

## 二、各子域建设方案

### 2.1 密钥与凭据安全

**Secret 管理规范**：

```
禁止明文凭据出现在：
❌ 代码文件（任何语言）
❌ Helm values.yaml（即使是 templates/）
❌ GitLab CI 配置文件中 echo 打印
❌ Kubernetes Manifest YAML（应引用 Secret 而非明文）

必须使用：
✅ K8s Secret（通过 kubectl create secret 或 External Secrets Operator）
✅ GitLab CI Variables（Masked 类型）
```

**gitleaks 自动扫描（CI 集成）**：

```yaml
# .gitlab-ci.yml 中添加
secret-scan:
  stage: security
  image: zricethezav/gitleaks:v8
  script:
    - gitleaks detect --source . --report-format json --report-path gitleaks-report.json
  artifacts:
    reports:
      secret_detection: gitleaks-report.json
  allow_failure: false  # 发现 Secret 则 CI 失败
```

### 2.2 依赖安全（CVE 分类 AI 辅助）

**当前 Trivy 扫描 + AI 过滤流程**：

```
CI 阶段：Trivy 扫描镜像
  ↓ 生成 JSON 格式 CVE 报告
  ↓ Claude API 过滤分析：
     - 排除误报（已修复的 CVE、操作系统级别低风险）
     - 按 CVSS 分数 + 可利用性分类（Critical/High/Medium）
     - 对 Critical CVE 给出修复建议（升级依赖版本）
  ↓ 输出精简报告（30 条 → 5 条关键项）
  ↓ CI 检查：Critical CVE > 0 则构建失败
  ↓ 飞书通知 Security Lead（含 AI 分析摘要）
```

**目标效果**：误报率从 ~70% 降低到 ~20%，有效 CVE 优先修复。

### 2.3 镜像供应链安全

**内部 Harbor 准入规则**：

```
只允许以下来源的镜像进入生产：
1. harbor.internal/* - 内部构建镜像（经 CI 扫描）
2. harbor.internal/mirror/* - 外部镜像经审核后的镜像（白名单制）

Kyverno 策略：restrict-image-registries
  → 禁止使用非 harbor.internal 的镜像
  → 禁止使用 :latest tag（详见 08-环境运维）
```

**镜像签名（Cosign，Phase 2）**：

```bash
# CI 阶段签名
cosign sign --key $COSIGN_PRIVATE_KEY harbor.internal/app:release_v1.0.0

# 部署前验证签名
cosign verify --key $COSIGN_PUBLIC_KEY harbor.internal/app:release_v1.0.0
```

### 2.4 接口鉴权安全

**鉴权标准化路径**：

```
Phase 1：统一 API Gateway 接入（APISIX）
  - JWT 认证：所有外部接口必须携带 JWT Token
  - API Key 管理：B2B 接口使用 API Key + 限流
  - 内部服务间通信：mTLS（K8s 内网可先用 NetworkPolicy 隔离）

Phase 2：统一鉴权审计
  - 所有鉴权失败请求写入 Loki
  - 异常 IP 或超频请求触发告警
  - 定期审查 API Key 权限最小化
```

### 2.5 K8s 配置安全（Kyverno）

**与 08-环境运维联动**，安全相关 Kyverno Policy 清单：

| Policy | 类型 | 说明 |
|--------|------|------|
| `disallow-root-user` | Block | 容器必须 runAsNonRoot |
| `require-resource-limits` | Block | 容器必须设置 CPU/Memory limits |
| `restrict-image-registries` | Block | 只允许 harbor.internal 镜像 |
| `disallow-privileged-containers` | Block | 禁止特权容器 |
| `require-network-policy` | Warn | 每个 namespace 应有 NetworkPolicy |

### 2.6 运行时安全告警（Falco）

**Falco 关键检测规则**：

```yaml
# 检测容器内 shell 执行（生产环境不应有交互式 shell）
- rule: Terminal Shell in Container
  condition: spawned_process and container and shell_procs
  output: "Shell in container (user=%user.name container=%container.name)"
  priority: WARNING

# 检测敏感文件读取
- rule: Read Sensitive File
  condition: open_read and sensitive_files and not proc_name_exists
  output: "Sensitive file read (file=%fd.name proc=%proc.name)"
  priority: ERROR
```

## 五、建设路径

### Phase 0（第 1-2 周）：基础安全门禁

| 任务 | 具体行动 | 验收标准 | Owner |
|------|---------|---------|-------|
| gitleaks CI 集成 | 在 GitLab CI 中添加 secret-scan stage | MR 中含 Secret 时 CI 自动失败 | DevOps |
| Kyverno 安全策略 | 与 08-环境运维 协同部署，4 条 Block 策略上线 | 特权容器/root 容器无法部署 | 运维 |
| Trivy AI 分类 | 开发 Claude API 分析脚本，接入 CI Trivy 报告 | CI 报告中 CVE 精简为 < 10 条关键项 | 运维 |

### Phase 1（第 3-4 周）：运行时防护

| 任务 | 具体行动 | 验收标准 | Owner |
|------|---------|---------|-------|
| Falco 部署 | Helm 安装 Falco，配置核心检测规则 | Shell-in-container 等异常可触发告警 | 运维 |
| 鉴权规范推进 | 梳理接口鉴权现状，标记无鉴权接口 | 已知无鉴权接口清单 + 修复计划 | 研发 Lead |
| RBAC 审查 | 审查 K8s ServiceAccount 权限，清理宽泛权限 | 无 ClusterAdmin 级别的非必要 ServiceAccount | 运维 |

### Phase 2（第 5-8 周）：体系化安全运营

| 任务 | 具体行动 | 验收标准 | Owner |
|------|---------|---------|-------|
| 安全周报自动化 | Trivy/gitleaks/Falco 数据 → AI 周报 → 飞书 | 每周自动推送安全状态摘要 | 运维 |
| 镜像签名试点 | Cosign 在发布流水线试点签名 | 生产镜像有签名，部署时可验证 | DevOps |
| 渗透测试 | 对外暴露 API 进行一次内部渗透测试 | 高危问题修复，测试报告归档 | 安全 |

## 六、关键指标

| 指标 | 当前值 | Phase 2 目标 | 采集方式 |
|------|--------|-------------|---------|
| CVE 有效发现率（精准度）| ~30%（误报多）| > 80% | AI 分类后 Critical CVE 确认数 / 总数 |
| Secret 泄露检测覆盖率 | 0%（无 CI 扫描）| 100%（每次 MR）| gitleaks CI 执行次数 |
| K8s 配置合规率 | 未统计 | > 98% | Kyverno 违规数 / 总请求数（与 08 共用）|
| Falco 告警误报率 | 无 Falco | < 20% | 人工确认有效数 / 总告警数 |
| Critical CVE 修复时效 | 未追踪 | < 7 天 | CVE 发现日期 → 修复 commit 时间差 |
