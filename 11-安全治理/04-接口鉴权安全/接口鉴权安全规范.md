# 接口鉴权安全规范

## 1. 目标

防止内部服务接口在无鉴权的情况下对外暴露，具体包括：

- 内部 API 被公网直接访问
- 接口未校验身份，任意调用
- 接口返回的响应中包含敏感数据
- 未授权的跨服务调用（东西向）

## 2. 控制项

| ID | 控制项 | 严重度 | 执行阶段 |
| --- | --- | --- | --- |
| C-01 | 所有对外 API 必须经过网关鉴权，禁止直接暴露 Pod/Service | 高 | 架构评审 |
| C-02 | 内部服务 API 默认不挂公网 Ingress | 高 | K8s 部署规范 |
| C-03 | 接口必须携带有效 Token/API Key 才能调用（无鉴权接口需审批豁免） | 高 | 开发规范 |
| C-04 | 错误响应不得暴露内部堆栈、数据库结构等敏感信息 | 中 | 开发规范 |
| C-05 | 服务间调用使用 Service Account + JWT，不使用明文密码 | 中 | 架构规范 |
| C-06 | 敏感接口（涉及 AI 推理、用户数据）须限流并记录访问日志 | 中 | 运维规范 |

## 3. 执行流程

### 3.1 接口暴露检查

新服务上线前检查清单：

```text
服务部署前
  ↓
1. 确认 Ingress 是否必要，内部服务不建 Ingress
2. 网关配置中绑定鉴权插件（JWT / API Key）
3. 接口文档标注哪些是公开接口，哪些需鉴权
4. 用 curl 测试：无 Token 访问是否返回 401/403
  ↓
通过 → 允许上线
不通过 → 修复后重测
```

### 3.2 鉴权方案选型

| 场景 | 推荐方案 |
| --- | --- |
| 前端调用后端 | JWT（网关统一验签） |
| 服务间内部调用 | Service Account + mTLS / 内网白名单 |
| 第三方接入 | API Key（网关管理，带限流） |
| AI 推理接口 | API Key + 请求速率限制 |

### 3.3 敏感数据响应检查

Code Review 中须检查：

- 错误响应是否使用统一格式（code + message），不暴露 stacktrace
- 接口不返回密码、密钥字段（即使是加密后的）
- 分页接口数据量是否有上限限制（防止批量爬取）
- 日志中是否脱敏处理了 Token、手机号、邮箱

### 3.4 API Key 管理规范

前端使用 AI 接口（如 LLM API）时的约束：

```text
禁止：在前端代码直接写入 OpenAI/内部 LLM 的 API Key
正确：前端请求 → 后端代理 → 后端用服务端 Key 调用 LLM
```

这确保 API Key 仅在后端服务端，永远不暴露给浏览器。

## 4. 工具推荐

| 场景 | 工具 | 备注 |
| --- | --- | --- |
| 网关鉴权 | APISIX / Kong / Nginx | 按现有选型 |
| API 接口测试 | Postman / Newman | CI 中自动化 |
| 接口安全扫描 | OWASP ZAP / nuclei | 定期执行 |
| mTLS 服务间认证 | Istio / Linkerd | 中期目标 |

## 5. KPI

| 指标 | 目标值 |
| --- | --- |
| 无鉴权公网暴露接口数 | 0 |
| 新服务上线前完成鉴权检查率 | 100% |
| 接口安全扫描覆盖率 | 大于等于 60%（季度） |
| P0 未授权访问事件 | 0 件/季度 |

## 6. 验收标准

- [ ] 所有 Ingress 资源已审计，内部服务无公网 Ingress
- [ ] 网关鉴权配置标准化，有模板可复用
- [ ] 前端代码中无直接调用 LLM API 的逻辑（后端代理）
- [ ] 已执行过一次接口暴露面排查（nuclei 或手工验证）
