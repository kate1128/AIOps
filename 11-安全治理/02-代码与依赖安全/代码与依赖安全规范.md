# 代码与依赖安全规范

## 1. 目标

在代码提交和 CI 构建阶段发现并阻断：

- 已知 CVE 漏洞依赖包（npm / pip / Maven / Go mod）
- 代码层面的安全缺陷（SQL 注入、XSS、路径遍历等）
- 废弃或不受信任的第三方库

## 2. 控制项

| ID | 控制项 | 严重度 | 执行阶段 |
| --- | --- | --- | --- |
| C-01 | CI 中必须运行依赖漏洞扫描，高危 CVE 阻断构建 | 高 | CI |
| C-02 | 每次 MR 需通过依赖扫描 | 中 | CI |
| C-03 | CRITICAL CVE 必须在 7 天内修复 | 高 | 响应规范 |
| C-04 | HIGH CVE 必须在 30 天内修复 | 中 | 响应规范 |
| C-05 | 生产依赖与开发依赖分离，禁止开发工具包进入生产镜像 | 中 | CR |
| C-06 | 新引入依赖包须在 Code Review 中说明必要性 | 低 | CR |

## 3. 执行流程

### 3.1 依赖漏洞扫描（SCA）

```text
MR 提交
  ↓
CI: dependency-scan job
  ↓
├─ npm audit / pip-audit / mvn dependency-check
├─ Trivy fs 扫描
  ↓
CRITICAL/HIGH CVE 存在 → 阻断 MR
MEDIUM CVE → 生成报告，允许合并但必须建 ticket
```

GitLab CI 示例：

```yaml
dependency-scan:
  stage: scan
  image: aquasec/trivy:latest
  script:
    - trivy fs --exit-code 1 --severity CRITICAL,HIGH .
  allow_failure: false
```

### 3.2 代码安全扫描（SAST）

使用 GitLab 内置 SAST 或 Semgrep：

```yaml
sast:
  stage: scan
  include:
    - template: Security/SAST.gitlab-ci.yml
```

重点检测规则：

- 前端：`eval()` 使用、`dangerouslySetInnerHTML`、不安全的 `postMessage`
- 后端：SQL 拼接、命令注入、路径遍历、不安全反序列化
- 通用：硬编码密钥（与 01-密钥与凭据安全 联动）

### 3.3 漏洞修复流程

```text
扫描发现 CVE
  ↓
1. 确认是否实际可利用（排除误报）
2. 创建 Jira / GitLab Issue（含 CVE ID、影响版本、修复版本）
3. 按严重度分配修复期限（CRITICAL: 7天，HIGH: 30天）
4. 升级依赖包，重新扫描验证
5. 关闭 Issue，更新 SBOM
```

## 4. 工具推荐

| 场景 | 工具 | 备注 |
| --- | --- | --- |
| npm 漏洞 | npm audit / yarn audit | 内置 |
| Python 漏洞 | pip-audit | 推荐，速度快 |
| 通用 SCA | Trivy fs | 多语言支持 |
| SAST | GitLab SAST / Semgrep | 按需选择 |
| SBOM 生成 | Syft | 与镜像扫描联动 |

## 5. KPI

| 指标 | 目标值 |
| --- | --- |
| CRITICAL CVE 修复率 | 7 天内 100% |
| HIGH CVE 修复率 | 30 天内大于等于 90% |
| CI 依赖扫描覆盖率 | 100% 主要仓库 |
| SAST 误报率 | 小于等于 15%（持续优化规则） |

## 6. 验收标准

- [ ] 所有后端/前端主仓库 CI 中包含 dependency-scan job
- [ ] 现存 CRITICAL CVE 已全部修复或有豁免记录
- [ ] 漏洞修复 SLA 已写入研发规范
- [ ] SBOM 已在 Harbor 或 MinIO 中归档
