# 安全治理 FAQ

---

**Q1：发现 API Key 或数据库密码泄露了，立即怎么处理？**

这是 P0 安全事件，立即执行：
1. **立刻吊销泄露的凭据**（阿里云控制台 / GitLab 删除 Access Token / 数据库改密码），不要等排查完再吊销
2. 检查泄露凭据的访问日志，确认是否被非授权方使用
3. 扫描 GitLab 仓库历史 commits（gitleaks `--no-git` 全量扫描），确认泄露范围
4. 重新颁发凭据，通过 K8s Secret 或 Vault 管理，不再明文存储
5. 在故障记录系统中记录事件经过和修复动作

即使没有证据证明被滥用，也必须走完整个吊销流程。

**Q2：Trivy 扫描出来一大堆 CVE，怎么判断哪些必须修？**

按实际风险而非 CVSS 分数判断：

| 优先级 | 条件 | 处理方式 |
|------|------|----------|
| 立即修 | CRITICAL + 有公开利用 PoC + 涉及对外接口 | 阻断 CI，本迭代修复 |
| 本周内修 | HIGH + 影响生产在用组件 | MR + 评审通过后发版 |
| 排期修 | MEDIUM，无已知利用 | 下个迭代 backlog |
| 可接受 | LOW，或受影响代码路径不可达 | 记录例外，定期复查 |

用 Trivy 的 `--ignore-unfixed` 过滤掉没有修复版本的 CVE，减少噪音。

**Q3：K8s 中有 `privileged: true` 的容器，必须改吗？**

是的，privileged 容器等同于宿主机 root，是高危配置。处理步骤：
1. 先统计：`kubectl get pods -A -o json | jq '.items[].spec.containers[].securityContext | select(.privileged==true)'`
2. 逐一确认是否真的需要特权（绝大多数服务不需要）
3. 替代方案：
   - 需要读写宿主机文件 → 用 `hostPath` volume 替代
   - 需要特定 Linux capability → 只授予需要的 `capabilities`，如 `NET_ADMIN`
   - GPU 相关 → HAMI/nvidia-device-plugin 本身需要特权，保留
4. 用 Kyverno Policy 强制拒绝新部署的 privileged 容器

**Q4：内部接口暴露了但没有鉴权，风险有多大？**

风险等级取决于接口功能。高风险场景：
- 内部管理接口（用户管理、权限配置）无鉴权 → 攻击者可提权
- 调试接口（`/debug`、`/actuator`）公网可达 → 泄露内部信息
- LLM API 直接暴露 → 资源滥用（有人免费调用你的 GPU）

快速自查：`kubectl get ingress -A` 列出所有 Ingress，逐一确认是否有 Auth 注解或后端鉴权逻辑。

**Q5：等保合规和日常安全治理是什么关系？**

等保合规是"审计"，日常安全治理是"实际防护"。两者互为支撑：
- 日常安全建设（gitleaks、Trivy、接口鉴权、审计日志）是等保要求的技术实现
- 等保审查会检查这些措施是否落地，且需要证据（日志、配置截图、扫描报告）

安全治理各子目录的规范文档本身就可以作为等保合规证据材料。日常执行有记录，合规审查时不需要临时准备。

---

**Q6：如何反馈安全问题？**

发现安全漏洞或可疑行为：立即通知安全负责人，走安全事件响应流程（[06-告警与响应运营](./06-告警与响应运营/)），不要在公开渠道（群消息、飞书）披露细节，防止信息扩散。
