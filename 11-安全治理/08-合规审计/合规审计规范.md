# 合规与审计治理规范

> 合规是企业的"底线"——满足法律法规和行业标准要求，通过审计证明合规状态。

---

## 一、合规框架

| 合规类别 | 适用标准/法规 | 适用范围 |
|----------|-------------|----------|
| **数据合规** | 个保法、GDPR、网络安全法 | 用户数据处理 |
| **安全合规** | 等保 2.0、ISO 27001 | 信息系统安全 |
| **行业合规** | 生成式AI管理办法、SOC2 | AI 产品 + SaaS 服务 |
| **供应链合规** | 第三方安全评估、SBOM | 软件供应链 |

---

## 二、等保 2.0 合规

### 等级划分

| 等级 | 适用 | 要求概述 |
|------|------|----------|
| **一级** | 不涉及敏感信息的一般系统 | 基本安全防护 |
| **二级** | 涉及敏感信息的一般系统 | 安全审计 + 访问控制 |
| **三级** | 涉及重要信息或大量用户 | 物理/网络/主机/应用/数据 全方位防护 |
| **四级** | 涉及极其重要信息 | 最高等级防护 |

### 等保三级关键控制项

| 控制项 | 要求 | 当前状态 |
|--------|------|----------|
| 身份鉴别 | 双因素认证、密码策略 | 🟡 部分 |
| 访问控制 | 最小权限、角色分离 | 🟡 部分 |
| 安全审计 | 操作日志、审计记录保留 6 个月 | ❌ 待建设 |
| 入侵防范 | 漏洞扫描、入侵检测 | 🟡 部分 |
| 数据完整性 | 传输加密、存储加密 | 🟡 部分 |
| 备份恢复 | 定期备份、恢复演练 | ✅ 已建设 |

## 三、审计日志规范

### 审计记录范围

| 类别 | 记录内容 | 保留期限 |
|------|----------|----------|
| **用户操作** | 登录/登出、权限变更、敏感数据访问 | 6 个月 |
| **系统操作** | 配置变更、服务启停、资源变更 | 12 个月 |
| **数据操作** | 数据导出、删除、批量修改 | 12 个月 |
| **安全管理** | 漏洞发现、安全事件、应急响应 | 3 年 |
| **发布变更** | 版本发布、配置变更、回滚操作 | 永久 |

### 审计日志格式

| 字段 | 说明 | 示例 |
|------|------|------|
| 时间戳 | 操作发生时间 | 2024-01-15T10:30:00+08:00 |
| 操作人 | 用户 ID 或系统账号 | zhangsan@company.com |
| 操作类型 | 操作类别 | DATA_EXPORT / CONFIG_CHANGE |
| 操作对象 | 被操作资源 | /api/v1/users/export |
| 操作结果 | 成功/失败 | SUCCESS / FAILURE |
| 来源 IP | 请求来源 | 10.0.1.100 |
| 详情 | 补充信息 | 导出用户数据 100 条 |

### 审计日志安全要求

- 审计日志**不可篡改**（使用只增存储或区块链摘要）
- 审计日志和管理员权限**分离**
- 审计系统自身操作也需记录
- 定期审计日志完整性检查

## 五、合规改进流程

```
发现合规差距
    │
    ▼
评估风险等级（高/中/低）
    │
    ▼
制定改进计划
    │
    ├── 高/中风险 → 60 天内完成整改
    │
    └── 低风险 → 下个季度内完成
    │
    ▼
实施整改
    │
    ▼
验证整改效果
    │
    ▼
更新合规状态
```

## 七、红线规则

| ❌ 红线行为 | 后果 |
|-------------|------|
| 未经授权导出用户数据 | 严重违规，追责 |
| 审计日志篡改或删除 | 严重违规，立即解职 |

| 隐瞒安全事件不报 | 严重违规，追责 |
| 使用明文密码/密钥 | 违规，立即整改 |
