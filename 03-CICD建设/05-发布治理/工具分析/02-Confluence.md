# Confluence — 团队知识库与文档协作平台

> 技术文档、架构决策、操作手册、Postmortem 的统一存储中心。

---

## 是什么

Confluence 是 Atlassian 出品的团队知识库平台，与 Jira 深度集成，也可以独立使用。它是研发团队记录架构设计、技术规范、运维手册、会议纪要、Postmortem 的标准工具。

注意：如果团队已经使用 GitLab，可以考虑用 GitLab Wiki + MkDocs 替代 Confluence，成本更低且与代码在同一平台。

---

## 核心能力

| 能力 | 说明 |
| --- | --- |
| 富文本编辑 | 所见即所得编辑器，支持表格、代码块、流程图 |
| Space 组织 | 按团队/项目创建独立 Space，权限隔离 |
| 模板 | 内置 Postmortem、架构设计、会议纪要等模板 |
| Jira 集成 | 内嵌 Jira Issue、显示 Epic 进度 |
| 权限控制 | 页面级别读写权限 |
| 版本历史 | 所有变更可追溯，支持回滚 |
| 全文搜索 | 跨 Space 全局搜索 |
| 评论与提及 | @同事，讨论聚焦在文档上 |

## Postmortem 模板

```markdown
# Postmortem：[简短标题]

**日期**：2024-XX-XX
**严重等级**：P0 / P1
**影响时长**：XX 分钟
**影响范围**：XX 用户受影响，XX 个请求失败
**On-Call**：@姓名

## 事故时间线

| 时间 | 事件 |
| --- | --- |
| 14:23 | 告警触发，P99 延迟超过 2s |
| 14:25 | On-Call 确认告警，开始排查 |
| 14:31 | 定位根因：Redis 主节点 OOM |
| 14:38 | 临时处理：扩大内存限制 |
| 14:45 | 服务恢复，告警解除 |

## 根因分析

**直接原因**：Redis 内存配置未跟随业务增长调整。
**根本原因**：缺少 Redis 内存使用率告警，导致问题未被提前发现。

## 影响评估

- 用户可感知的故障时长：22 分钟
- Error Budget 消耗：XX%

## 行动项

| 行动 | 负责人 | 截止日期 |
| --- | --- | --- |
| 添加 Redis 内存使用率告警（> 80% 告警） | @张三 | 本周内 |
| 定期容量评估机制（每月） | @李四 | 下月起 |
| Redis 内存自动扩容方案调研 | @王五 | 两周内 |

## 经验教训

- ✅ 做得好：On-Call 响应及时，定位速度快
- ❌ 需改进：缺少主动容量管理，依赖被动告警
```

## 替代方案（低成本选项）

| 方案 | 优点 | 缺点 |
| --- | --- | --- |
| Confluence | 功能全，与 Jira 集成好 | 收费，有使用惯性门槛 |
| Notion | 界面现代，易上手 | 大文档性能差，搜索弱 |
| GitLab Wiki + MkDocs | 与代码同平台，免费 | 需要自己维护格式 |
| 语雀 | 中文友好，适合国内团队 | 数据在阿里云，私有化贵 |

推荐：已用 Jira 的团队用 Confluence；已用 GitLab 且规模小的团队用 GitLab Wiki + MkDocs 作为轻量替代，免费且够用。
