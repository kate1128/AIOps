# 发布治理规范

发布治理关注的是从“准备发布”到“形成可交付版本”的管理动作，确保版本可追溯、可审批、可回滚、可复盘。

与其他子目录的边界如下：

- 01-CI引擎：负责流水线触发、分支策略、Runner 规划
- 02-制品仓库：负责镜像、JAR、Helm Chart 等制品存储与生命周期
- 03-安全扫描：负责漏洞扫描、签名、SBOM 等安全门禁
- 04-持续部署：负责不同环境的部署执行与回滚
- 本文：负责版本规则、发布审批、交付清单、Release Notes、DORA 回顾

---

## 1. 版本管理

### 版本号规范

采用语义化版本（SemVer）：`MAJOR.MINOR.PATCH`

```text
v2.3.1
│ │ └── PATCH：Bug 修复，向后兼容
│ └──── MINOR：新功能，向后兼容
└────── MAJOR：不兼容的重大变更
```

特殊版本标记：

- `v2.3.1-rc.1`：Release Candidate
- `v2.3.1-beta.1`：Beta 测试版
- `v2.3.1-alpha.1`：Alpha 内测版

### 分支与版本关系

- `develop`：日常开发集成，不产出正式版本
- `release/x.y.z`：发布准备分支，用于集成测试和发布修复
- `main`：正式发布基线，对应可上线版本
- `hotfix/*`：线上紧急修复分支，修复后必须回合并到 `develop`

### 发布流程

```text
1. 创建 release 分支（release/v2.3.0）
2. 在 release 分支上完成集成测试和 Bug 修复
3. 通过测试后合并到 main，打 Tag
4. 触发 CI/CD 自动构建、扫描、归档、部署
5. 输出 Release Notes 和交付清单
6. 合并回 develop 分支
```

## 2. Release Notes 与交付清单

每次正式发布必须包含以下内容：

- 新功能（Features）
- 缺陷修复（Bug Fixes）
- 变更改进（Improvements）
- 已知问题（Known Issues）
- 升级注意事项（Breaking Changes / Migration Steps）

### 交付清单最小集合

| 项目 | 说明 |
| --- | --- |
| 版本号 | 统一版本标识，和 Git Tag、镜像 Tag 保持一致 |
| 制品列表 | 镜像、JAR、Helm Chart、脚本包等清单 |
| 发布说明 | 版本变更摘要和升级说明 |
| 配置变更 | 新增或修改的环境变量、配置项 |
| 数据变更 | 数据库迁移脚本、回滚要求 |
| 验证结果 | 冒烟测试、回归测试、关键功能验证 |
| 文档更新 | 飞书知识库、服务部署文档、运维手册是否已同步 |
| 回滚方案 | 回滚命令、回滚版本、负责人 |

建议产出一个 `release-manifest.yaml` 作为统一交付索引，由 CI 在 Tag 发布时自动归档到制品仓库或 MinIO。

### 飞书知识库更新要求

每次正式发布或影响部署方式的 Hotfix，都必须同步更新飞书知识库中的服务部署文档，至少覆盖以下内容：

- 服务版本号和发布日期
- 镜像地址、Tag 或部署包版本
- 部署步骤或发布顺序的变化
- 新增配置项、环境变量、密钥依赖
- 数据迁移、前置检查、回滚步骤
- 值班联系人和故障排查入口

建议在飞书知识库中为每个服务维护固定页面，发布时只更新对应服务页，避免文档散落在消息和临时表格中。

## 3. 发布审批

发布审批不替代 CI 门禁，而是在进入生产前增加业务与运维视角的风险确认。

### 审批流程

```text
开发完成
  -> Code Review 通过
  -> 自动化测试和安全扫描通过
  -> 填写变更单（变更内容、影响范围、回滚方案）
  -> 技术 Lead 审批
  -> QA 验证签字
  -> 运维确认部署窗口
  -> 执行发布
  -> 线上验证
  -> 更新飞书知识库和服务部署文档
  -> 关闭变更单
```

### 审批工具建议

| 方案 | 用途 |
| --- | --- |
| GitLab Merge Request + Approval Rules | 代码层审批 |
| Jira / 飞书审批流 | 变更单审批 |
| GitLab Protected Environments | 生产部署触发权限控制 |

### 紧急发布（Hotfix）

- 允许走快速通道，由技术负责人单人审批
- 24 小时内补齐变更单、验证记录、复盘结论
- Hotfix 必须同步回合并到 `develop`

## 4. 与制品和部署的衔接

本规范不重复定义制品存储和部署方式，但要求发布时满足以下约束：

- 版本 Tag、镜像 Tag、Release Notes、交付清单必须一致
- 生产发布必须引用不可变版本，禁止使用 `latest`
- 部署前必须能定位到对应制品和对应提交
- 部署后必须能反查本次发布包含哪些 Issue、MR、制品版本
- 发布完成后必须能在飞书知识库中找到最新的服务部署文档和回滚说明

## 5. DORA 与发布复盘

03-CICD 的优化方案已定义 DORA 指标采集方向，这里补充使用口径：

| 指标 | 口径 | 关注点 |
| --- | --- | --- |
| 部署频率 | 成功发布到生产的次数 | 发布节奏是否稳定 |
| 变更前置时间 | 提交到生产可用的总耗时 | 流程是否过长 |
| 变更失败率 | 导致回滚或故障的发布占比 | 发布质量 |
| 故障恢复时间 | 发布故障到恢复完成的耗时 | 回滚与应急能力 |

### 每月发布回顾会建议议题

1. 本月发布次数、失败次数、回滚次数
2. 发布平均耗时和审批平均耗时
3. 高风险变更是否都提前识别并验证
4. 哪些重复手工步骤可以自动化
5. 下月只选择 1-2 个治理改进项推进
