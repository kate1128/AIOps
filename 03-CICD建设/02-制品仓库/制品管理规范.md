# 制品仓库

制品仓库负责存储 CI 阶段产出的所有可部署物（Docker 镜像、JAR 包、Helm Chart），并维护其**版本、生命周期和访问权限**，是 CI 和 CD 之间的中转站。

---

## 当前状态

| 项目 | 状态 | 说明 |
|---|---|---|
| 镜像仓库 | ✅ Harbor 已在用 | 生产/pre/dev 项目隔离 |
| Tag 规范 | 🟡 不完整 | 部分镜像 Tag 不规范（用 latest）|
| JAR 存储 | ❌ 无规范 | 手动归档，无版本追踪 |
| Helm Chart 存储 | ❌ 未建立 | 待 ArgoCD 接入后同步建 |
| 镜像清理策略 | ❌ 未配置 | dev 镜像无限积累，磁盘风险 |
| 代理缓存 | ❌ 未配置 | 拉取 DockerHub 镜像无缓存，受限速影响 |

## 制品类型与存储位置

```
制品类型
├── 源码制品（Git Tag）              — 唯一可追溯源
├── 构建制品
│   ├── Java: JAR                    — 二进制部署用
│   ├── Python: Wheel                — AI 模型服务用
│   ├── Docker Image                 — 容器部署用
│   └── Helm Chart                   — K8s 部署用
├── 部署制品
│   ├── docker-compose.yaml          — Docker 多容器编排
│   ├── docker-run.sh                — 单容器启动脚本
│   ├── .env.template                — 环境变量模板（脱敏）
│   └── ansible-playbook             — 自动化部署剧本
└── 发布包
    ├── release-manifest.yaml        — 多服务版本组合清单
    └── ConfigMap / config.env       — 环境差异配置
```

| 制品类型 | 来源 | 存储位置 | 示例 |
|---|---|---|---|
| Docker 镜像 | docker build | Harbor | `harbor.internal/platform/api:v1.2.0` |
| Helm Chart | helm package | Harbor OCI | `harbor.internal/helm/vllm-service:1.0.0` |
| JAR 包 | mvn package | MinIO | `smartvision-archive/api-service/1.2.0/api-service-1.2.0.jar` |
| 部署脚本包 | CI 打包 | Harbor OCI Artifact | `harbor.internal/deploy/ai-backend:1.2.0` |
| AI 模型权重 | 手动上传 | MinIO（独立管理）| `/models/Qwen2-7B/` |
| 客户交付包 | CI 打包 | FTP | `/releases/smartvision/1.2.0/smartvision-1.2.0.tar.gz` |

## 制品 Tag 规范

```
分支/场景          Tag 格式                              示例
─────────────────────────────────────────────────────────────
dev 分支           dev-{commit_short_sha}                dev-a3f8c21
pre 分支           rc-{semver}                           rc-1.2.0
main / 正式发布    {semver}                              1.2.0
紧急 Hotfix        {semver}-hotfix.{n}                   1.2.1-hotfix.1
```

### JAR 制品存储

```
# 格式：{project-name}-{version}.jar
# 存入 MinIO，路径规则：
smartvision-archive/{project-name}/{version}/
  ├── java-service-1.2.3.jar
  ├── manifest.json          # 版本、commit SHA、构建时间
  └── checksums.sha256

# 示例路径：
smartvision-archive/java-service/1.2.3/java-service-1.2.3.jar
smartvision-archive/java-service/dev-a3f8c21/java-service-dev-a3f8c21.jar
```

### 禁止规则

| ❌ 禁止 | 原因 |
|---|---|
| `latest` Tag | 无法追溯版本来源 |
| 无意义 Tag（`v1`、`test`、`fix`）| 无法识别环境与版本 |
| 手动修改已发布 Tag | 破坏不可变性 |
| 不同构建使用相同 Tag | 覆盖后无法回滚 |

```
CI Build
   │ docker push harbor.internal/platform/api:dev-a3f8c21
   ▼
Harbor（dev 项目）
   │ 30 天后 GC 自动清理
   │
   │ MR 合并到 main，打 rc tag
   ▼
Harbor（pre 项目）
   │ 90 天保留
   │
   │ 发布正式版本
   ▼
Harbor（prod 项目）
   │ 永久保留
   │
   └── 同步到灾备仓库（待建）
```

## 清理策略配置

在 Harbor 项目设置中配置 Tag 保留规则：

```
platform 项目 → Tag Retention：
  规则 1：保留匹配 dev-* 的最近 10 个 Tag
  规则 2：保留匹配 rc-* 的所有 Tag（90天）
  规则 3：保留所有 semver Tag（永久）
  其余：30天后删除
```

## 混合部署场景制品方案

| 服务类型 | 制品形态 | 存储 | 部署方式 |
|---|---|---|---|
| Java 二进制服务 | JAR + version.json | Harbor OCI Artifact / MinIO | Ansible 从 Harbor 拉取 JAR 部署到目标机 |
| ai-backend（Docker）| Docker 镜像 + docker-compose.yaml + docker-run.sh + .env.template | Harbor 镜像 + OCI Artifact | Ansible 在目标机 `docker pull` + `docker compose up -d` |
| scheduler（K8s）| Docker 镜像 + Helm Chart | Harbor | 阶段一：CI kubectl set image；阶段二：ArgoCD GitOps |

#### docker-run.sh 关键规范

```bash
#!/bin/bash
# 版本: ${VERSION}
set -euo pipefail
IMAGE=${IMAGE:-harbor.internal/platform/ai-backend:${VERSION}}
# 敏感变量必须通过 .env 文件注入，禁止硬编码
docker run -d \
  --name ai-backend --restart unless-stopped \
  --env-file ${DATA_DIR}/.env \
  --memory=4g --cpus=2 \
  --log-opt max-size=10m --log-opt max-file=3 \
  ${IMAGE}
```

## 镜像构建优化

### 多阶段构建（Python 服务）

```dockerfile
FROM python:3.11-slim AS builder
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

FROM python:3.11-slim
RUN groupadd -r app && useradd -r -g app app
WORKDIR /app
COPY --from=builder /usr/local/lib/python3.11/site-packages /usr/local/lib/python3.11/site-packages
COPY src/ ./src/
USER app
CMD ["python", "src/main.py"]
```

### 层缓存加速

```yaml
build:
  variables:
    DOCKER_BUILDKIT: 1
  script:
    - docker build \
        --cache-from ${IMAGE_NAME}:dev-latest \
        --cache-to type=inline \
        -t ${IMAGE_NAME}:${IMAGE_TAG} \
        -t ${IMAGE_NAME}:dev-latest .
    - docker push ${IMAGE_NAME}:${IMAGE_TAG}
    - docker push ${IMAGE_NAME}:dev-latest
```

### 基础镜像安全原则

| 措施 | 说明 |
|---|---|
| 固定版本 | 不使用 `latest`，使用 SHA256 摘要锁定 |
| 非 root 运行 | Dockerfile 中创建 `app` 用户 |
| 使用 distroless | 减少攻击面，仅含运行时依赖 |
| 定期重建 | CI 定时任务拉取最新基础镜像重建 |

## 自动化归档

### 归档目录结构

```
archive/
├── java-service/
│   └── 1.2.3/
│       ├── java-service-1.2.3.jar
│       ├── manifest.json          # 版本、commit、构建时间
│       ├── checksums.sha256
│       └── sbom.json              # CycloneDX SBOM
├── ai-backend/
│   └── 1.2.3/
│       ├── ai-backend-1.2.3.tar   # docker save 产物
│       ├── docker-compose-1.2.3.yaml
│       ├── docker-run-1.2.3.sh
│       ├── .env.template
│       ├── manifest.json
│       └── checksums.sha256
└── scheduler/
    └── 1.2.3/
        ├── scheduler-1.2.3.tar
        └── ...
```

### CI 自动归档（Git Tag 触发）

```yaml
archive-artifact:
  stage: archive
  script:
    - docker login $HARBOR_HOST -u $HARBOR_USER -p $HARBOR_PASS
    - docker pull ${IMAGE_NAME}:${CI_COMMIT_TAG}
    # 生成 SBOM
    - syft ${IMAGE_NAME}:${CI_COMMIT_TAG} -o cyclonedx-json > sbom.json
    # 导出镜像并生成校验和
    - docker save ${IMAGE_NAME}:${CI_COMMIT_TAG} -o ${CI_PROJECT_NAME}-${CI_COMMIT_TAG}.tar
    - sha256sum ${CI_PROJECT_NAME}-${CI_COMMIT_TAG}.tar > checksums.sha256
    # 上传到 MinIO
    - mc cp ./${CI_PROJECT_NAME}-${CI_COMMIT_TAG}.tar \
        smartvision-archive/${CI_PROJECT_NAME}/${CI_COMMIT_TAG}/
    - mc cp ./sbom.json \
        smartvision-archive/${CI_PROJECT_NAME}/${CI_COMMIT_TAG}/
  rules:
    - if: $CI_COMMIT_TAG
    - if: $CI_COMMIT_BRANCH == "main"
```

### 保留策略

| 环境 | Harbor 保留 | 对象存储（MinIO）| FTP |
|---|---|---|---|
| dev 构建 | 最近 10 个 Tag / 30 天 | 90 天后删除 | 不上传 |
| pre 构建 | 所有 rc-* / 90 天 | 365 天后删除 | 不上传 |
| 正式发布 | 永久 | 90 天后转冷存储，10 年后删除 | 永久保留（客户可随时下载）|

## FTP 客户交付

> FTP 专用于**正式发布版本**的客户交付，客户私有化部署时从 FTP 下载完整离线包。仅 Git Tag 触发时上传，dev/pre 构建不上传。

### FTP 目录结构

```
/releases/
└── smartvision/
    └── 1.2.0/
        ├── smartvision-1.2.0.tar.gz        # 完整离线包（含所有镜像 + 脚本）
        ├── smartvision-1.2.0.tar.gz.sha256  # 完整性校验
        ├── manifest.json                    # 版本清单（各组件版本号、commit）
        ├── CHANGELOG.md                     # 本版本变更说明
        ├── install/
        │   ├── install.sh                   # 一键安装脚本
        │   ├── .env.template                # 环境变量模板（无敏感值）
        │   └── docker-compose.yaml          # 完整编排文件
        └── images/
            ├── api-service-1.2.0.tar        # docker save 产物（可离线 load）
            ├── ai-backend-1.2.0.tar
            └── ...
```

### manifest.json 格式

```json
{
  "version": "1.2.0",
  "build_time": "2026-05-28T10:30:00Z",
  "git_commit": "a3f8c21d",
  "git_tag": "v1.2.0",
  "components": [
    { "name": "api-service",  "version": "1.2.0", "image": "harbor.internal/platform/api-service:1.2.0" },
    { "name": "ai-backend",   "version": "1.2.0", "image": "harbor.internal/platform/ai-backend:1.2.0" },
    { "name": "frontend",     "version": "1.2.0", "image": "harbor.internal/platform/frontend:1.2.0" }
  ]
}
```

### CI 自动上传 FTP（Git Tag 触发）

```yaml
deploy-ftp:
  stage: release
  script:
    # 1. 拉取所有组件镜像并打包
    - |
      for IMAGE in $RELEASE_IMAGES; do
        docker pull $IMAGE
        NAME=$(echo $IMAGE | awk -F'/' '{print $NF}' | tr ':' '-')
        docker save $IMAGE -o images/${NAME}.tar
      done
    # 2. 打包完整交付包
    - tar -czf smartvision-${CI_COMMIT_TAG}.tar.gz images/ install/ manifest.json CHANGELOG.md
    - sha256sum smartvision-${CI_COMMIT_TAG}.tar.gz > smartvision-${CI_COMMIT_TAG}.tar.gz.sha256
    # 3. 上传到 FTP
    - |
      curl --ftp-create-dirs \
        -T smartvision-${CI_COMMIT_TAG}.tar.gz \
        ftp://${FTP_HOST}/releases/smartvision/${CI_COMMIT_TAG}/ \
        --user ${FTP_USER}:${FTP_PASS}
    - |
      curl --ftp-create-dirs \
        -T smartvision-${CI_COMMIT_TAG}.tar.gz.sha256 \
        ftp://${FTP_HOST}/releases/smartvision/${CI_COMMIT_TAG}/ \
        --user ${FTP_USER}:${FTP_PASS}
    - |
      curl --ftp-create-dirs \
        -T manifest.json \
        ftp://${FTP_HOST}/releases/smartvision/${CI_COMMIT_TAG}/ \
        --user ${FTP_USER}:${FTP_PASS}
  rules:
    - if: '$CI_COMMIT_TAG =~ /^v\d+\.\d+\.\d+/'
      when: manual      # 正式上传需人工确认，防止误推
  environment:
    name: ftp-release
```

所需 CI/CD 变量（在 GitLab 项目设置中配置）：

| 变量名 | 说明 |
|---|---|
| `FTP_HOST` | FTP 服务器地址 |
| `FTP_USER` | FTP 用户名 |
| `FTP_PASS` | FTP 密码（Protected Variable）|
| `RELEASE_IMAGES` | 本次发布的镜像列表（空格分隔）|

### 安全注意事项

| 要求 | 说明 |
|---|---|
| 使用 FTPS（FTP over TLS）| 明文 FTP 传输存在凭证和内容泄露风险，生产必须用 FTPS 或 SFTP |  
| 密码不能硬编码 | `FTP_PASS` 必须设置为 GitLab Protected Variable，不能写在 `.gitlab-ci.yml` 中 |
| 上传后校验 SHA256 | 客户下载后需验证 `.sha256` 文件，确保包完整性 |
| `.env.template` 不含真实密码 | 模板只含占位符，客户部署时自行填写 |
