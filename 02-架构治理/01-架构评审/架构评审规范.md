# 架构评审规范

> 架构评审确保重大技术决策有记录、有共识、可追溯。

---

## 一、评审范围

| 变更类型 | 评审方式 | 说明 |
|----------|----------|------|
| **重大架构变更** | 会议评审 | 微服务拆分、数据库分片、新框架引入 |
| **中等变更** | ADR 异步评审 | 新增组件、接口重设计 |
| **常规变更** | 代码 Review | 功能实现、优化 |

---

## 二、ADR (Architecture Decision Record)

### ADR 模板

```markdown
# ADR-{编号}: {标题}

## 状态
[提议 → 已接受 → 已废弃 → 已替代]

## 背景
{为什么要做这个决策，当前的问题或机会}

## 选项
| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|

## 决策
{最终选择哪个方案及核心理由}

## 影响
{对现有系统、团队、依赖的影响}

## 关联
{关联的其它 ADR、需求、文档}
```

### ADR 管理

- 每个 ADR 有唯一编号
- ADR 存储在 Git 仓库中
- 已废弃的 ADR 保留记录并注明原因
- 定期 Review ADR，确认是否仍然有效

---

## 三、架构委员会

| 角色 | 职责 |
|------|------|
| **架构师** | 主持评审、技术决策 |
| **技术负责人** | 提供实现视角和影响评估 |
| **产品经理** | 提供业务视角和优先级 |
| **QA 负责人** | 评估可测试性和质量影响 |
