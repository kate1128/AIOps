# 架构治理 FAQ

**Q1：架构评审会不会太慢影响迭代？**

评审分两级：常规变更走快速通道(ADR 异步评审)，重大架构变更才需要会议评审。

**Q2：技术选型一定是委员会决定吗？**

不是。团队可以在技术雷达的"候选"区自由探索，但进入"核心"区的技术需要评审。

**Q3：架构债怎么量化？**

用"修复成本"衡量：代码复杂度、测试覆盖缺口、已知设计缺陷数、重构预估人天。

**Q4：API 向后兼容怎么保证？**

CI 中集成 oasdiff 检查，每次 MR 自动对比 OpenAPI 规范文件，有破坏性变更则 MR 失败。故意升级接口时，新增 `/v2` 路径并存或在 `.oasdiff-ignore` 登记原因后 CI 可通过。详见 [工具分析/oasdiff.md](./工具分析/oasdiff.md)。

---

**Q5：SonarQube 扫出来一堆问题，团队根本处理不完怎么办？**

接入初期只上报不拦截（`allow_failure: true`），Quality Gate 只检查新增代码而不追溯历史。存量技术债通过月报 Top 3 机制逐步消减，避免一次性产生大量告警导致团队忽视报告。

**Q6：技术债 AI Agent 发给飞书的结论准确吗，能直接按它说的改代码吗？**

不能直接改。Agent 收到的是 SonarQube 的指标数字（bugs 数、复杂度、技术债时长），生成的是优先级排序和方向性建议，具体重构方案仍需开发人员评估。月报的价值是"强迫团队每月看一眼技术债趋势"，而不是替代 Code Review。

**Q7：oasdiff 只有 OpenAPI 文件提交到 Git 才能用，我们现在是代码注解自动生成的怎么办？**

需要先把生成的 OpenAPI 文件纳入 Git：在 CI 生成后执行 `git add api/openapi.yaml` 并提交到仓库。这一步是前置工作，通常半天内可完成。之后 oasdiff 才能做 MR 间的对比。

**Q8：技术债 Agent 用的 Qwen2.5-Coder 部署在 dev 集群，会影响开发环境的 GPU 资源吗？**

Agent 以 K8s CronJob 运行，每月仅触发一次，推理完成后 Pod 立即退出，不长期占用 GPU。可配置在业务低峰（凌晨）执行，并设置低优先级 PriorityClass 确保不抢占其他服务资源。

**Q9：ADR 用 GitHub Copilot 起草，怎么保证内容不泄露公司技术方案？**

GitHub Copilot Chat 的内容策略：Business/Enterprise 版本默认不用对话内容训练模型，且可在组织级别关闭遥测。在 VS Code 中起草时，输入的背景描述和 Copilot 生成的草稿均在本地处理后由开发人员审核，最终归档到内网 Git 仓库。如有严格保密要求，可只描述技术约束而不提具体业务场景。

**Q10：飞书妙记的会议转录内容安全吗？**

飞书妙记数据存储在飞书企业版云端，受飞书数据协议约束。若会议涉及高度敏感内容（如融资、人事），建议关闭妙记录制，仅对常规技术评审会议开启。
