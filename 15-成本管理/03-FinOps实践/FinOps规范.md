# FinOps 实践规范

> FinOps 的核心是让工程、财务和业务团队协作，共同对云支出负责。本规范定义实践流程、角色和工具链。

---

## 一、FinOps 三阶段模型

```
告知（Inform）→ 优化（Optimize）→ 运营（Operate）

告知：看清楚钱花在哪了（成本可视化、归因、分摊）
优化：找到可以节省的地方并执行
运营：形成持续循环，成本管理融入日常工作流程
```

**当前阶段**：处于"告知"向"优化"的过渡期，优先完成成本可视化和基础告警。

---

## 二、角色与职责

| 角色 | 人员 | 职责 |
|------|------|------|
| **FinOps 负责人** | 运维负责人 | 维护成本看板，组织月度 Review，推动优化落地 |
| **资源 Owner** | 各服务研发 Leader | 对本服务的资源成本负责，审批资源扩容申请 |
| **财务对接** | 财务/行政 | 提供预算数据，接收月度成本报告 |
| **管理层** | CTO / 运营负责人 | 审批超出预算的资源，制定成本目标 |

---

## 三、月度 FinOps 会议

### 会议时间
每月第一周，月度成本报告完成后召开（30 分钟）。

### 议程模板

```
1. 上月成本总结（10 min）
   - 总费用与预算对比
   - TOP 5 成本增长项
   - 已落地的优化措施效果

2. 浪费识别与处置（10 min）
   - 本月发现的低利用率/闲置资源
   - 指定负责人和处置截止时间

3. 下月预算确认（5 min）
   - 预期费用变化原因
   - 是否需要调整预算

4. Action Items 确认（5 min）
```

## 四、预算制定流程

### 年度预算

```
每年 12 月完成下一年度预算规划：

1. 运维提供本年度月度成本趋势
2. 产品提供下年度增长预期（用户量、请求量）
3. 技术负责人评估基础设施扩容计划
4. 按季度拆解预算，允许 ±10% 弹性
5. 财务审批，录入预算系统
```

### 预算分级

| 级别 | 月度超支 | 处理方式 |
|------|---------|---------|
| 正常 | ≤ 10% | 无需特别处理，月报记录 |
| 关注 | 10-30% | FinOps 负责人分析原因，次周给出解释 |
| 预警 | 30-50% | FinOps 负责人 + 管理层面谈，制定削减计划 |
| 超限 | > 50% | 立即冻结非必要资源扩容，管理层决策 |

## 五、资源申请与审批流程

### 新增云资源审批

```
申请人填写《云资源申请单》→
  ├── 预期费用 < 5000 元/月：团队 Leader 审批
  ├── 5000-20000 元/月：运维负责人审批
  └── > 20000 元/月：CTO 审批

《云资源申请单》必填项：
- 资源用途（服务名称、功能描述）
- 规格（CPU/内存/GPU/存储）
- 预期使用时长
- 预期月费用（估算）
- 是否有替代方案（如用现有资源）
```

### 临时/测试资源管理

所有临时资源**必须**打 `ttl` 标签，标注销毁日期：

```bash
# 创建临时测试实例时
aliyun ecs CreateInstance \
  --Tag '[{"Key":"ttl","Value":"2026-06-15"},{"Key":"env","Value":"dev"}]'
```

每周执行自动检查，超过 `ttl` 日期的资源自动告警，24h 内未处理则自动关机。

## 六、成本节省追踪

### 优化措施跟踪表

| 措施 | 预期节省 | 实施状态 | 实施日期 | 实际节省 | 负责人 |
|------|---------|---------|---------|---------|--------|
| Dev GPU 下班关机 | ~30%/月 | 🔲 待实施 | - | - | 运维 |
| 清理宿主机裸进程 | ~10%/月 | 🔲 待实施 | - | - | AI Infra |
| 校准 Pod resource request | ~5% | 🔲 待评估 | - | - | 研发 |
| 使用 Spot 实例（非核心服务）| ~40% | 🔲 评估中 | - | - | 运维 |

### 节省效果计算

```
月节省金额 = 优化前费用 - 优化后费用（连续 2 个月平均）
年化节省 = 月节省金额 × 12
ROI = 年化节省 / 实施成本（人力+工具成本）
```
