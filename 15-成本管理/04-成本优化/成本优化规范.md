## 一、快速降本清单（无需架构改动）

### 立即可做（0 成本，节省 10-30%）

| 措施 | 预期节省 | 操作步骤 | 风险 |
|------|---------|---------|------|
| Dev GPU 节点下班关机 | ~30%/月（节点级）| 设置 crontab + 阿里云定时开机 | 低 |
| 删除未挂载的云盘 | 直接节省 | 控制台查询未挂载云盘，确认后删除 | 低 |
| 删除未使用的快照 | 直接节省 | 清理超过保留期的快照 | 无 |
| 删除未绑定的 EIP | 直接节省（EIP 不绑定也收费）| 控制台查询空闲 EIP | 无 |
| 清理 Harbor 中过期镜像 | 存储费用 | 设置镜像保留策略（保留最近 10 个版本）| 低 |

### 1 周内可做（节省 5-15%）

| 措施 | 预期节省 | 操作步骤 | 风险 |
|------|---------|---------|------|
| 迁移宿主机裸进程到 K8s | 合规+节省（减少超用）| 逐个迁移，见 04-AI Infra 基建文档 | 中 |

## 二、资源规格校准

### 问题：Pod Resource Request 虚高

大量服务在申请资源时为了"保险"设置了过高的 Request，导致节点资源被"占用"但实际不使用，新 Pod 无法调度，造成资源浪费。

### 检查命令

```bash
# 查看所有 Pod 的 CPU/Memory Request vs 实际使用
kubectl top pods -A --containers=true 2>/dev/null | head -50

# 用 Goldilocks 自动推荐合适的 Resource Request/Limit
helm install goldilocks fairwinds-stable/goldilocks \
  --namespace goldilocks --create-namespace

# 为需要分析的 namespace 启用 Goldilocks
kubectl label ns prod goldilocks.fairwinds.com/enabled=true

# 查看推荐值
kubectl port-forward svc/goldilocks-dashboard 8080:80 -n goldilocks
# 访问 http://localhost:8080 查看每个服务的推荐规格
```

### 规格调整原则

| 服务类型 | Request 建议 | Limit 建议 |
|---------|------------|----------|
| 无状态 API 服务 | P75 实际使用量 | P99 实际使用量 × 1.5 |
| 批处理任务 | 实际使用量 × 1.2 | 实际使用量 × 2 |
| 内存密集型（如向量模型）| P90 实际使用量 | 实际使用量 × 1.3 |
| GPU 工作负载 | 按实际需求精确设置 | = Request（不超分配）|

### 校准流程

```
1. Goldilocks 分析所有服务，导出推荐值
2. 按服务逐一评估（不要批量修改）
3. pre 环境先改，观察 1 周
4. 生产环境在低流量时段修改
5. 监控修改后的服务稳定性（关注 OOMKill 事件）
```

## 三、预留实例与节省计划

### 适用场景

预留实例适合**持续稳定运行**的资源。评估标准：
- 利用率 > 70%，连续 3 个月以上稳定
- 未来 1 年内不会大幅缩减规模

### 阿里云预留实例（RI）

| 类型 | 折扣 | 条件 |
|------|------|------|
| 标准 RI（1年）| 约 30-40% 折扣 | 付款方式灵活（全预付/半预付/无预付）|
| 标准 RI（3年）| 约 50-60% 折扣 | 适合核心服务 |

**当前建议**：
- 核心 CPU 节点（7 台工作节点中的常驻节点）购买 1 年 RI
- GPU 节点由于型号更新快，暂不建议 RI，评估按需+Spot 组合

### Spot 实例使用策略

Spot 实例（抢占式实例）在特定工作负载中节省 70-80%：

| 适合使用 Spot 的工作负载 | 不适合使用 Spot |
|---------------------|---------------|
| 批量数据处理任务 | 在线推理服务（响应时间敏感）|
| 定时分析任务 | 有状态服务（数据库）|
| Dev/测试环境 | 需要 SLA 保证的服务 |
| 离线模型评估 | |

```yaml
# K8s 中使用 Spot 节点的 Pod 配置
spec:
  tolerations:
  - key: "alibabacloud.com/eci"
    operator: "Exists"
  nodeSelector:
    "node.kubernetes.io/instance-type": "spot"
  # 确保 Pod 可以被驱逐并在其他节点重建
  terminationGracePeriodSeconds: 30
```

## 四、存储成本优化

### MinIO 数据分层

```
热数据（频繁访问）→ 高性能 SSD 存储
温数据（偶尔访问，如近 3 个月）→ 普通 HDD 或云盘
冷数据（归档，如模型旧版本、历史备份）→ OSS 归档存储（费用仅为标准存储的 1/10）
```

MinIO 生命周期规则配置：

```bash
# 设置 lifecycle：30 天后移入冷层（需要 MinIO 支持分层或直接推 OSS）
mc ilm add myminio/models \
  --expiry-days 0 \
  --transition-days 30 \
  --transition-storage-class "GLACIER"
```

# 使用 Harbor 的垃圾回收功能清理未引用的镜像层

## 五、成本优化 OKR

### 示例季度目标

```
Objective：将月均 GPU 成本降低 20% 并提升利用率

KR1：GPU 平均 SM 利用率从当前 ~30% 提升到 > 55%（3 个月内）
KR2：消除所有宿主机裸进程，GPU 成本完全可归因
KR3：Dev GPU 节点通过自动关机节省 ≥ 30% 夜间成本
KR4：生产 CPU 节点 Pod Resource Request 与实际使用偏差 < 20%
```
