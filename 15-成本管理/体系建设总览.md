# 成本管理 体系建设总览

> 成本管理不是省钱，而是把每一分钱花在刀刃上。在 AI 基础设施主导的技术架构中，GPU 资源是最大的成本项，也是最难优化的。目标是在不影响 SLO 的前提下，通过可视化、分析和自动化，持续提升资源利用率。

---

## 一、现状与痛点

### 1.1 核心子域评估

| 子域 | 当前状态 | 主要痛点 | 影响 |
|------|---------|---------|------|
| **GPU 成本** | 🔴 无追踪 | GPU 利用率未统计，空闲 GPU 机器持续计费 | 每月可能有 30-50% GPU 资源闲置浪费 |
| **K8s 资源成本** | 🔴 无 Kubecost | CPU/内存超量申请普遍，实际使用率低 | 资源请求虚高，影响集群调度效率 |
| **云账单分析** | 🟡 月度人工对账 | 账单分析靠人工，季度一次，发现问题滞后 | 成本异常发现周期长（季度级）|
| **成本分摊** | 🔴 无 | 无法按项目/团队分摊成本，无人对成本负责 | 浪费无人追责 |

### 1.2 典型问题场景

**场景一：GPU 服务器闲置 60% 却持续计费**
7 台 A100 服务器的实际 GPU 利用率日均不到 40%，但每台每月费用约 3-5 万元。如果能将利用率提升到 70%+，同样的业务量可以减少 2-3 台机器，节省 6-15 万/月。

**场景二：开发环境资源占用生产配额**
dev namespace 申请了 8 核 16G，但实际使用只有 0.5 核 1G（requests vs actual usage 差 16 倍）。大量 dev Pod 占用了集群调度资源，导致 prod 高峰时扩容缓慢。

**场景三：阿里云账单突增没人发现**
某月引入了一个日志量很大的功能，日志写入 OSS 的成本比上月增加了 40%。但因为是季度对账，3 个月后才发现，白白多花了数万元。

## 二、各子域建设方案

### 2.1 GPU 利用率监控与优化

**指标采集（DCGM Exporter，与 04-AI Infra 共用）**：

```
DCGM Exporter（每台 GPU 节点）→ Prometheus → Grafana Dashboard
指标：DCGM_FI_DEV_GPU_UTIL / DCGM_FI_DEV_MEM_USED
```

**GPU 利用率分层优化策略**：

| 利用率区间 | 状态 | 建议操作 |
|-----------|------|---------|
| > 80% | 高负载 | 评估是否需要扩容 |
| 40%-80% | 健康 | 维持现状 |
| 20%-40% | 偏低 | 检查调度优化空间 |
| < 20% | 闲置 | AI 分析原因，评估缩减 |

**AI 优化建议（月度报告）**：

```
每月 1 日：DCGM 利用率数据 → Claude API 分析：
  - 闲置节点 + 低谷时段识别
  - 推理参数优化建议（batch_size / max_model_len）
  - 混合部署可行性
  → 输出报告（预期节省 + 操作步骤）→ 飞书推送
```

### 2.2 K8s 资源成本分析（Kubecost）

**核心能力**：

- 每个 Pod/Namespace/Deployment 的 CPU/内存请求 vs 实际使用
- 识别超量申请（requests/actual > 3x 告警）
- 按 Namespace 分摊成本（dev/pre/prod/ai-infra）

**自动化浪费扫描**：

```
每周执行 cost_optimizer.py：
  → 找出 requests/limits 比值 > 5 的 Pod
  → 推荐合适的 requests 值（actual_usage * 1.3 缓冲）
  → 生成 MR 草稿更新 Helm values
  → 飞书通知 Owner 审核合并
```

### 2.3 云账单异常检测（当日告警）

```yaml
- alert: CloudBillDailySpike
  expr: cloud_billing_daily_cost > cloud_billing_daily_cost_avg_7d * 1.3
  annotations:
    summary: "当日云费用超过近 7 天均值 30%，请检查异常消费"
```

**月度成本报告**：阿里云账单 API + Claude API 分析增减原因 + 优化建议 + 下月预算估算 → 飞书推送。

### 2.4 成本分摊

| Namespace 类型 | 归属团队 | 分摊比例 |
|--------------|---------|---------|
| dev-* | 研发部门 | 按使用量 |
| pre-* | 研发/测试共担 | 5:5 |
| prod-* | 各产品线 | 按服务归属 |
| ai-infra | AI Infra 团队 | 全额 |

## 三、AI 提效落地

| AI 提效机会 | 落地方案 | 提效幅度 | 实施状态 |
|------------|---------|---------|---------|
| GPU 利用率分析 + 优化建议 | DCGM 数据 → Claude 月度优化报告 | 月度 → 日度感知，节省 20-30% GPU | 🟡 DCGM 已有，报告待建设 |
| 资源浪费扫描 | Kubecost + 自动 MR 优化 requests | 季度人工 → 周度自动 | 🔴 待部署 Kubecost |
| 成本异常检测 | 云账单 API → Prometheus → 当日告警 | 月度发现 → 当日告警 | 🔴 待建设 |
| 月度成本报告 | 多维数据 → Claude 报告 | 2-3h 手工 → 30min 自动 | 🔴 待建设 |

**ROI 估算**：GPU 利用率 40% → 70% 可节省 3-5 台 A100（¥9-25 万/月），Claude API 成本 ~¥200-400/月，ROI >> 100x。

## 五、建设路径

### Phase 0（第 1-2 周）：可见性

| 任务 | 具体行动 | 验收标准 | Owner |
|------|---------|---------|-------|
| GPU 利用率 Dashboard | 基于 DCGM 配置 Grafana | 可查实时 + 7 天趋势 | AI Infra |
| Kubecost 部署 | Helm 安装，配置 Namespace 分配 | Grafana 可查各 Namespace 月度费用 | 运维 |
| 云账单接入 | 账单 API 采集脚本 + Prometheus 推送 | 可查日度账单明细 | 运维 |

### Phase 1（第 3-4 周）：告警自动化

| 任务 | 具体行动 | 验收标准 | Owner |
|------|---------|---------|-------|
| GPU 闲置告警 | 利用率 < 20% 持续 1 小时告警 | 飞书通知 AI Infra | 运维 |
| 云账单异常告警 | 当日费用 > 7 天均值 130% 告警 | 测试可触发 | 运维 |
| 资源浪费扫描 | Kubecost API 周报 | 每周推送超量申请 Pod 列表 | 运维 |

### Phase 2（第 5-8 周）：AI 优化闭环

| 任务 | 具体行动 | 验收标准 | Owner |
|------|---------|---------|-------|
| GPU 优化月报 | DCGM → Claude → 优化建议 → 飞书 | 节省 ≥ 10% | AI Infra |
| 成本分摊月报 | Kubecost + 账单 → 各团队报告 | Lead 可见本团队月度成本 | 运维 |
| 自动优化 MR | 超量申请 → 自动提 MR | 每周 ≥ 1 个 MR | DevOps |
