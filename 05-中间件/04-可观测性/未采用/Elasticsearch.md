# Elasticsearch - 日志存储与搜索
> 与Loki 的对比参考> 当前使用：Loki

---

## 是什么
Elasticsearch 是全文搜索引擎，与Logstash + Kibana 组成经典 ELK 日志栈。相比 Loki ?只索引 Label 不索引内容的设计，ES **索引日志全文**，支持复杂的全文检索、聚合分析和可视化?
---

## 与Loki 的核心区别
| 维度 | Loki | Elasticsearch |
|------|------|---------------|
| **存储引擎** | 对象存储（S3/MinIO）| 本地磁盘 + S3 |
| **索引策略** | 只索引 Label | 索引全部文本（倒排索引）|
| **资源消耗 * | 无 | 高（内存大户，JVM Heap 调优）|
| **全文搜索** | LogQL 过滤 + Label 匹配 | 全文检索 + 高亮 + 评分 |
| **查询能力** | 适合查错、过滤 | 适合分析、聚合、统计 |
| **运维难度** | 低（单二进制）| 高（JVM / 集群 / 分片管理）|
| **硬件成本** | 低（对象存储便宜）| 高（内存和磁盘需求大）|

---

## 引入 Elasticsearch 你能得到什么
| 收益 | 说明 |
|------|------|
| ✅ 全文检索| 搜索任意关键词、模糊匹配、正则，不依赖日志格式 |
| ✅ 聚合分析 | 按字段统计、分组、趋势分析，数据挖掘能力极强 |
| ✅ 生态丰富| Kibana 可视化、Watcher 告警、Machine Learning |
| ✅ 长期成熟 | ES 已存在十余年，大量生产案例和社区支持 |

## 引入 Elasticsearch 的代价
| 代价 | 说明 |
|------|------|
| ⛔ 资源消耗巨大| ES 是出名的内存大户，节点起步就要 32GB+ 内存 |
| ⛔ 运维复杂 | JVM 调优、分片管理、索引生命周期、集群监控都需要专业人员 |
| ⛔ 硬件成本高| 同样数据量，ES 的存储成本是 Loki 的 3-5 倍|
| ⛔ 简单场景过度| 如果需求只是查错（看日志找异常），LogQL 足够 |
