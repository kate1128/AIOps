# 中间件 FAQ

---

**Q1：生产数据库应该用阿里云 RDS 托管还是 K8s 内自建？**

生产环境强烈推荐托管 RDS，理由：
- 自动主从复制、自动故障切换，无需人工介入
- 内置自动备份，备份到云存储，恢复操作有 GUI
- 安全补丁、版本升级由云厂商维护
- 避免 K8s 存储（PVC）本身故障导致数据丢失的双重风险

唯一适合 K8s 内自建的场景：私有化交付，客户环境没有云数据库服务。此时用 Helm Chart（如 Bitnami PostgreSQL）+ PVC + 定时备份到 MinIO。

**Q2：Kafka 和 RabbitMQ 当前都在用，应该合并吗？**

不建议强制合并，两者定位有差异：

| | Kafka | RabbitMQ |
|---|---|---|
| 适合场景 | 高吞吐日志/事件流、数据管道 | 任务队列、RPC、需要消息确认的业务消息 |
| 消息保留 | 可回放（按时间/偏移量）| 消费即删 |
| 延迟 | 毫秒级 | 微秒级，延迟更低 |

当前如果已经稳定使用，不要为了"统一"而迁移，迁移成本高且风险大。新业务选型时优先考虑 Kafka（已有运维经验），只在需要任务队列语义时用 RabbitMQ。

**Q3：中间件升级如何操作才不影响业务？**

标准步骤：
1. **Pre 环境先升级**，跑 1-2 周，观察指标
2. **应用兼容性验证**：检查 SDK 版本与新中间件版本兼容矩阵
3. **生产蓝绿切换**（以 Redis 为例）：
   - 启动新版本实例（双写模式）
   - 数据全量同步完成后切流量
   - 观察 24h 后下线旧实例
4. **数据库升级**：小版本（如 PostgreSQL 14 → 14.x）直接滚动升级；大版本（14 → 15）需 `pg_upgrade` 或导出导入

**Q4：Milvus 向量数据库高可用怎么保证？**

Milvus 分布式版本（Milvus Cluster）天然支持高可用，但需要：
- etcd 集群（至少 3 节点）
- MinIO 作为对象存储（Milvus 所有向量数据存在 MinIO 中）
- Pulsar 或 Kafka 作为消息队列

当前如果是 Milvus Standalone 部署，单点故障风险高。建议：启用定时快照备份（`milvus backup` 工具），生产环境评估迁移到 Cluster 模式。

**Q5：中间件连接信息应该放在哪里管理？**

必须通过 K8s Secret 管理，禁止明文写在代码或配置文件中：

```bash
kubectl create secret generic db-credentials \
  --from-literal=host=<rds-endpoint> \
  --from-literal=password='<password>' \
  -n prod
```

同时在 GitLab CI 中接入 gitleaks 扫描，防止凭据误入代码库。详见 [11-安全治理/01-密钥与凭据安全](../11-安全治理/01-密钥与凭据安全/密钥与凭据安全规范.md)。
