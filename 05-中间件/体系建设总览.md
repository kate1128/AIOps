# 中间件 体系建设总览

> 中间件是支撑所有业务服务的技术底座——数据库、消息队列、网关、存储这些水电煤一旦不稳，整个产品体系都会受影响。中间件治理的目标是让这些基础设施从"靠运气"变成"靠工程"。

---

## 一、现状与痛点

### 1.1 核心子域评估

| 子域 | 当前状态 | 主要痛点 | 影响 |
|------|---------|---------|------|
| **数据库** | 🟡 RDS 托管 + K8s 自建混合 | 无连接池（PgBouncer 未部署），高并发时连接数飙升；慢查询无自动分析 | 偶发性 DB 连接耗尽导致服务不可用 |
| **消息队列** | 🟡 Kafka + RabbitMQ 并存 | Kafka Consumer 堆积无自动告警，排查依赖人工经验，耗时 2-6 小时 | 消息堆积无法及时发现，业务延迟 |
| **网关** | 🟡 Nginx Ingress 在用 | 无统一 API 管理，限流/认证各服务自己实现，标准不一 | 接口保护能力参差不齐 |
| **AI 基础设施** | 🔴 Dify 未部署 | 无统一 AI 工作流平台，各服务自己集成 LLM，重复造轮子 | 开发效率低，缺乏统一管控 |
| **可观测接入** | 🟡 监控覆盖约 40% | 大量中间件无 Prometheus Exporter，故障靠人工发现 | 平均 MTTR 偏高 |

### 1.2 典型问题场景

**场景一：DB 连接池耗尽导致服务雪崩**
某次业务高峰期，PostgreSQL 最大连接数（100）被打满，新连接请求全部失败，导致依赖该 DB 的 5 个服务同时报错。排查发现原因是上游某服务连接未正确释放，且无 PgBouncer 兜底。如果有连接池，该问题可以完全避免。

**场景二：Kafka 堆积 6 小时后才被发现**
Kafka 某 Topic 消费者停止消费（应用 Bug 导致），消息持续堆积 6 小时。因为没有堆积告警，直到业务方反映数据延迟才开始排查。恢复后补消费历史堆积又花了 2 小时。

**场景三：慢查询排查半天**
业务反馈某功能响应慢，运维拿到 SQL 后需要手动执行 `EXPLAIN ANALYZE`，分析执行计划，研判是缺索引还是查询写法问题，全程依赖 DBA 经验，耗时 1-4 小时。

## 二、各子域建设方案

### 2.1 数据库治理

**核心目标**：连接池保障高并发稳定性，慢查询自动分析，备份有效性持续验证。

**连接池接入流程（PgBouncer）**：

```
应用服务（多实例）
  ↓ 连接到 PgBouncer（K8s Service: pgbouncer.db.svc.cluster.local:5432）
PgBouncer（池化管理，最大连接数由此控制）
  ↓ 按需建立真实连接
PostgreSQL RDS（max_connections 上限）
```

配置要点：
- `pool_mode = transaction`（事务级复用，适合短连接高并发）
- `max_client_conn = 500`（应用侧最大连接数，比 DB 上限高 5 倍）
- `default_pool_size = 20`（每个用户/数据库的默认连接池大小）

**慢查询 AI 自动分析流程**：

```
pg_stat_statements 采集慢查询（mean_exec_time > 1s）
  ↓ Prometheus postgres_exporter 暴露指标
  ↓ 告警触发（慢查询次数 > 10/min）或每日定时
  ↓ 自动执行 EXPLAIN (ANALYZE, BUFFERS, FORMAT JSON)
  ↓ Claude API 分析执行计划（识别顺序扫描/索引失效/锁等待）
  ↓ 输出优化建议（索引创建 SQL / 查询改写 / 参数调整）
  ↓ 推送飞书（含慢 SQL + 建议 + 预期优化效果）
```

### 2.2 消息队列治理

**核心目标**：Kafka 消息堆积分钟级告警，AI 自动给出根因分类和处置建议。

**Kafka 堆积监控告警流程**：

```
kafka_exporter 采集 Consumer Group Lag 指标
  ↓ Prometheus 存储
  ↓ 告警规则：kafka_consumer_lag > 10000 持续 5min
  ↓ 告警触发 → AI 诊断助手自动聚合上下文：
     - 当前 Consumer Group 状态（STABLE/REBALANCING/DEAD）
     - 消费速率 vs 生产速率趋势（过去 30 分钟）
     - 最近有无应用发布/重启记录（GitLab API）
  ↓ AI 输出根因分类（消费者宕机 / 消费速率不足 / Rebalance 风暴）
  ↓ 附带处置建议 → 飞书通知 OnCall
```

### 2.3 网关规范化（APISIX）

**核心目标**：统一 API 入口，集中管理限流/认证/监控，替代各服务自行实现的分散方案。

**APISIX 接入路径**：

```
Phase 1（准入规范）：
  新接口必须通过 APISIX 暴露，旧接口逐步迁移
  配置规范：rate_limit（100 req/min 默认）+ jwt-auth + prometheus 插件

Phase 2（能力建设）：
  接入 Prometheus，所有路由的 QPS/延迟/错误率可查
  配置灰度路由（canary 插件，按 Header/比例 分流）

Phase 3（API 治理）：
  结合 03-CICD 的 oasdiff，在网关层面检测 API 版本兼容性
```

### 2.4 AI 基础设施（Dify 平台）

**核心目标**：统一 AI 工作流平台，各业务场景复用 LLM 接入层，避免重复开发。

**Dify 覆盖的核心场景**：

| 场景 | 当前实现 | Dify 替代方案 |
|------|---------|------------|
| 知识库问答（RAG）| 各服务自己集成 | Dify 知识库统一管理 + 飞书 Bot |
| CI 失败分析（03-CICD）| 直接调 Claude API | 接入 Dify 工作流，统一鉴权和限流 |
| License 生成辅助（14）| 手动操作 | Dify 表单工作流，运营自助使用 |
| 交付文档生成（13）| 手动撰写 | Dify 模板填充工作流 |

## 五、建设路径

### Phase 0（第 1-2 周）：监控补盲

| 任务 | 具体行动 | 验收标准 | Owner |
|------|---------|---------|-------|
| PgBouncer 部署 | helm install bitnami/pgbouncer 到 db namespace，应用服务切换连接串 | DB 连接数峰值 < max_connections 80%，应用无报错 | 运维 |
| postgres_exporter 补全 | 确保所有 PostgreSQL 实例（含 K8s 内自建）接入 Prometheus | Grafana 可查询所有 DB 连接数/慢查询数/锁等待 | 运维 |
| kafka_exporter 部署 | helm install bitnami/kafka-exporter，接入 Prometheus | Grafana 可查 Consumer Group Lag，告警规则上线 | 运维 |
| 中间件监控覆盖率 | 盘点所有中间件，补充缺失 Exporter | 监控覆盖率从 40% → 80% | 运维 |

### Phase 1（第 3-4 周）：AI 诊断接入

| 任务 | 具体行动 | 验收标准 | Owner |
|------|---------|---------|-------|
| 慢查询 AI 分析 | 开发告警 Webhook + Claude API 分析脚本 + 飞书推送 | 慢查询告警触发后 2 分钟内收到飞书分析报告 | 运维 + 研发 |
| Kafka 堆积 AI 诊断 | 告警触发 → 聚合 Consumer 状态 + 速率数据 → Claude 分析 | 堆积告警触发后 5 分钟内收到根因建议 | 运维 |
| 容量预测配置 | 为磁盘/内存/连接数配置 `predict_linear(30d)` 预测告警 | 到达上限前 7 天提前预警 | 运维 |

### Phase 2（第 5-8 周）：平台化与规范化

| 任务 | 具体行动 | 验收标准 | Owner |
|------|---------|---------|-------|
| Dify 部署 | K8s 部署 Dify，接入内网 vLLM，建立核心工作流 | 知识库问答、CI 分析、License 生成 3 个工作流上线 | AI Infra |
| APISIX 评估 | 在 pre 环境试点 APISIX，迁移 2-3 个核心接口 | pre 环境核心接口经 APISIX 路由，限流/监控可用 | 架构 + 运维 |
| 跨环境配置 diff | 开发自动 diff 脚本 + AI 差异分析 | 每次发布前自动生成三环境配置差异报告 | 运维 |

## 六、关键指标

| 指标 | 当前值 | Phase 2 目标 | 采集方式 |
|------|--------|-------------|---------|
| 中间件监控覆盖率 | ~40% | > 90% | Prometheus 接入中间件数 / 总中间件数 |
| DB 连接池使用率 | 无（未部署 PgBouncer）| < 80% 峰值 | PgBouncer metrics → Prometheus |
| Kafka 堆积发现时延 | 小时级（靠人工）| < 5 分钟 | kafka_exporter 告警触发时间 |
| 慢查询分析时间 | 1-4 小时（人工）| < 15 分钟 | 告警触发 → 飞书收到报告的时间差 |
| 中间件 MTTR | 未采集 | < 30 分钟 | Alertmanager 告警时间差 |
| Dify 工作流覆盖场景数 | 0 | ≥ 5 | Dify 已上线工作流数量 |
