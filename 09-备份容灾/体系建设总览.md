# 备份容灾 体系建设总览

> 备份是预防，容灾是应对。再好的系统也有出问题的时候——磁盘损坏、误删数据、集群灾难——备份容灾体系决定了"最坏的情况下，我们能恢复到什么状态、需要多长时间"。

---

## 一、现状与痛点

### 1.1 核心子域评估

| 子域 | 当前状态 | 主要痛点 | 影响 |
|------|---------|---------|------|
| **数据库备份** | 🟡 RDS 自动备份 7 天 | K8s 内自建 PostgreSQL 无备份策略；备份有效性从未验证 | 自建 DB 数据丢失风险高 |
| **对象存储备份** | 🟡 MinIO 数据有 PVC | PVC 无定期快照；无跨节点副本 | 单点故障导致数据不可恢复 |
| **K8s 集群备份** | 🔴 无 | 无 etcd 备份，无 Velero 配置备份 | 集群故障后无法快速恢复 |
| **AI 模型备份** | 🟡 存在 MinIO | 无版本管理，旧版本随时可能被覆盖 | 模型回滚困难 |
| **容灾演练** | 🔴 从未演练 | 没有验证过 RTO/RPO 目标是否可达 | 灾难时恢复时间未知 |

### 1.2 核心指标定义

| 指标 | 含义 | 目标（Phase 2）|
|------|------|-------------|
| RPO（Recovery Point Objective）| 最多允许丢失多少时间的数据 | 数据库 < 1h；模型 < 24h |
| RTO（Recovery Time Objective）| 从灾难到恢复服务的最长时间 | 核心服务 < 4h；非核心 < 24h |
| 备份有效率 | 可成功恢复的备份比例 | > 99%（每月验证）|

### 1.3 典型风险场景

**风险一：K8s 内自建 PostgreSQL 数据丢失**
私有化环境中，客户 K8s 集群使用 Bitnami PostgreSQL Helm Chart，PVC 存储在本地磁盘。一旦节点磁盘损坏，数据无任何保护。RDS 托管实例有自动备份，但私有化场景完全依赖客户存储。

**风险二：etcd 数据损坏**
etcd 存储 K8s 所有集群状态（Deployment/Service/ConfigMap/Secret）。如果 etcd 数据损坏且没有备份，恢复集群状态需要逐一重新部署所有服务，可能需要 1-3 天。

**风险三：误删 MinIO Bucket**
MinIO 上存储了 AI 模型权重和训练数据集。如果运维误执行 `mc rm --recursive`，且没有版本控制，数据永久丢失。

## 二、各子域建设方案

### 2.1 数据库备份（pgBackRest）

**备份策略**：

```
全量备份：每周日 02:00（保留 4 周）
增量备份：每日 02:00（保留 7 天）
WAL 连续归档：实时，保留 48h（实现 RPO < 1h）

备份目标：MinIO（内网对象存储）
  pgBackRest → s3://backup/postgres/{env}/{db_name}/{date}/
```

**备份 K8s CronJob 示例**：

```yaml
apiVersion: batch/v1
kind: CronJob
metadata:
  name: pg-backup-incremental
  namespace: db
spec:
  schedule: "0 2 * * *"
  jobTemplate:
    spec:
      template:
        spec:
          containers:
            - name: backup
              image: harbor.internal/tools/pgbackrest:2.49
              command:
                - pgbackrest
                - --stanza=prod-db
                - --type=incr
                - backup
```

**备份有效性验证（每月自动）**：

```
每月第一个周六 03:00：
  1. 在 verify namespace 启动临时 PostgreSQL 实例
  2. 从 MinIO 最新全量备份恢复
  3. 执行数据完整性检查（行数对比 + 关键表 checksum）
  4. 记录恢复时间（RTO 测量）
  5. 销毁临时实例
  6. 结果推送飞书（备份有效 ✅ / 失败 ❌ + 失败详情）
```

### 2.2 K8s 集群备份（Velero）

**Velero 覆盖范围**：

```
Velero 定时备份（每日 01:00）：
  - 所有 Namespace 下的 Deployment/Service/ConfigMap
  - PVC 快照（依赖存储插件）
  - K8s RBAC（ServiceAccount/Role/RoleBinding）
  - CRD 和 CRD 实例

备份目标：MinIO s3://backup/velero/{cluster}/{date}/
保留策略：7 天
```

**etcd 独立备份**（每 6 小时）：

```bash
ETCDCTL_API=3 etcdctl snapshot save /backup/etcd-snapshot-$(date +%Y%m%d%H%M).db \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/healthcheck-client.crt \
  --key=/etc/kubernetes/pki/etcd/healthcheck-client.key
```

### 2.3 AI 模型版本管理（MinIO 版本控制）

**MinIO 存储桶配置**：

```bash
# 启用版本控制（防止误删/覆盖）
mc version enable myminio/models

# 对象生命周期：旧版本保留 90 天
mc ilm add myminio/models --expiry-days 90 --noncurrent-version-expiry-days 90
```

**模型命名规范**：

```
models/
├── qwen2.5-coder-32b/
│   ├── v1.0.0/
│   │   └── model.safetensors
│   └── v1.1.0/
│       └── model.safetensors
└── bge-m3/
    └── v1.0.0/
```

### 2.4 容灾演练流程

**季度容灾演练计划**：

```
演练类型 A（数据库恢复演练，每季度）：
  1. 在非生产时段，从备份完整恢复一个业务数据库
  2. 测量 RTO（实际恢复时间）
  3. 验证数据完整性
  4. 记录演练报告

演练类型 B（K8s 命名空间恢复，每半年）：
  1. 完全删除 pre 环境某个 namespace
  2. 从 Velero 备份完整恢复
  3. 验证服务可用性
  4. 记录恢复时间
```

## 五、建设路径

### Phase 0（第 1-2 周）：补盲

| 任务 | 具体行动 | 验收标准 | Owner |
|------|---------|---------|-------|
| MinIO 版本控制 | 对 models/datasets bucket 启用版本控制 | mc version info 显示 enabled | 运维 |
| etcd 定期备份 | CronJob 每 6 小时备份 etcd 到 MinIO | MinIO 中有最近 24h 的 4 个 etcd 快照 | 运维 |
| 备份状态监控 | 为所有 CronJob 配置 Prometheus 监控 + 失败告警 | 备份 Job 失败时 30 分钟内飞书告警 | 运维 |

### Phase 1（第 3-4 周）：K8s 备份

| 任务 | 具体行动 | 验收标准 | Owner |
|------|---------|---------|-------|
| Velero 部署 | Helm 安装 Velero + MinIO Backend，配置每日全量备份 | Velero Backup 任务成功，MinIO 有备份文件 | 运维 |
| pgBackRest 部署 | 为 K8s 内 PostgreSQL 接入 pgBackRest，配置每日增量 | 手动执行一次恢复测试成功 | 运维 |

### Phase 2（第 5-8 周）：验证与演练

| 任务 | 具体行动 | 验收标准 | Owner |
|------|---------|---------|-------|
| 月度自动验证 | 月度恢复测试 CronJob + 飞书通知 | 连续 3 个月验证结果均为 ✅ | 运维 |
| 季度演练 | 完成第一次 DB 恢复演练，AI 生成演练报告 | 演练报告归档到知识库，RTO 数据记录 | SRE |
| RTO/RPO 数据记录 | 每次演练记录实际恢复时间 | 已知 RTO < 4h（核心服务）| SRE |

## 六、关键指标

| 指标 | 当前值 | Phase 2 目标 | 采集方式 |
|------|--------|-------------|---------|
| 数据库备份覆盖率 | RDS 100%；K8s 内 0% | 100% | pgBackRest / Velero 备份 Job 状态 |
| 备份有效率 | 未验证 | > 99%（月度验证）| 月度恢复测试结果 |
| K8s 集群备份频率 | 无 | 每日 + etcd 每 6h | Velero Schedule 执行情况 |
| 实际 RTO（核心 DB）| 未知 | < 4h | 季度演练测量 |
| 实际 RPO（核心 DB）| 未知 | < 1h | WAL 归档时间戳间隔 |
