# AI 模型与数据集备份规范

## 1. 目标

保护生产在用的 AI 模型文件和训练数据集：

- 模型文件不因存储故障丢失
- 历史模型版本可追溯、可回滚
- 大文件备份不阻塞业务存储
- RPO 小于等于 7 天，RTO 小于等于 24 小时

## 2. 保护对象分类

| 类型 | 描述 | 典型大小 | 优先级 |
| --- | --- | --- | --- |
| 生产在用模型 | 当前线上推理使用的模型权重 | 几 GB 到几百 GB | P0 |
| 历史版本模型 | 已发版但未下线的历史版本 | 同上 | P1 |
| 微调模型 | 基于基础模型微调的定制版本 | 几 GB 到几十 GB | P0 |
| 训练数据集 | 用于训练/微调的标注数据 | 几 GB 到几 TB | P1 |
| 模型配置文件 | tokenizer、config.json 等 | MB 级 | P0 |

## 3. 模型版本管理

### 3.1 版本命名规范

```text
<model-name>-<base-version>-<finetune-date>
示例：
  llama3-8b-base              # 基础模型
  llama3-8b-ft-20260501       # 2026-05-01 微调版本
  qwen2-7b-smartvision-v1.2   # 业务定制版本
```

### 3.2 模型元数据索引

每个模型版本维护一份 JSON 索引文件（存入 Git）：

```json
{
  "name": "qwen2-7b-smartvision-v1.2",
  "base_model": "Qwen/Qwen2-7B",
  "finetune_date": "2026-05-01",
  "storage_path": "minio://backup-models/qwen2-7b/v1.2/",
  "size_gb": 14.5,
  "md5": "abc123...",
  "status": "production",
  "deployed_services": ["inference-service-a"]
}
```

## 4. 备份方案

### 4.1 MinIO 冷桶归档

```bash
# 生产模型同步到归档桶（增量模式）
mc mirror --overwrite \
  minio-primary/models/production/ \
  minio-backup/backup-models/

# 验证同步完整性
mc diff minio-primary/models/production/ minio-backup/backup-models/
```

### 4.2 备份调度策略

| 模型类型 | 备份频率 | 保留策略 |
| --- | --- | --- |
| 生产在用模型 | 版本发布时触发 | 永久保留（至少保留近 3 个版本） |
| 微调模型 | 训练完成后触发 | 保留 90 天，重要版本永久 |
| 训练数据集 | 每周增量 | 保留 1 年 |
| 模型配置文件 | 随模型版本 | 同模型版本 |

### 4.3 大文件传输优化

```bash
# 使用分片上传（大于 5GB 的文件）
mc cp --part-size 512MiB model.bin minio/backup-models/

# 并发上传（多文件时）
mc mirror --workers 4 ./models/ minio/backup-models/
```

## 5. 模型恢复流程

```text
1. 确定需要恢复的模型版本（查询元数据索引）
2. 从 MinIO 归档桶下载对应版本文件
   mc cp minio/backup-models/qwen2-7b/v1.2/ ./restored-model/ --recursive
3. 验证文件完整性（MD5 校验）
   md5sum -c checksums.md5
4. 部署到推理服务（更新 vLLM model-path 配置）
5. 验证推理服务正常响应
```

## 6. KPI

| 指标 | 目标值 |
| --- | --- |
| 生产模型备份覆盖率 | 100% |
| 模型文件 MD5 校验通过率 | 100% |
| 模型恢复演练 RTO | 小于等于 24 小时（季度演练） |
| 模型版本索引完整率 | 100% 生产版本 |
