# 容灾演练规范

## 1. 目标

**备份没有经过演练等于没有备份。** 本规范要求：

- 每季度执行一次容灾演练
- 演练覆盖 L1 核心数据的恢复验证
- 记录实测 RTO 并与目标值对比
- 演练结果输出报告，驱动改进

## 2. 演练类型

| 演练类型 | 频率 | 对象 | 影响范围 |
| --- | --- | --- | --- |
| 数据库恢复演练 | 每季度 | 生产数据库副本恢复到测试环境 | 无（测试环境） |
| etcd 快照恢复演练 | 每半年 | 测试集群 etcd 恢复 | 测试集群 |
| PVC 恢复演练 | 每季度 | Velero 恢复到测试命名空间 | 无（测试环境） |
| 模型文件恢复演练 | 每半年 | 从归档桶恢复模型到测试节点 | 无 |
| 全链路演练 | 每年 | 模拟全站灾难，完整恢复 | 预约停机窗口 |

## 3. 演练流程

### 3.1 演练前准备

```text
演练前 1 周
  ├─ 确认演练范围和负责人
  ├─ 准备测试环境（隔离，不影响生产）
  ├─ 确认最新备份文件可用
  └─ 通知相关人员（SRE、研发、运维）
```

### 3.2 演练执行步骤（数据库示例）

```text
1. 记录演练开始时间（T0）
2. 从 MinIO 下载最近全量备份
3. 恢复到测试数据库实例
4. 应用增量备份/binlog 到目标时间点
5. 执行数据完整性验证
   - 核心表记录数对比
   - 关键业务数据抽样校验
6. 记录恢复完成时间（T1），计算 RTO = T1 - T0
7. 与目标 RTO（4 小时）对比
8. 记录演练结果
```

### 3.3 数据完整性验证 checklist

```sql
-- MySQL 验证示例
SELECT TABLE_NAME, TABLE_ROWS
FROM information_schema.TABLES
WHERE TABLE_SCHEMA = 'production_db'
ORDER BY TABLE_ROWS DESC;

-- 与生产数据行数对比（允许误差 < 0.1%）
```

## 5. 演练未通过处理

如果演练 RTO 超标或恢复失败：

1. 不通过演练视为 P1 问题，立即建 Issue
2. 分析根因（备份文件损坏/流程不熟/工具问题）
3. 30 天内完成修复并补充演练
4. 更新恢复文档和操作手册

## 6. KPI

| 指标 | 目标值 |
| --- | --- |
| 季度演练执行率 | 100% |
| 演练 RTO 达标率 | 大于等于 90% |
| 演练报告输出率 | 100% |
| 演练发现问题 30 天关闭率 | 大于等于 80% |
