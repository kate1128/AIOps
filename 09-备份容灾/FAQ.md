# 备份容灾 FAQ

---

**Q1：RTO 和 RPO 是什么？当前目标是多少？**

- **RPO（Recovery Point Objective）**：最大数据丢失时间窗口。比如 RPO = 1h，意味着故障发生时最多丢失 1 小时内的数据。
- **RTO（Recovery Time Objective）**：最大服务恢复时间。比如 RTO = 4h，意味着要在 4 小时内恢复服务。

当前建设目标：

| 数据类型 | RPO 目标 | RTO 目标 |
|------|---------|----------|
| PostgreSQL / Milvus | ≤ 1h（WAL 增量备份）| ≤ 4h |
| Redis | ≤ 24h（每日 AOF 备份）| ≤ 2h（重建可接受）|
| K8s 配置 / etcd | ≤ 6h（快照间隔）| ≤ 8h |
| AI 模型文件 | ≤ 7天（周级归档）| ≤ 24h（从归档恢复）|

---

**Q2：备份验证需要多久做一次？**

- **每次备份后自动验证**：脚本检查备份文件大小是否正常（防空备份），验证 PostgreSQL dump 文件可 `pg_restore` 不报错
- **每月做一次完整恢复演练**：恢复到隔离测试实例，验证数据完整性和业务可用性，记录实测 RTO
- **每季度做一次全链路容灾演练**：包括数据库 + K8s 配置 + 存储，模拟生产环境完全不可用场景

---

**Q3：AI 模型文件体积很大（几十 GB），怎么高效备份？**

不需要每天全量备份，推荐按版本增量归档：
1. 模型文件内容不变，只在上线新版本时归档一次
2. 用 MinIO 的版本控制（Versioning）自动保留历史版本
3. 建立模型索引文档（版本号、路径、大小、上线时间），快速定位需要恢复的版本
4. 长期不用的旧模型版本移入冷存储（MinIO 生命周期规则配置）

---

**Q4：etcd 备份脚本出错，怎么确认备份是否可用？**

```bash
# 验证备份文件完整性（不会触发任何集群操作）
ETCDCTL_API=3 etcdctl snapshot status /backup/etcd-$(date +%Y%m%d).db --write-out=table

# 正常输出包含：Hash、Revision、Total Keys、Total Size
# 若报错则备份文件损坏，立即重新备份并告警
```

每次备份脚本执行后应自动运行此验证命令，将结果（成功/失败）推送到告警通道。

---

**Q5：MinIO 本身出故障，备份数据是不是也丢了？**

是的，MinIO 是单点时存在这个风险。防范策略：
1. **生产 MinIO 开启纠删码（Erasure Code）**，至少 4 节点模式，允许 2 节点故障
2. **备份数据双写到阿里云 OSS**（作为异地冷备），MinIO + OSS 同时挂掉的概率极低
3. 或使用阿里云 OSS 直接作为备份目标，省去 MinIO 高可用运维

---

**Q6：如何反馈问题或建议？**

备份失败告警走 SRE OnCall 流程（P1 级别）；备份策略调整（RPO/RTO 目标修改、新增备份对象）在月度运维会议上提出，更新到对应子目录规范文档。
