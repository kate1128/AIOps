# 产品需求管理 FAQ

---

**Q1：需求太多，怎么判断做哪个？**

用 RICE 模型打分排序：(Reach × Impact × Confidence) / Effort。分数高的先做。建议每个迭代只做 Top 3-5 个需求，保证完成质量。

---

**Q2：P0 紧急需求和迭代计划冲突怎么办？**

P0 需求可以中断当前迭代，但需满足：
1. 确认是真实 P0（产品 + 技术负责人会审）
2. 当前迭代腾出人力，非 P0 需求延后
3. P0 需求上线后补齐复盘记录

---

**Q3：需求评审总是不通过/反复改怎么办？**

常见原因和对策：
- PRD 缺少验收标准 → 模板强制必填
- 技术方案不明确 → 评审前先技术预研
- 业务目标不清晰 → 先回答"怎么衡量做成了"

---

**Q4：如何避免"需求做了但没人用"？**

1. PRD 中必须定义成功标准
2. 上线后 2-4 周做效果回顾
3. 数据不达标的转入优化迭代
4. 持续无人使用的功能考虑下架

---

**Q5：需求变更频繁怎么办？**

1. 区分变更等级，大变更必须重新评审
2. 统计变更原因，是需求没想清楚还是外部变化
3. 持续变更的需求重新评估优先级
4. 下个迭代放入 backlog 重新排队

---

**Q6：需求和 Bug 的优先级怎么对比？**

| 类型 | 优先级对比原则 |
| --- | --- |
| P0 Bug vs P0 需求 | Bug 优先，修完再开发 |
| P1 Bug vs P0 需求 | 评估影响面，通常 Bug 优先 |
| 常规 Bug vs 新需求 | 按 RICE 统一排序 |
