# 产品需求管理 体系建设总览

> 需求管理是产品团队的核心工作流——需求从哪来、怎么评优先级、怎么写 PRD、怎么评审、怎么追踪到上线。任何环节失控都会直接导致研发资源浪费和交付质量下降。

## 一、现有方案分析

> 当前能力评估与问题诊断。
## 一、当前工具与流程

| 项目 | 状态 | 说明 |
| --- | --- | --- |
| 需求收集入口 | ❌ 无统一入口 | 需求来自飞书消息、邮件、口头沟通、客户群，容易遗漏 |
| 需求文档(PRD) | 🟡 有模板但未标准化 | 各产品经理自行编写 PRD，格式不统一，缺少验收标准 |
| 需求评审 | 🟡 有但流程不固定 | 评审参与人员不固定，评审结论常无书面记录 |
| 优先级管理 | ❌ 无框架 | 依赖产品经理个人判断，缺少数据驱动的优先级模型 |
| 路线图(Roadmap) | ❌ 无 | 团队不清楚未来 1-3 个月的规划方向 |
| 需求追踪 | 🟡 部分 | 需求在 Jira 中有记录，但状态更新滞后，跨团队不可见 |
| 需求变更管理 | ❌ 无 | 开发中需求变更没有规范流程，容易导致返工 |
| 上线回顾 | ❌ 无 | 上线后很少复盘需求是否达到预期目标 |
| 与缺陷管理联动 | ❌ 无 | 高频 Bug 不反向驱动需求优化，产品改进缺少数据支撑 |

## 二、核心问题

| 问题 | 影响 |
| --- | --- |
| 需求无统一入口 | 重要需求可能被遗漏，优先级判断不透明 |
| 缺少优先级模型 | 团队做得多但价值产出低，资源分散 |
| PRD 质量参差不齐 | 开发过程中大量沟通返工，实现与原意偏离 |
| 无路线图同步 | 跨部门（销售、客户成功）无法提前规划 |
| 需求变更无管控 | 迭代中途频繁加需求，交付周期失控 |

## 三、本次优化方向

- 建立统一需求收集入口（飞书表单 + 定期评审会）
- 引入 RICE 或 ICE 优先级评估模型
- 标准化 PRD 模板，明确准入准出标准
- 建立季度路线图同步机制
- 需求变更走审批流程，变更必须进 backlog 重新排期

## 二、需求收集入口设计

统一使用飞书多维表格作为需求收集入口：

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| 需求标题 | 是 | 一句话简述 |
| 需求背景 | 是 | 为什么要做，解决什么问题 |
| 期望效果 | 是 | 预期达成的业务目标 |
| 提出方 | 是 | 客户 / 内部 / 产品 / 市场 |
| 关联模块 | 是 | AI / Web / API / Infra |
| 紧急程度 | 是 | 紧急 / 重要 / 一般 |
| 附件/参考 | 否 | 竞品截图、用户反馈原文 |

**收集节奏：**
- 随时可提交，产品经理每周一次整理
- 紧急需求（P0）实时通知产品负责人

## 三、优先级评估：RICE 模型

| 维度 | 说明 | 评分(1-5) |
| --- | --- | --- |
| **Reach（覆盖）** | 影响多少用户或客户 | 1=少于10，5=全部客户 |
| **Impact（影响）** | 对核心指标的提升程度 | 1=微乎其微，5=关键突破 |
| **Confidence（信心）** | 对预期效果的把握程度 | 1=纯猜想，5=数据验证 |
| **Effort（投入）** | 研发人天估算（反向评分） | 1=>40人天，5=<5人天 |

**RICE Score = (Reach × Impact × Confidence) / Effort**

优先级队列按 RICE Score 降序排列，作为迭代排期的主要依据。

## 四、PRD 标准化

### PRD 模板核心结构

```markdown
## 1. 背景与目标
- 业务背景（为什么要做）
- 成功标准（怎么衡量做成了）
- 关联需求/文档

## 2. 需求范围
- 核心功能（必须交付）
- 扩展功能（有条件交付）
- 非目标（明确本期不做）

## 3. 用户流程
- 角色定义
- 用户故事 / Use Case
- 异常流程

## 4. 功能规格
- 页面/接口描述
- 交互逻辑
- 边界条件
- 错误处理

## 5. 验收标准
- 功能验收点（可用于测试用例）
- 性能要求（如有）
- 安全要求（如有）

## 6. 依赖与风险
- 外部依赖
- 技术风险
- 合规风险
```

## 五、需求变更管控

### 变更分类

| 变更类型 | 处理方式 |
| --- | --- |
| **小调整**（UI 文案、字段增减）| 开发自行处理，更新 PRD 备注 |
| **中变更**（新增接口、流程调整）| 产品经理确认，同步更新排期 |
| **大变更**（新增功能、范围重划）| 重新评审，进 backlog 排队 |

### 变更流程

```
需求变更请求
  → 产品经理评估影响范围
  → 中/大变更 → 需求重新评审
  → 更新 PRD 版本号
  → 通知所有相关人员
```

## 三、落地行动清单

> 按优先级排列的具体执行步骤。
## 第一步：建立统一需求收集入口

```markdown
# 飞书需求收集表

## 必填字段
- 需求标题
- 需求背景
- 提出方（客户/内部/产品/市场）
- 关联模块（AI/Web/API/Infra）
- 紧急程度（紧急/重要/一般）

## 选填字段
- 期望效果
- 竞品参考
- 附件

## 管理规则
- 产品经理每周一整理上周需求
- 紧急需求实时通知产品负责人
- P0 需求 24 小时内启动评估
```

## 第二步：定义 RICE 优先级模型

```markdown
# RICE 评分标准

| 分数 | Reach | Impact | Confidence | Effort |
|------|-------|--------|------------|--------|
| 1 | <10用户 | 微乎其微 | 纯猜想 | >40人天 |
| 2 | 10-50 | 略有提升 | 有参考 | 20-40 |
| 3 | 50-200 | 中等 | 有数据支撑 | 10-20 |
| 4 | 200-1000 | 显著 | 已验证 | 5-10 |
| 5 | 全部客户 | 关键突破 | 数据验证 | <5人天 |

Score = (Reach × Impact × Confidence) / Effort
```

## 第三步：制定 PRD 模板并在团队推行

```markdown
# 验收 Checklist
- [ ] PRD 模板已创建并纳入文档库
- [ ] 所有产品经理已培训 PRD 编写标准
- [ ] PRD 评审会固定为每周/双周例会
- [ ] PRD 必须包含验收标准
- [ ] PRD 必须包含成功标准（怎么衡量做成了）
```

## 第四步：建立需求评审流程

```markdown
# 评审流程

## 评审前（产品准备）
- 提前 2 个工作日发送 PRD 和评审议题
- 确保 PRD 内容完整、无歧义

## 评审中（四方评审）
- 产品：讲解需求背景和业务目标
- 技术：评估可行性和实现方案
- 设计：确认交互和视觉方案
- QA：确认验收标准和测试方案

## 评审后
- 结论：通过 / 有条件通过 / 拒绝
- 修改项限期整改
- 评审结论形成书面记录
```

## 第五步：建立路线图同步机制

```markdown
# 同步节奏
- 季度：全公司路线图同步会
- 月度：研发+产品规划同步
- 迭代：双周迭代会同步具体排期

# 季度路线图模板
- 本季度回顾（交付了什么、未完成什么）
- 下季度重点方向（3-5 个关键目标）
- 核心需求列表（按 RICE 排序 Top 10）
- 资源规划（人力分配、外部依赖）
```

## 第六步：需求变更管控

```markdown
# 变更流程
- 小变更（UI 文案）：开发自行处理，更新 PRD
- 中变更（新增接口）：产品确认，调整排期
- 大变更（范围重划）：重新评审，进 backlog

# 禁止规则
- ❌ 迭代中途直接加需求
- ❌ 变更后不更新 PRD
- ❌ 变更不通知相关干系人
```

## 验收 Checklist

- [ ] 飞书需求收集表已创建并发布
- [ ] RICE 优先级模型已定义，团队已了解
- [ ] PRD 模板已发布，产品经理已培训
- [ ] 需求评审会已固定为周期性活动
- [ ] 季度路线图已输出并同步
- [ ] 需求变更流程已建立
- [ ] 已完成第一期需求整理和优先级排序
