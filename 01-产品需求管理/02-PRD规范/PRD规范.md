# PRD 编写规范

> 产品需求文档(PRD)是产品、技术、设计、QA 之间的"契约"，标准化的 PRD 能大幅减少沟通成本和交付偏差。

---

## 一、PRD 模板结构

```markdown
# [需求标题]

## 1. 背景与目标

### 业务背景
（当前存在什么问题、有什么机会）

### 成功标准
（怎么衡量做成了，量化指标）

### 关联文档
（相关 PRD、竞品分析、用户调研报告链接）

---

## 2. 需求范围

### 核心功能（Must Have）
- （必须交付的功能列表）

### 扩展功能（Nice to Have）
- （有条件才做的功能）

### 非目标（Out of Scope）
- （明确本期不做，防止范围蔓延）

---

## 3. 用户流程

### 角色定义
| 角色 | 描述 | 权限 |
|------|------|------|

### 主流程
```
（核心正常流程）
```

### 异常流程
```
（网络异常、权限不足、数据为空等）
```

### 用户故事
```
As a [角色]
I want [功能]
So that [价值]
```

---

## 4. 功能规格

### 功能列表
| 功能点 | 描述 | 优先级 | 备注 |
|--------|------|--------|------|

### 交互说明
（关键交互逻辑、页面跳转、状态变化）

### 边界条件
（输入验证、极限值、并发情况）

### 错误处理
（各种异常情况的表现和提示文案）

---

## 5. 验收标准（Acceptance Criteria）

### 功能验收
```
Given [前置条件]
When [操作]
Then [期望结果]
```

### 性能要求（如有）
- 接口响应时间 < Xms
- 并发支持 X 用户
- 页面加载时间 < Xs

### 安全要求（如有）
- 权限校验
- 数据脱敏
- 操作审计

---

## 6. 依赖与风险

### 外部依赖
| 依赖项 | 说明 | 预计完成时间 |
|--------|------|-------------|

### 技术风险
| 风险 | 影响 | 应对方案 |
|------|------|----------|
| ... | ... | ... |

### 合规风险
（数据合规、隐私合规等）
```

---

## 二、PRD 编写原则

| 原则 | 说明 |
|------|------|
| **一个 PRD 只解决一个问题** | 不要在一个 PRD 里塞多个不相关的功能 |
| **写清楚"为什么"** | 背景和成功标准比功能列表更重要 |
| **验收标准必须可验证** | 避免"用户体验好"这种主观描述，用"页面加载 < 2s" |
| **明确非目标** | 防止评审时范围蔓延 |
| **区分 Must Have 和 Nice to Have** | 给技术团队交付优先级参考 |
| **保持版本历史** | 每次修改更新版本号 + 变更说明 |

---

## 三、PRD 版本管理

| 版本 | 说明 |
|------|------|
| v0.1 | 初稿，产品经理完成 |
| v0.2 | 技术反馈后修订 |
| v0.3 | 设计反馈后修订 |
| v1.0 | 评审通过，正式定稿 |
| v1.1 | 评审后小调整 |
| v2.0 | 需求变更后大版本更新 |

---

## 四、PRD 编写 Checklist

- [ ] 背景描述是否回答了"为什么要做"？
- [ ] 成功标准是否量化、可验证？
- [ ] 核心功能和扩展功能是否区分清楚？
- [ ] 非目标（Out of Scope）是否明确？
- [ ] 用户角色和权限是否定义清楚？
- [ ] 验收标准是否覆盖正常流程和异常流程？
- [ ] 性能/安全要求是否已考虑？
- [ ] 依赖和风险是否已识别？
- [ ] 是否有竞品分析或用户调研支撑？
