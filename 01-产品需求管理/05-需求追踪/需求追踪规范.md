# 需求追踪规范

> 需求追踪确保每个需求从提出到交付都有状态、有负责人、有记录，避免需求"石沉大海"。

---

## 一、需求状态定义

| 状态 | 含义 | 负责人 |
|------|------|--------|
| **New** | 新建，待产品经理初筛 | 提报人 |
| **Triaging** | 产品经理评估中，补全信息 | 产品经理 |
| **Scored** | RICE 评分完成，等待排期 | 产品经理 |
| **Scheduled** | 已确定迭代版本 | 产品经理 |
| **In Progress** | 开发中 | 开发工程师 |
| **In Review** | 代码审查 / 设计审查中 | 开发工程师 |
| **In Testing** | QA 测试中 | QA |
| **Delivered** | 已上线，等待效果回顾 | 产品经理 |
| **Closed** | 效果回顾完成，正式关闭 | 产品经理 |
| **Rejected** | 不符合方向 / 重复 / 不可行 | 产品经理 |

---

## 二、需求状态流转

`
New → Triaging → Scored → Scheduled → In Progress → In Review
                                                            │
                                                            ▼
Closed ← Delivered ← In Testing ←────────────────────────────┘
`

**强制规则：**
- 状态不能跳跃（如 New → In Progress）
- Delivered 必须有上线时间和版本号
- Closed 必须有效果回顾记录
- Rejected 必须填写拒绝原因

## 三、需求与代码关联

### 分支命名

`
feature/PROJ-123-short-desc      # 新功能
enhance/PROJ-456-short-desc      # 功能增强
fix/PROJ-789-short-desc          # Bug 修复
`

### Commit 规范

`
feat: 添加用户权限管理功能 PROJ-123
enhance: 优化搜索响应速度 PROJ-456
fix: 修复登录超时问题 PROJ-789
`

## 四、需求变更追踪

| 变更类型 | 记录方式 |
|----------|----------|
| 范围增加 | 更新 PRD 版本，需求备注中记录变更 |
| 范围减少 | 标记为"延期"，并备注原因 |
| 优先级调整 | 更新 RICE 分数和优先级字段 |
| 需求取消 | 状态改为 Rejected，填写原因 |

### 变更历史模板

`markdown
## 变更记录

| 版本 | 日期 | 变更内容 | 变更人 | 原因 |
|------|------|----------|--------|------|
| v1.0 | 2024-01-01 | 初版定稿 | ... | - |
| v1.1 | 2024-01-10 | 新增 XX 功能 | ... | 客户反馈 |
| v1.2 | 2024-01-15 | 移除 YY 功能 | ... | 技术不可行 |
`

## 五、定期 Review

| 频率 | 内容 | 参与人 |
|------|------|--------|
| **每日站会** | 需求进度同步，阻塞项识别 | 研发+产品 |
| **双周迭代评审** | 已交付需求演示，下迭代预告 | 产品+研发+QA |
| **月度需求Review** | 需求吞吐率、交付周期、变更率分析 | 产品负责人 |

## 六、工具配置

### Jira / 项目管理工具配置

- Issue Type：Story / Task / Enhancement
- 自定义字段：模块、RICE Score、PRD 链接
- 工作流：按上述状态配置流转规则
- 看板：按状态分类的 Kanban 视图
- 仪表盘：需求吞吐率、交付周期、积压量

### Automations 建议

- New → Triaging 超过 3 天未处理 → 通知产品经理
- Scheduled → In Progress 超过迭代开始时间未移动 → 通知技术负责人
- 需求状态发生变化 → 飞书群通知相关干系人
