# 需求收集与优先级规范

> 定义需求从提出到进入优先级队列的标准流程，确保每个需求都被正确记录和评估。

---

## 一、需求来源分类

| 来源 | 说明 | 收集方式 |
|------|------|----------|
| **客户反馈** | 客户直接提出的功能需求 | 售前/CSM 录入收集表 |
| **内部产品** | 产品经理规划的功能迭代 | 直接录入需求池 |
| **Bug 驱动** | 高频 Bug 反映的系统性缺陷 | 缺陷管理中提取 |
| **市场竞品** | 竞品分析产生的需求 | 产品经理整理 |
| **技术优化** | 研发提出的架构优化、技术债清理 | 技术负责人评估 |
| **数据驱动** | 基于用户行为数据发现的机会 | 数据分析师提出 |

---

## 二、需求记录标准

每个需求进入需求池时必须包含以下字段：

| 字段 | 必填 | 说明 |
|------|------|------|
| 标题 | 是 | 一句话描述：`[模块] 动词 + 对象` |
| 背景 | 是 | 为什么要做，当前有什么问题 |
| 期望效果 | 是 | 达成后的业务/技术变化 |
| 提出方 | 是 | 客户 / 内部 / 产品 / 市场 |
| 提出时间 | 是 | 自动记录 |
| 关联模块 | 是 | AI / Web / API / Infra |
| 紧急度 | 是 | 紧急 / 重要 / 一般 |
| 关联 Bug | 否 | 如有相关的 Bug 单号 |

## 三、优先级评估：RICE 模型

### 评分标准

| 分数 | Reach（覆盖） | Impact（影响） | Confidence（信心） | Effort（投入） |
|------|--------------|---------------|-------------------|---------------|
| 1 | < 10 用户 | 微乎其微 | 纯猜想 | > 40 人天 |
| 2 | 10-50 | 略有提升 | 有竞品参考 | 20-40 人天 |
| 3 | 50-200 | 中等提升 | 有用户调研 | 10-20 人天 |
| 4 | 200-1000 | 显著提升 | 有数据支撑 | 5-10 人天 |
| 5 | 全部客户 | 关键突破 | 已验证有效 | < 5 人天 |

### 计算公式

```
RICE Score = (Reach × Impact × Confidence) / Effort
```

### 优先级队列

| RICE 分数区间 | 优先级 | 建议排期 |
|--------------|--------|----------|
| ≥ 20 | P0 - 战略 | 当前/下个迭代 |
| 10 - 20 | P1 - 重要 | 下 1-2 个迭代 |
| 3 - 10 | P2 - 常规 | backlog，按秩序 |
| < 3 | P3 - 低优 | 积压，定期清理 |

## 四、周期性整理流程

| 频率 | 活动 | 产出 |
|------|------|------|
| **每周** | 产品经理整理新需求，去重、分类、补全信息 | 本周新增需求清单 |
| **双周** | 研发+产品联合评审，RICE 打分 | 迭代待办列表 |
| **月度** | 复盘需求交付情况，调整优先级 | 月度需求报告 |
| **季度** | 路线图对齐，长期需求规划 | 季度路线图 |

---

## 五、需求状态流转

```
新建(New)
  → 待评估(Triaging)    — 产品经理初筛，补全信息
  → 评估完成(Scored)    — RICE 打分完成
  → 排期中(Scheduled)  — 已确定迭代
  → 开发中(In Progress)
  → 已交付(Delivered)
  → 已关闭(Closed)      — 效果回顾后关闭
  └→ 已拒绝(Rejected)   — 不符合方向/重复/不可行
```

---

## 六、优先级调整原则

| 场景 | 处理方式 |
|------|----------|
| 新需求比现有迭代需求优先级更高 | 下个迭代优先安排，不中断当前迭代 |
| P0 紧急需求 | 产品+技术负责人会审，允许中断 |
| 需求提出 2 个月未评估 | 自动降级为 P3 |
| 需求评估后 6 个月未排期 | 自动关闭，需重新提交 |
