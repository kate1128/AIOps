# Sentry — 错误追踪平台（未采用）

## 概述

Sentry 是业界最广泛使用的应用错误追踪平台，支持 50+ 语言/框架的异常捕获、SourceMap 解析、性能追踪。从错误发生到根因定位提供完整工具链。

## 核心能力

- **异常捕获**: 支持所有主流语言和框架的自动异常捕获
- **SourceMap**: 自动解析混淆后的前端 JavaScript 错误堆栈
- **性能追踪**: 集成 tracing，查看事务级别的性能瓶颈
- **版本对比**: 跟踪每个版本的错误率变化趋势

## 为何未采用

| 原因 | 说明 |
|------|------|
| 自托管成本 | 自托管 Sentry 需要 12+ 容器（包括 Kafka、ClickHouse、PostgreSQL、Redis），运维负担极重 |
| SaaS 限制 | SaaS 版数据存储在美国，不符合数据本地化要求 |
| 功能重叠 | PostHog 的错误追踪 + 产品分析在相同复杂度下覆盖更广场景 |
| 资源开销 | 即使最小配置也需要 8 核 32GB 内存，是全项目最重的组件 |

## 轻量级替代

| 工具 | 容器数 | 数据库 |
|------|--------|--------|
| Sentry (自托管) | 12+ | PostgreSQL + ClickHouse + Kafka + Redis |
| GlitchTip | 4 | PostgreSQL + Redis |

## 何时可以考虑

- 错误追踪成为团队最高优先级需求
- 有预算购买 Sentry SaaS 且数据合规问题已解决
- 团队已有 ClickHouse 运维经验，能承担 Sentry 的运维开销
