## 方案一（Grafana Alloy + Prometheus + Loki）覆盖分析

> **方案一核心逻辑**：Grafana Alloy DaemonSet 作为唯一采集端，内置 `prometheus.exporter.unix`（替代独立 node-exporter）、`loki.source.file`（替代 Promtail），通过 `prometheus.scrape` 组件主动抓取各 Exporter 端口，将指标 remote_write 到 Prometheus，日志 push 到 Loki。
>
> **判断维度**：① Alloy 能否直接抓到指标端口 ② 是否需要额外部署 Exporter ③ 是否需要中间件侧配置改动

### 覆盖情况总表

| 中间件 | 可观测结论 | Alloy 角色 | 仍需额外组件 | 中间件侧需改动 | 注意事项 |
|--------|-----------|-----------|------------|--------------|---------|
| **PostgreSQL** | ✅ 完整支持 | 抓取 `:9187/metrics` | 需部署 `postgres-exporter` Sidecar/Pod | 无需改 PG 配置 | exporter 通过数据库连接串读 pg_stat_* 视图 |
| **Redis** | ✅ 完整支持 | 抓取 `:9121/metrics` | 需部署 `redis-exporter` Pod | 无 | 支持单机 / Sentinel / Cluster 三种模式 |
| **Milvus** | ✅ 完整支持 | 抓取 `:9091/metrics` | 无（内置端点） | 无 | Milvus 依赖的 etcd / Kafka 需额外覆盖 |
| **Kafka** | ⚠️ 需多步操作 | 抓取 `:9999`（JMX）和 `:9308`（kafka-exporter） | 需部署 `kafka-exporter` Pod | Kafka 启动参数加 `-javaagent` 挂载 JMX Exporter | JMX Agent 需嵌入 Kafka 进程，Kafka 侧改动不可跳过 |
| **RabbitMQ** | ✅ 完整支持 | 抓取 `:15692/metrics` | 无（内置插件） | 需启用 `rabbitmq_prometheus` 插件（3.8+ 默认已开） | 低版本 RabbitMQ 需升级或手动启用插件 |
| **Nginx** | ✅ 完整支持 | 抓取 `:9113/metrics`（exporter）或 `:10254/metrics`（Ingress） | 需部署 `nginx-prometheus-exporter` Pod（独立 Nginx）| Nginx 配置开启 `stub_status` 端点 | K8s Ingress Controller 自带指标，无需额外 exporter |
| **Docker** | ✅ K8s 场景完整 | 通过抓取 kubelet `:10250/metrics/cadvisor` 覆盖容器指标 | 无（K8s 场景 kubelet 内含 cAdvisor） | Docker `daemon.json` 开启 `metrics-addr`（纯 Docker 场景才需要）| 纯 Docker 非 K8s 场景需单独部署 cAdvisor 容器 |
| **Kubernetes** | ✅ 完整支持 | 抓取 KSM `:8080` + kubelet `:10250` | 仍需部署 `kube-state-metrics`（KSM）；Alloy 内置 unix exporter 只替代 node-exporter | 无 | Alloy 不能替代 KSM，KSM 必须独立部署 |
| **MinIO** | ✅ 完整支持 | 抓取 `:9000/minio/v2/metrics/cluster` | 无（内置端点） | 需在 MinIO 侧生成 Prometheus Bearer Token | Alloy scrape 配置须附带 `Authorization: Bearer <token>` |
| **Harbor** | ✅ 完整支持 | 分别抓取 core/jobservice/registry 各 `/metrics` | 无（内置端点） | Harbor 配置文件开启 `metric.enabled: true` | 多个微服务端口需在 Alloy 中配置多个 scrape job |
| **GitLab** | ⚠️ 需认证配置 | 抓取 `/-/metrics`（Rails/Sidekiq）和 `:9236/metrics`（Gitaly）| 无（内置端点） | GitLab 需关闭 metrics 端点认证或配置 Bearer Token | Omnibus 部署默认开启 IP 白名单，需放开 Alloy Pod CIDR |
| **NFS** | ✅ 原生覆盖 | Alloy 内置 `prometheus.exporter.unix`（含 nfs/nfsd collector）| **无**（Alloy unix exporter 即 node-exporter，nfs collector 默认开启） | 无 | 这是方案一相比方案二的优势：无需独立 node-exporter DaemonSet |
| **HAMI** | ✅ 完整支持 | 抓取 `:8080/metrics` | 无（内置端点） | 无 | 与 DCGM Exporter 配合使用效果最佳（分配指标 vs 硬件指标互补）|
| **GPU/NPU（DCGM）** | ✅ 完整支持 | 抓取 `:9400/metrics` | 仍需部署 `dcgm-exporter` DaemonSet（GPU 节点） | 无（Helm 一键部署） | DCGM DaemonSet 只调度到有 GPU 的节点，Alloy 通过 ServiceMonitor 或 static_config 抓取 |
| **Firecrawl** | ⚠️ 需代码埋点 | 抓取自定义 `/metrics` 端口 | 无外部 Exporter | **必须**在 Firecrawl 源码中集成 `prometheus_client` 并 `start_http_server()` | 无社区 Exporter，指标内容完全由开发团队决定 |
| **Sandbox（gVisor）** | ✅ 双路覆盖 | ① kubelet cAdvisor 覆盖容器级资源；② 抓取 gVisor sentry `:9001/metrics` | 无额外 exporter | gVisor 启动需加 `--metrics-server=:9001` 参数 | RunC（非 gVisor）容器全部由 cAdvisor 覆盖，无需额外操作 |

### 结论分类

**🟢 零额外工作（Alloy 直接覆盖，无需任何新组件）**
- **NFS** — Alloy 内置 unix exporter 含 nfs collector，这是方案一独有优势

**🟡 需部署 Exporter 但 Alloy 完整接管采集（主流情况）**
- PostgreSQL（postgres-exporter）、Redis（redis-exporter）、Nginx（nginx-prometheus-exporter）、Kubernetes（kube-state-metrics）、GPU/NPU（dcgm-exporter）

**🟢 中间件内置端点，Alloy 直接抓取（最理想情况）**
- Milvus、RabbitMQ、MinIO、Harbor、HAMI、Sandbox（gVisor 路径）

**🟠 需要中间件侧配置改动**
- **Kafka** — Kafka 启动参数必须加 JMX JavaAgent，无法绕过
- **MinIO** — 需生成 Bearer Token 并配置到 Alloy
- **GitLab** — 需放开 metrics 端点 IP 白名单或关闭认证
- **Harbor** — 需在 `harbor.yml` 开启 `metric.enabled`
- **RabbitMQ** — 老版本需手动启用 prometheus 插件
- **Nginx** — 独立部署需开启 `stub_status`
- **Sandbox** — gVisor 需加 `--metrics-server` 启动参数

**🔴 需要应用开发团队配合**
- **Firecrawl** — 必须在源码中埋点，无社区 Exporter，无法绕过

### Alloy 配置骨架（方案一完整采集）

```river
// ===== 节点指标（替代 node-exporter，含 NFS collector）=====
prometheus.exporter.unix "node" {
  set_collectors = ["cpu", "meminfo", "diskstats", "filesystem", "nfs", "nfsd", "netdev"]
}

prometheus.scrape "node_metrics" {
  targets    = prometheus.exporter.unix.node.targets
  forward_to = [prometheus.remote_write.central.receiver]
}

// ===== Kubernetes 组件 =====
prometheus.scrape "kube_state_metrics" {
  targets = [{ __address__ = "kube-state-metrics.kube-system.svc:8080" }]
  forward_to = [prometheus.remote_write.central.receiver]
}

prometheus.scrape "kubelet_cadvisor" {
  targets = [{ __address__ = "NODE_IP:10250", __scheme__ = "https" }]
  metrics_path  = "/metrics/cadvisor"
  tls_config { insecure_skip_verify = true }
  forward_to = [prometheus.remote_write.central.receiver]
}

// ===== 中间件 Exporter（统一抓取） =====
prometheus.scrape "middleware_exporters" {
  targets = [
    { __address__ = "postgres-exporter.db.svc:9187",  service = "postgresql" },
    { __address__ = "redis-exporter.cache.svc:9121",  service = "redis" },
    { __address__ = "nginx-exporter.ingress.svc:9113", service = "nginx" },
    { __address__ = "kafka-exporter.kafka.svc:9308",  service = "kafka" },
    { __address__ = "kafka.kafka.svc:9999",           service = "kafka-jmx" },
    { __address__ = "dcgm-exporter.gpu.svc:9400",     service = "gpu-dcgm" },
  ]
  forward_to = [prometheus.remote_write.central.receiver]
}

// ===== 内置 metrics 端点（直接抓取） =====
prometheus.scrape "builtin_metrics" {
  targets = [
    { __address__ = "milvus-proxy.ai.svc:9091",       service = "milvus" },
    { __address__ = "rabbitmq.mq.svc:15692",          service = "rabbitmq" },
    { __address__ = "minio.storage.svc:9000",
      __metrics_path__ = "/minio/v2/metrics/cluster", service = "minio" },
    { __address__ = "harbor-core.harbor.svc:8080",    service = "harbor-core" },
    { __address__ = "harbor-jobservice.harbor.svc:8080", service = "harbor-jobservice" },
    { __address__ = "gitlab-gitaly.gitlab.svc:9236",  service = "gitlab-gitaly" },
    { __address__ = "hami-device-plugin.gpu.svc:8080", service = "hami" },
    { __address__ = "firecrawl.ai.svc:9100",          service = "firecrawl" },
    { __address__ = "sandbox-node:9001",              service = "gvisor-sentry" },
  ]
  forward_to = [prometheus.remote_write.central.receiver]
}

// ===== 需要 Bearer Token 的端点 =====
prometheus.scrape "minio_auth" {
  targets = [{ __address__ = "minio.storage.svc:9000",
               __metrics_path__ = "/minio/v2/metrics/cluster" }]
  authorization {
    type        = "Bearer"
    credentials = env("MINIO_PROMETHEUS_TOKEN")
  }
  forward_to = [prometheus.remote_write.central.receiver]
}

prometheus.scrape "gitlab_rails" {
  targets = [{ __address__ = "gitlab.devops.svc:80",
               __metrics_path__ = "/-/metrics" }]
  authorization {
    type        = "Bearer"
    credentials = env("GITLAB_METRICS_TOKEN")
  }
  forward_to = [prometheus.remote_write.central.receiver]
}

// ===== 日志采集（Loki） =====
local.file_match "k8s_pods" {
  path_targets = [{ __path__ = "/var/log/pods/*/*.log" }]
}

loki.source.file "k8s_logs" {
  targets    = local.file_match.k8s_pods.targets
  forward_to = [loki.write.loki_backend.receiver]
}

// ===== 远端写入 =====
prometheus.remote_write "central" {
  endpoint { url = "http://prometheus.observability.svc:9090/api/v1/write" }
}

loki.write "loki_backend" {
  endpoint { url = "http://loki.observability.svc:3100/loki/api/v1/push" }
}
```
