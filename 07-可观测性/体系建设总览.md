# 可观测性 体系建设总览

> 可观测性（Observability）是系统自证健康的能力：当出现问题时，运维和研发能在最短时间内定位根因、止损恢复。三大支柱——指标（Metrics）、日志（Logs）、链路追踪（Traces）——必须协同工作，而不是三个互相孤立的工具。

---

## 一、现状与痛点

### 1.1 核心子域评估

| 子域 | 当前状态 | 主要痛点 | 影响 |
|------|---------|---------|------|
| **指标监控** | 🟡 Prometheus + Grafana 在用 | 告警规则不完善，误报率高（> 60%）；中间件指标覆盖不足 | OnCall 疲劳，告警被忽视 |
| **日志可观测** | 🔴 日志分散 | 各服务日志未统一收集，排查问题需逐个 Pod 查 | 日志定位耗时 30-120 分钟 |
| **链路追踪** | 🔴 未建设 | 微服务调用链路不透明，跨服务问题无法快速定位 | 复杂故障排查耗时 1-4 小时 |
| **可视化与告警** | 🟡 Grafana 基础 Dashboard | 缺乏业务层面的 SLO Dashboard；告警通知只有邮件 | 业务告警不直观，飞书未接入 |
| **流量可观测** | 🔴 未建设 | 无 API 层面的流量拓扑和异常流量检测 | 流量突增/异常无法快速感知 |

### 1.2 典型问题场景

**场景一：三级排查才定位到是中间件问题**
业务报告某接口超时，运维先看应用日志（各 Pod 分散），再查 Prometheus 指标，最后才发现是 PostgreSQL 连接池耗尽。整个过程走了三个工具，耗时 1.5 小时。如果有统一的链路追踪，数据库调用超时会直接在 Trace 中显示。

**场景二：告警轰炸导致忽视**
某次上线后，同一个问题触发了 12 条独立告警（Pod CPU / 内存 / 请求延迟 / 错误率各多条）。OnCall 工程师需要在 12 条告警中逐一判断哪些是根因、哪些是衍生告警，信噪比极低。

**场景三：Loki 未部署，日志只能 kubectl logs**
定位问题时只能 `kubectl logs <pod> | grep error`，历史日志靠 Pod 内存，重启后消失。某次 Pod OOM Killed 后，问题日志完全丢失，无法复盘。

## 二、各子域建设方案

### 2.1 统一可观测平台（Grafana Stack）

**推荐架构（Grafana Stack）**：

```
应用服务 / 中间件
  ├── Prometheus Exporter → Prometheus → Grafana（指标）
  ├── Promtail / Fluent Bit → Loki → Grafana（日志）
  └── OpenTelemetry SDK → Tempo → Grafana（链路追踪）

告警路由：Alertmanager → 飞书 Bot（OnCall 组）/ PagerDuty
```

三个后端（Prometheus / Loki / Tempo）全部在 Grafana 中统一查询，支持从 Metrics 跳转到 Logs，再跳转到 Traces，实现"三支柱联动"排查。

### 2.2 指标监控（Prometheus）

**告警降噪方案**：

```yaml
# 分层告警规则策略
groups:
  - name: infra.critical
    rules:
      # P0 级：服务完全不可用（5 分钟确认，避免闪断误报）
      - alert: ServiceDown
        expr: up{job="app"} == 0
        for: 5m

  - name: infra.warning
    rules:
      # P1 级：性能劣化（持续告警 + 恢复告警）
      - alert: HighErrorRate
        expr: rate(http_requests_total{status=~"5.."}[5m]) / rate(http_requests_total[5m]) > 0.05
        for: 2m
        annotations:
          summary: "{{ $labels.service }} 错误率 {{ $value | humanizePercentage }}"
```

**告警分组**（减少告警风暴）：
- `group_by: ['alertname', 'cluster']` — 同类告警合并为一条
- `group_wait: 30s` — 等待相关告警汇聚
- `repeat_interval: 4h` — 重复告警间隔，避免轰炸

### 2.3 日志可观测（Loki）

**日志采集接入流程**：

```
K8s Pod（应用日志输出到 stdout/stderr）
  ↓
Promtail DaemonSet（每个节点自动采集所有 Pod 日志）
  ↓
Loki（日志存储 + 索引，按 namespace/service/pod_name 标签）
  ↓
Grafana Explore（LogQL 查询）
  ↓
告警规则（关键词告警：ERROR/FATAL/OOM/panic）

示例 LogQL：
{namespace="prod", service="api"} |= "ERROR" | json | line_format "{{.message}}"
```

**结构化日志规范（应用侧配合）**：

```json
{
  "timestamp": "2025-01-01T12:00:00Z",
  "level": "ERROR",
  "service": "user-api",
  "trace_id": "abc123",
  "message": "DB connection timeout",
  "duration_ms": 5023
}
```

`trace_id` 字段确保日志能与 Tempo 链路追踪关联。

### 2.4 链路追踪（Tempo + OpenTelemetry）

**接入方式**：

```python
# 应用侧接入（Python 示例）
from opentelemetry import trace
from opentelemetry.exporter.otlp.proto.grpc.trace_exporter import OTLPSpanExporter

tracer_provider = TracerProvider()
tracer_provider.add_span_processor(
    BatchSpanProcessor(OTLPSpanExporter(endpoint="http://tempo.monitoring.svc:4317"))
)
trace.set_tracer_provider(tracer_provider)
```

接入后能力：
- 服务依赖拓扑图（自动发现）
- 跨服务请求耗时分解（哪个下游慢）
- 请求失败时自动关联相关日志

### 2.5 告警路由（飞书集成）

```
Alertmanager → Webhook → 飞书告警 Bot

飞书告警卡片模板（含以下字段）：
- 告警级别（P0/P1/P2）
- 告警名称 + 服务名
- 触发时间 + 当前值 vs 阈值
- Grafana Dashboard 直链（一键跳转查看）
- OnCall 认领按钮
```

## 五、建设路径

### Phase 0（第 1-2 周）：指标告警规范化

| 任务 | 具体行动 | 验收标准 | Owner |
|------|---------|---------|-------|
| Alertmanager 飞书路由 | 配置 Alertmanager Webhook → 飞书 Bot，告警卡片模板 | 测试告警可在飞书 OnCall 群收到 | 运维 |
| 告警规则审查 | 审查所有现有告警规则，删除/合并重复，添加 `for` 时间窗口 | 告警规则减少 30%，误报率下降 | 运维 |
| SLO Dashboard | 为核心服务配置 Grafana SLO Panel（可用率/延迟/错误率）| 3 个核心服务有 SLO Dashboard | 运维 + 研发 |

### Phase 1（第 3-6 周）：三支柱联通

| 任务 | 具体行动 | 验收标准 | Owner |
|------|---------|---------|-------|
| Loki + Promtail 部署 | Helm 安装 Loki Stack，配置 Promtail 采集所有 namespace | Grafana 可查全量 K8s Pod 日志 | 运维 |
| Tempo 部署 | Helm 安装 Tempo，应用侧接入 OTLP SDK（从 1 个服务开始）| 至少 1 个服务可在 Grafana 查 Trace | 运维 + 研发 |
| 日志告警规则 | 为关键错误关键词配置 Loki Alert | ERROR/FATAL 类日志告警在 2 分钟内触发 | 运维 |

### Phase 2（第 7-10 周）：AI 辅助智能化

| 任务 | 具体行动 | 验收标准 | Owner |
|------|---------|---------|-------|
| 告警 AI 研判 | 告警 Webhook → 聚合上下文 → Claude API → 飞书富文本卡片 | 告警研判建议准确率 > 70% | 运维 + AI Infra |
| 日志异常分析 | Loki Alert → 聚合日志片段 → Claude 归因 | 日志异常告警附带 AI 分析摘要 | 运维 |
| PromQL 生具 | 在飞书机器人中支持自然语言 → PromQL 查询 | 日常查询效率提升 60% | AI Infra |

## 六、关键指标

| 指标 | 当前值 | Phase 2 目标 | 采集方式 |
|------|--------|-------------|---------|
| 有效告警率 | < 30% | > 70% | 告警总数 / 人工确认有效数 |
| 日志覆盖率 | < 20%（手动 kubectl logs）| > 95% | Loki 采集 Pod 数 / 集群总 Pod 数 |
| 链路追踪覆盖率 | 0% | > 60%（核心服务）| 接入 OTLP 的服务数 / 总服务数 |
| 平均故障定位时间（MTTD）| 30-120 min | < 15 min | Alertmanager 触发 → Jira 创建时间差 |
| PromQL 人均查询时间 | 10-30 min | < 2 min | 问卷调研 |
