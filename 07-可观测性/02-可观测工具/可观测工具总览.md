# 可观测工具总览

> 可观测性（Observability）由三大支柱组成：**指标（Metrics）**、**日志（Logs）**、**链路追踪（Traces）**。本文梳理 SmartVision 项目评估过的全部工具，按采集 → 存储 → 可视化 → 告警 → 分析 的完整链路分类，给出选型结论。

## 六、方案对比：三大主流可观测栈

### 6.1 Grafana 全栈

```
采集：Alloy / OTel SDK
指标：Prometheus → Grafana
日志：Loki → Grafana
链路：Tempo → Grafana
告警：Alertmanager → PrometheusAlert → 飞书
```

| 优势 | 劣势 |
|------|------|
| 三维联动（指标→日志→链路跳转） | Loki 全文检索弱于 ES |
| 统一可视化（Grafana） | Tempo 生产案例少于 Jaeger |
| CNCF 生态原生 | 配置复杂度较高 |
| 存储成本低（对象存储） | — |

### 6.2 ELK 全栈

```
采集：Filebeat / Fluentd
存储：Elasticsearch
可视化：Kibana
告警：ElastAlert / Kibana Alerting
```

| 优势 | 劣势 |
|------|------|
| 全文检索能力极强 | 存储成本高（5-10x Loki） |
| 生产案例丰富 | JVM 调优运维负担重 |
| Kibana 功能成熟 | 无原生链路追踪 |

### 6.3 Netdata 独立方案

```
采集+存储+可视化+告警：Netdata（单 binary）
云端聚合：Netdata Cloud
```

| 优势 | 劣势 |
|------|------|
| 零配置、秒级精度 | 缺乏多节点统一视图 |
| 800+ 采集器内置 | 不支持飞书告警 |
| 内置 ML 异常检测 | 与 Prometheus 集成有限 |

## 九、FAQ

### Netdata 可以和 Prometheus 集成吗？

可以。Netdata 内置 Prometheus 格式的指标导出端点，无需额外安装 exporter。Prometheus 或 Grafana Alloy 都可以抓取 Netdata 的 HTTP metrics 端点，再由 Grafana 统一展示。

```text
Netdata（内置 Prometheus 端点）
  -> /api/v1/allmetrics?format=prometheus
Prometheus / Grafana Alloy
  -> Grafana
```

Prometheus 抓取配置示例：

```yaml
- job_name: netdata
  scrape_interval: 5s
  metrics_path: /api/v1/allmetrics?format=prometheus&variables=yes
  scheme: http
  static_configs:
    - targets:
        - netdata-node1.example.com:19999
        - netdata-node2.example.com:19999
        - netdata-node3.example.com:19999
```

Netdata 常用 Prometheus 端点：

| 格式 | metrics_path | 说明 |
| --- | --- | --- |
| 标准格式 | `/api/v1/allmetrics?format=prometheus&variables=yes` | 适合单节点，支持变量替换 |
| 多主机格式 | `/api/v1/allmetrics?format=prometheus_all_hosts&source=raw` | 适合多主机聚合场景 |

多主机场景建议补齐以下标签，否则部分 Dashboard 可能无法正常工作：

| 标签 | 说明 |
| --- | --- |
| `job` | 任务名称 |
| `env` | 环境，如 `production` |
| `instance` | 实例标识 |
| `group` | 分组名称 |
| `source` | 来源标识 |

多主机抓取配置示例：

```yaml
- job_name: netdata
  metrics_path: /api/v1/allmetrics?format=prometheus_all_hosts&source=raw
  relabel_configs:
    - source_labels: [__address__]
      regex: ^(.+)\.\w+:\d+
      target_label: instance
      action: replace
  static_configs:
    - targets: [netdata.hostname.here:19999]
      labels:
        env: production
        group: applications
        source: newproject
```

集成后可使用 Linux 主机综合监控、CPU Stats、Memory Stats、Disks Stats、Network Stats、Containers Stats、Apps Stats、Redis Overview、MongoDB Overview、Elasticsearch Overview 等社区 Dashboard。

### Netdata 可以替代 Grafana Alloy 吗？

不能完全替代。Netdata 和 Grafana Alloy 的定位不同，更适合作为互补关系：Netdata 偏实时主机监控平台，Alloy 偏通用遥测数据采集与转发管道。

| 维度 | Netdata | Grafana Alloy |
| --- | --- | --- |
| 核心职责 | 实时监控平台，采集/存储/可视化一体 | 遥测数据采集、处理和转发管道 |
| 信号类型 | 主要是 Metrics | Metrics、Logs、Traces、Profiling |
| 数据转发 | 可导出到外部系统 | 专为多后端转发设计 |
| OpenTelemetry | 不原生支持 | OTLP 兼容，基于 OTel Collector 生态 |
| 日志采集 | 有限支持 | 原生支持 Loki、文件、Docker、K8s 等 |
| Trace 采集 | 不支持 | 原生支持 OTLP、Jaeger、Zipkin 等 |
| Profiling | 不支持 | 支持 Pyroscope 相关采集链路 |

Netdata 无法替代 Alloy 的典型场景包括：完整日志管道、分布式 Trace、OpenTelemetry 管道、向 Grafana Cloud / Mimir / Loki / Tempo 统一转发、多信号 Kubernetes 原生采集。

如果已经部署 Netdata，也可以由 Alloy 抓取 Netdata 的 Prometheus 端点：

```alloy
prometheus.scrape "netdata" {
  targets = [{ __address__ = "localhost:19999" }]
  metrics_path = "/api/v1/allmetrics?format=prometheus&variables=yes"
  forward_to = [prometheus.remote_write.default.receiver]
  job_name = "netdata"
}
```

总结：仅做本地实时主机指标监控时，Netdata 可以满足；若需要日志、Trace、Profiling 或统一接入 LGTM / Grafana Cloud，Alloy 仍然不可替代。

### Netdata 可以替代 Node Exporter 吗？

在一定程度上可以，但不建议在已有 Prometheus / Grafana 标准体系中直接替换。Netdata 能导出主机指标，覆盖 CPU、内存、磁盘、网络等 Node Exporter 常见能力；但指标命名、标签模型、Dashboard 和告警规则并不完全兼容。

| 维度 | Netdata | Node Exporter |
| --- | --- | --- |
| 核心职责 | 实时监控平台，采集/存储/可视化一体 | Prometheus 主机指标 exporter |
| CPU / 内存 / 磁盘 / 网络 | 支持 | 支持 |
| Prometheus 格式输出 | 内置支持，需配置 metrics path | 原生支持 |
| 部署复杂度 | 较重，完整监控平台 | 极轻，单一二进制 |
| Dashboard 兼容 | 需使用 Netdata 专用 Dashboard | 兼容 Node Exporter 生态 Dashboard |

Prometheus 抓取 Netdata 的示例：

```yaml
- job_name: netdata
  scrape_interval: 5s
  metrics_path: /api/v1/allmetrics?format=prometheus&variables=yes
  scheme: http
  static_configs:
    - targets:
        - netdata-node1.example.com:19999
```

限制：Netdata 导出的指标名称与 Node Exporter 不同，现有基于 `node_cpu_seconds_total` 等指标的 Dashboard 和告警规则无法直接复用；多主机 Prometheus 格式还需要补齐 `job`、`env`、`instance`、`group`、`source` 等标签。

结论：从零搭建且更重视实时主机可视化时，Netdata 可以作为主机指标方案；已有 Prometheus 标准监控体系时，Node Exporter 或 Alloy 内置 `prometheus.exporter.unix` 更适合作为 Node Exporter 的无缝替代。

### Netdata 相当于集成了哪些 Exporter？

可以近似理解为 Netdata 内置了大量采集插件，覆盖部分常见 Prometheus Exporter 的能力，并通过统一 Prometheus 端点导出。但它不是这些 exporter 的完全等价替代。

| 服务 | 对应 Exporter | Netdata 采集方式 |
| --- | --- | --- |
| Linux 主机（CPU/内存/磁盘/网络） | `node_exporter` | 内置 cgroups / 系统采集 |
| Redis | `redis_exporter` | `python.d.plugin` |
| MySQL | `mysqld_exporter` | `python.d.plugin` |
| Apache | `apache_exporter` | `python.d.plugin` |
| Elasticsearch | `elasticsearch_exporter` | `python.d.plugin` |
| PHP-FPM | php-fpm exporter | `python.d.plugin` |
| Docker / LXC 容器 | cAdvisor | `cgroups.plugin` |
| Systemd 服务 | process exporter | `cgroups.plugin` |

需要注意三点：

1. Netdata 指标命名与独立 exporter 不兼容，现有标准 exporter Dashboard 和告警规则通常不能直接复用。
2. Netdata 多主机 Prometheus 格式需要补齐 `job`、`env`、`group`、`source` 等标签。
3. Netdata 覆盖范围不等于 Alloy 内置 `prometheus.exporter.*` 生态，无法证明能覆盖 Kafka、PostgreSQL、MongoDB、SNMP、CloudWatch、Azure 等全部场景。

结论：Netdata 可以替代部分常见 exporter 的采集能力，但在 Grafana / Prometheus 生态深度集成场景下，Alloy 内置 exporter 或标准 Prometheus exporter 仍是更稳妥的选择。
