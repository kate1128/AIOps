# 私有化交付 体系建设总览

> 私有化交付是把 SaaS 形态的产品搬到客户的内网环境运行——这意味着我们失去了对基础设施的控制权，面对各种各样的客户环境差异，交付质量完全依赖我们的工程化程度。交付效率和交付质量直接影响客户续费和口碑。

---

## 一、现状与痛点

### 1.1 核心子域评估

| 子域 | 当前状态 | 主要痛点 | 影响 |
|------|---------|---------|------|
| **实施规范** | 🟡 有流程但不完整 | 依赖个人经验，新人独立完成交付困难 | 实施质量不稳定，踩坑时间长 |
| **配置管理** | 🔴 手工适配 | 每个客户环境手工调整配置，易出错且无复用 | 配置错误导致功能异常 |
| **Preflight 检查** | 🟡 有脚本，不完整 | 检查结果靠人工判断，客户环境不满足条件才发现 | 交付中途中断，重新来过 |
| **故障诊断** | 🔴 靠个人经验 | 客户环境没有内部工具，排查时间 1-4 小时 | 客户等待时间长，体验差 |
| **交付文档** | 🟡 模板存在但不完整 | 文档质量参差，部分关键步骤缺失 | 客户自助维护困难 |

### 1.2 典型问题场景

**场景一：Preflight 通过但交付失败**
客户环境通过了基础检查（CPU/内存/磁盘），但没有检查 K8s 版本（客户是 1.20，最低需要 1.24），导致 CRD 安装失败。实施人员在客户现场排查 2 小时才定位原因。

**场景二：GPU 驱动版本不兼容**
客户采购的是 A100，但驱动版本是旧版，与 vLLM 需要的 CUDA 版本不匹配。Preflight 检查未覆盖 CUDA 版本，只有模型加载时才失败。来回沟通 + 客户更新驱动花了 2 天。

**场景三：每个客户都重新写配置**
每次私有化部署都需要重新整理配置文件（数据库地址/Harbor 地址/域名/TLS 证书），没有统一的 values 模板，容易遗漏。某次遗漏了 Redis 密码配置，导致登录功能异常，客户现场才发现。

## 二、各子域建设方案

### 2.1 Preflight 自动化检查

**完整 Preflight 检查项**：

```bash
#!/bin/bash
# preflight-check.sh - 私有化部署前置检查

check_kubernetes() {
  echo "=== K8s 版本检查 ==="
  K8S_VERSION=$(kubectl version --short 2>/dev/null | grep "Server Version" | awk '{print $3}')
  # 要求 >= 1.24
  check_version "$K8S_VERSION" "1.24" "K8s 版本"
}

check_resources() {
  echo "=== 资源检查 ==="
  TOTAL_CPU=$(kubectl get nodes -o json | jq '[.items[].status.capacity.cpu | tonumber] | add')
  TOTAL_MEM=$(kubectl get nodes -o json | jq '[.items[].status.capacity.memory | gsub("Ki"; "") | tonumber] | add')
  # 要求 >= 32 核 / 64GB
  echo "总 CPU: $TOTAL_CPU 核（要求 ≥ 32）"
  echo "总内存: $((TOTAL_MEM/1024/1024)) GB（要求 ≥ 64）"
}

check_gpu() {
  echo "=== GPU/CUDA 检查 ==="
  nvidia-smi --query-gpu=name,driver_version,memory.total --format=csv
  # 检查 CUDA 版本兼容性
  CUDA_VERSION=$(nvidia-smi | grep "CUDA Version" | awk '{print $9}')
  echo "CUDA 版本: $CUDA_VERSION（要求 ≥ 11.8）"
}

check_storage() {
  echo "=== 存储检查 ==="
  kubectl get storageclass  # 是否有默认 StorageClass
  # 检查 MinIO/NFS 可用性
}

check_network() {
  echo "=== 网络检查 ==="
  # 检查所需端口是否开放
  # 检查内部 Harbor 是否可达
  # 检查 DNS 解析
}
```

**Preflight AI 报告生成**：

```
运行 preflight-check.sh 生成 JSON 结果
  ↓ Claude API 分析：
    - 逐项评估通过/警告/失败
    - 对失败项给出修复方法
    - 整体交付可行性评估（可以继续/需要先解决 X 问题）
  ↓ 生成 HTML 报告（1.5-2 小时手工 → 15 分钟自动）
  ↓ 发送给客户 + 内部存档
```

### 2.2 配置适配模板（Helm Values）

**标准 values 模板结构**：

```yaml
# values-customer-template.yaml
# 由 AI 工具根据客户信息自动生成初稿

global:
  # 客户基本信息（AI 根据交付问卷填写）
  customerName: "{{ customer_name }}"
  environment: "prod"

  # 镜像仓库（客户提供或内网搭建）
  imageRegistry: "{{ harbor_endpoint }}"

database:
  host: "{{ db_host }}"
  port: 5432
  name: "smartvision"
  # 密码通过 K8s Secret 注入，此处留空
  passwordSecretRef: "db-credentials"

redis:
  host: "{{ redis_host }}"
  port: 6379

ai:
  modelPath: "{{ model_path }}"
  gpuMemoryFraction: 0.9
  maxConcurrentRequests: 10

ingress:
  hosts:
    - "{{ domain }}"
  tls:
    secretName: "tls-certificate"
```

**配置适配 AI 辅助流程**：

```
实施工程师填写交付问卷（客户环境信息）
  ↓ Claude API 处理问卷 → 生成 values.yaml 初稿
  ↓ 工程师 review + 补充敏感信息（密码等）
  ↓ 30 分钟 review vs 原先 1-2 小时手工填写
```

### 2.3 部署故障诊断 Playbook

**标准故障排查 Playbook 目录**：

```
13-私有化交付/playbooks/
├── pod-crashloop.md        # Pod 崩溃循环排查
├── image-pull-failed.md    # 镜像拉取失败
├── gpu-not-detected.md     # GPU 未识别
├── db-connection-failed.md # 数据库连接失败
├── vllm-load-failed.md     # 模型加载失败
└── ingress-404.md          # Ingress 访问 404
```

**每个 Playbook 格式**：
```markdown
# 故障：[名称]

## 症状
kubectl 看到什么？日志显示什么？

## 快速诊断命令
```bash
kubectl get pods -n smartvision
kubectl logs <pod> --tail=50
```

## 常见原因 + 解决方法
1. 原因A → 解决方法A
2. 原因B → 解决方法B

## 如需升级支持
提供以下信息给内部 SRE：...
```

## 五、建设路径

### Phase 0（第 1-2 周）：规范化基础

| 任务 | 具体行动 | 验收标准 | Owner |
|------|---------|---------|-------|
| Preflight 脚本完善 | 补充 K8s 版本/CUDA 版本/StorageClass/网络连通性检查 | 检查项覆盖 100% 已知踩坑问题 | 实施工程师 |
| Helm values 模板 | 整理标准 values 模板，所有必填项有注释说明 | 新项目交付可直接基于模板 | 实施工程师 |
| Playbook 补全 | 整理 Top 5 高频故障的排查 Playbook | 5 个 Playbook 文档完整可用 | 实施工程师 |

### Phase 1（第 3-4 周）：AI 工具接入

| 任务 | 具体行动 | 验收标准 | Owner |
|------|---------|---------|-------|
| Preflight AI 报告 | 脚本输出 → Claude API → HTML 报告 | 试用一个客户项目，报告质量验收 | 实施 + AI Infra |
| 配置生成工具 | 开发 Dify 工作流（交付问卷 → values 初稿）| 配置生成时间 < 15 分钟 | AI Infra |
| Dify 知识库上线 | 将 Playbook + 产品文档接入 Dify RAG | 实施工程师可通过飞书 Bot 查询故障处置 | AI Infra |

### Phase 2（第 5-8 周）：标准化交付包

| 任务 | 具体行动 | 验收标准 | Owner |
|------|---------|---------|-------|
| 交付文档自动化 | Dify 工作流生成部署说明书 + 运维手册 | 文档生成时间 < 1h，质量通过产品审核 | 实施 Lead |
| 交付复盘机制 | 每次交付完成后 30 分钟复盘，问题归档到知识库 | 知识库月度增量 ≥ 3 条有效经验 | 实施 Lead |
