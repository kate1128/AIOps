# 版本发布管理 体系建设总览

> 发布管理是连接"研发完成"和"用户使用"之间的最后一公里。好的发布体系让团队自信地频繁发布，坏的发布体系让团队每次发布都提心吊胆。目标是把发布变成一件无聊的、可预期的日常操作。

---

## 一、现状与痛点

### 1.1 核心子域评估

| 子域 | 当前状态 | 主要痛点 | 影响 |
|------|---------|---------|------|
| **版本号规范** | 🟡 基本遵循 SemVer | 部分团队用日期作版本号，规范不统一 | 版本含义不清晰，依赖方难以判断升级影响 |
| **发布流程** | 🟡 有流程但不完整 | 发布前 Checklist 靠人工记忆，漏检概率高 | 发布后频繁发现遗漏 |
| **变更日志** | 🔴 无自动化 | Release Notes 靠人工翻 commit，30-60 分钟；质量参差不齐 | 发布文档质量差，客户不清楚更新内容 |
| **灰度与回滚** | 🟡 有 helm rollback | 无灰度发布（流量切分），发布即全量 | 问题在全量发布后才发现，影响范围大 |
| **环境同步** | 🔴 手动管理 | dev/pre/prod 配置差异无追踪，环境漂移严重 | pre 测试通过但 prod 失败 |

### 1.2 典型问题场景

**场景一：发布 Checklist 遗漏数据库迁移**
发布时遗漏了 DB Schema 变更脚本，应用启动失败。回滚应用后还需要确认 DB 状态，整个过程耗时 1 小时，影响用户 20 分钟。

**场景二：Release Notes 内容不完整**
私有化客户询问"这次更新改了什么"，技术对接人需要花 30-60 分钟手动翻阅 commit log，整理后发给客户，格式不规范，常漏掉重要变更。

**场景三：pre 通过 prod 挂**
pre 环境测试通过后，发布到 prod，发现某个环境变量配置不同导致功能异常。因为没有自动化的环境 diff，这个差异在发布前没有被检测到。

## 二、各子域建设方案

### 2.1 版本号规范（SemVer）

**版本号格式**：

```
MAJOR.MINOR.PATCH[-prerelease]

示例：
  1.2.3         - 正式版本
  1.2.3-rc.1    - 发布候选（Release Candidate）
  1.3.0-beta.2  - Beta 版本

发布规则：
  PATCH 升级：Bug 修复，不含破坏性变更
  MINOR 升级：新功能，向后兼容
  MAJOR 升级：破坏性变更（API 不兼容）
```

**Git 标签对应**（与 03-CICD 联动）：
- `v1.2.3` → 触发 CI 生产发布流水线
- 镜像 tag：`release_v1.2.3`（由 CI 自动设置）

### 2.2 发布流程与 Checklist 自动化

**发布准备 Checklist（GitLab Issue 模板）**：

```markdown
## 发布 Checklist - v{版本号}

### 研发完成
- [ ] 所有功能 Issue 已关闭
- [ ] MR 已合并到 main 分支
- [ ] CI 全绿（测试/扫描/构建）

### 测试验证
- [ ] pre 环境全量回归通过
- [ ] 性能基准对比无回归
- [ ] 新功能验收测试通过

### 发布准备
- [ ] DB 迁移脚本已测试（如有）
- [ ] 配置变更已同步到 prod（环境 diff 检查）
- [ ] Helm values.yaml 已更新
- [ ] Runbook/操作手册已更新（如有）
- [ ] Release Notes 已生成

### 发布操作
- [ ] 生产发布（gitlab tag v{版本号} 触发 CI）
- [ ] 发布后观测 15 分钟（Grafana SLO + 错误率）
- [ ] Checklist 完成确认
```

**AI 辅助 Checklist 验证**：

```
CI 发布阶段触发 → 自动验证：
  - GitLab: 所有 Issue 已 Closed？
  - Prometheus: pre 环境 SLO > 99.5%？
  - Trivy: 无 Critical CVE？
  - 环境 diff: pre vs prod 配置无意外差异？
→ 生成验证报告（通过/不通过 + 原因）
→ 推送飞书，需发布审批人确认后继续
```

### 2.3 Release Notes 自动化（AI 生成）

**生成流程**：

```
触发：git tag v1.2.3 推送
  ↓
GitLab API：获取上一个 Tag 到当前 Tag 的所有 commit log
  ↓
Claude API 处理：
  - 解析 Conventional Commits 格式（feat/fix/perf/breaking）
  - 将 commit 归类为 "新功能 / Bug修复 / 性能优化 / 破坏性变更"
  - 过滤无意义 commit（"fix typo" / "merge branch" 等）
  - 生成面向用户的自然语言描述
  ↓
生成结构化 Release Notes（中英文双版本）
  ↓
自动创建 GitLab Release（附带 Release Notes + 镜像 tag）
  ↓
推送飞书（含发布摘要 + GitLab Release 链接）
```

**Conventional Commits 规范**（研发侧配合）：

```
feat(inference): 支持 Qwen2.5 模型批量推理
fix(api): 修复用户权限判断逻辑错误
perf(db): 为 user_id 字段添加索引，接口延迟从 500ms 降至 50ms
BREAKING CHANGE: API /v1/chat 返回格式变更
```

### 2.4 灰度发布（Argo Rollouts）

**金丝雀发布流程**：

```
发布 v1.3.0：
Phase 1: 10% 流量 → 新版本，观测 5 分钟
  → 自动检查：错误率 < 1%，P99 < SLO 阈值
Phase 2: 50% 流量 → 新版本，观测 10 分钟
  → 自动检查：同上
Phase 3: 100% 流量 → 新版本

任一阶段指标异常 → 自动回滚（无需人工操作）
```

### 2.5 环境配置同步

**自动 diff 检查（发布前）**：

```bash
# 比对 pre 和 prod Helm values 差异
diff <(helm get values app -n pre) <(helm get values app -n prod) \
  | grep -v "^---" > env-diff.txt

# Claude API 分析差异：
# - 标记风险差异（可能导致功能不一致）
# - 区分预期差异（如 replica 数量）
```

## 五、建设路径

### Phase 0（第 1-2 周）：发布规范化

| 任务 | 具体行动 | 验收标准 | Owner |
|------|---------|---------|-------|
| 发布 Checklist 模板 | 在 GitLab 创建 Issue 模板，每次发布前必须创建并完成 | 最近 5 次发布均有 Checklist Issue | 发布 Lead |
| 版本号规范统一 | 制定 SemVer 规范文档，研发 + 测试达成共识 | 所有新版本使用 vX.Y.Z 格式 | 研发 Lead |
| Conventional Commits | 在团队推广 Conventional Commits，更新 README | MR 提交说明中 feat/fix 格式使用率 > 80% | 研发 |

### Phase 1（第 3-4 周）：AI 发布文档

| 任务 | 具体行动 | 验收标准 | Owner |
|------|---------|---------|-------|
| Release Notes AI 生成 | 开发 Tag → Claude API → GitLab Release 自动化脚本 | 下一次发布自动生成 Release Notes | DevOps |
| 环境 diff 自动检查 | 发布前自动比对 pre vs prod Helm values | 差异报告附在 Checklist Issue 中 | DevOps |
| Checklist 自动验证 | CI Tag 阶段自动检查 SLO/CVE/配置 | 验证报告推送飞书，审批后发布 | DevOps |

### Phase 2（第 5-8 周）：灰度化发布

| 任务 | 具体行动 | 验收标准 | Owner |
|------|---------|---------|-------|
| Argo Rollouts 试点 | 在 1 个核心服务上配置金丝雀发布 | 发布 10% → 100% 可分阶段控制 | DevOps |
| 自动回滚 | 配置发布后 5 分钟观测窗口，错误率超阈值自动回滚 | 模拟失败场景可在 3 分钟内自动回滚 | DevOps |

## 六、关键指标

| 指标 | 当前值 | Phase 2 目标 | 采集方式 |
|------|--------|-------------|---------|
| Release Notes 生成时间 | 30-60 min | < 5 min（AI 自动）| 手工计时对比 |
| 发布 Checklist 完成率 | 无追踪 | 100% | GitLab Issue Checklist 统计 |
| 发布后回滚率 | 未统计 | < 5% | GitLab 发布数 / 回滚操作数 |
| 发布到生产耗时（从 Tag 到服务就绪）| 20-40 min | < 15 min（流水线优化）| CI 执行时间 |
| 环境配置差异发现时间 | 发布后（可能更晚）| 发布前自动检查 | 自动 diff 执行时间 |
