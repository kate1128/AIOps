# 版本发布管理 FAQ

---

**Q1：什么时候需要升 MAJOR 版本？**

API 不兼容变更时升 MAJOR。包括：
- 删除或重命名 API 接口
- 修改请求/响应字段
- 修改数据库 Schema 且不向后兼容

MAJOR 为 0 的阶段（0.x.x）表示开发阶段，任何变更都可以不升 MAJOR。

---

**Q2：紧急热修复的版本号怎么处理？**

从当前生产版本 Tag 创建 hotfix 分支，修复后 PATCH +1：
```
v1.0.0 → (hotfix) → v1.0.1
```
hotfix 版本可以和正在开发的版本并行，互不影响。

---

**Q3：灰度发布发现问题的处理流程？**

1. 暂停灰度，停止继续放量
2. 已灰度用户保持当前版本，不回滚（减少影响面）
3. 评估问题严重程度：
   - 严重 → 回滚所有灰度节点
   - 轻微 → 修复后继续灰度
4. 修复后走 hotfix 流程

---

**Q4：pre 环境和 prod 环境需要完全一致吗？**

目标是尽可能一致，但可以接受差异：
- 硬件配置可以不同（但建议接近）
- 数据使用脱敏生产数据
- 不能访问生产数据库
- 密钥和证书独立

---

**Q5：版本发布和私有化交付的版本关系？**

版本发布（本目录）是标准版本的发布流程。私有化交付（13-私有化交付）是基于标准版本 +
客户定制配置的输出。标准版本的 Tag 是私有化交付的基础。

---

**Q6：发布频率的建议？**

| 阶段 | 频率 | 说明 |
|------|------|------|
| 初创期 | 每周 1-2 次 | 快速迭代验证 |
| 成长期 | 每双周 1 次 | 固定迭代节奏 |
| 成熟期 | 每月 1 次 | 稳定为主 |

建议保持稳定的发布节奏，不要为了发版而发版。
