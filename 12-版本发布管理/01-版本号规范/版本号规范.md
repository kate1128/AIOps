# 版本号规范

> 遵循 SemVer 2.0 语义化版本规范，版本号反映兼容性和变更范围，是团队和客户的"契约"。

---

## 一、版本号格式

```
MAJOR.MINOR.PATCH-PRERELEASE+BUILD
   │      │      │       │        │
   │      │      │       │        └── 构建元数据（CI 自动生成，如 CI 流水线编号）
   │      │      │       └────────── 预发布标识（alpha / beta / rc）
   │      │      └────────────────── 补丁版本（向下兼容的 Bug 修复）
   │      └───────────────────────── 次版本（向下兼容的新功能）
   └──────────────────────────────── 主版本（不兼容的 API 变更）
```

## 二、版本号变更规则

| 变更类型 | 版本变更 | 示例 |
|----------|----------|------|
| 不兼容的 API / 协议变更 | MAJOR +1, MINOR/PATCH 归零 | 1.0.0 → 2.0.0 |
| 向下兼容的新功能 | MINOR +1, PATCH 归零 | 1.0.0 → 1.1.0 |
| 向下兼容的 Bug 修复 | PATCH +1 | 1.0.0 → 1.0.1 |
| 预发布版本 | 追加 -alpha.N / -beta.N / -rc.N | 1.0.0-alpha.1 |
| 热修复(hotfix) | PATCH +1 | 1.0.0 → 1.0.1 |

## 三、预发布标识

| 标识 | 含义 | 示例 | 说明 |
|------|------|------|------|
| alpha | 内测版 | 1.0.0-alpha.1 | 功能未完成，仅内部开发测试 |
| beta | 公测版 | 1.0.0-beta.1 | 功能完成，邀请体验用户测试 |
| rc | 发布候选 | 1.0.0-rc.1 | 冻结功能，只修 Bug，准备正式发布 |

## 四、版本号在 Git 中的体现

```
# Tag 命名规范
v{MAJOR}.{MINOR}.{PATCH}[-{PRERELEASE}]

# 示例
v1.0.0           # 正式版
v1.0.0-alpha.1   # Alpha 内测
v1.0.0-beta.1    # Beta 公测
v1.0.0-rc.1      # 发布候选
v1.0.1           # 热修复
```

## 五、特殊情况

| 场景 | 处理方式 |
|------|----------|
| 开发阶段（0.x）| MAJOR = 0，MINOR 递增表示新功能，不保证兼容性 |
| 首次发布 | 从 0.1.0 开始 |
| 依赖不兼容升级 | 必须升 MAJOR |
| 废弃 API | 先标记 deprecated，下一个 MAJOR 版本再移除 |
