# 发布流程规范

> 标准化发布流程，确保每次上线都有章可循、可追溯、可回滚。

---

## 一、发布阶段

### 阶段 1：发布准备

```
触发条件：迭代结束 / 需求达成 / 热修复完成
```

| 操作 | 负责人 | 门禁条件 |
|------|--------|----------|
| 确认需求完成度 | 产品经理 | 需求状态 = Delivered |
| PR 合入主干分支 | 开发工程师 | 代码 Review 通过 |
| CI 构建 + 安全扫描 | CI 自动 | 构建成功，扫描通过 |
| 创建 Release 分支 | CI 自动 | 从主干创建 |
| 生成变更日志 | CI 自动 | Conventional Commits 自动生成 |
| 更新版本号 | CI 自动 | 基于 Tag 自动递增 |

### 阶段 2：Dev 环境部署

```
触发条件：Release 分支创建完成
```

| 操作 | 负责人 |
|------|--------|
| 自动部署到 dev 环境 | CI 自动 |
| 冒烟测试验证 | QA |
| 确认功能正常 | 产品经理（可选） |

### 阶段 3：Pre 环境部署

```
触发条件：dev 环境验证通过
```

| 操作 | 负责人 | 门禁条件 |
|------|--------|----------|
| 部署到 pre 环境 | 技术负责人 | dev 验证通过 |
| 预发环境功能测试 | QA | 测试用例全部通过 |
| 预发环境性能测试 | 技术负责人（可选）| 性能指标达标 |
| 灰度检查 | 技术负责人 | 配置正确、依赖版本兼容 |

### 阶段 4：生产发布

```
触发条件：pre 环境验证通过，审批完成
```

| 操作 | 负责人 |
|------|--------|
| 提交发布审批 | 技术负责人 |
| 审批通过 | 审批人 |
| 执行灰度发布 | 运维 / 值班人员 |
| 监控观察 | 值班人员 |
| 全量发布 | 运维 |
| 发布后监控 | 值班人员（15 分钟）|

### 阶段 5：发布确认

| 操作 | 负责人 |
|------|--------|
| 标记 Git Tag | CI 自动 |
| 创建 GitLab Release | CI 自动 |
| 发送发布通知 | CI 自动 |
| 更新发布记录 | CI 自动 |

---

## 二、环境部署策略

| 环境 | 部署方式 | 触发方式 | 数据隔离 |
|------|----------|----------|----------|
| dev | 自动 | CI 触发 | 测试数据 |
| pre | 自动或手动 | 开发/QA 触发 | 脱敏数据 |
| prod | 审批后自动 | 审批通过触发 | 生产数据 |

---

## 三、禁止规则

| ❌ 禁止 | 原因 |
|---------|------|
| 跳过 pre 环境直接上线 | 未验证的功能风险不可控 |
| 发布当天合入新功能代码 | 发布内容应在发布前确认冻结 |
| 手动修改生产环境配置 | 配置应通过 GitOps / 配置中心管理 |
| 跳过安全扫描 | 带漏洞上线可能是安全事故 |
| 发布后不观察监控 | 发布引起的异常可能未被发现 |
