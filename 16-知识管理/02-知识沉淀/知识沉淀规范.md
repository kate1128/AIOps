# 知识沉淀规范

> 知识的价值在于流动——把隐形知识变为显性文档，让每个人都能访问和复用。

---

## 一、知识来源

| 来源 | 触发时机 | 负责人 |
|------|----------|--------|
| **故障复盘** | P0/P1 故障发生后 72h 内 | OnCall 负责人 |
| **方案设计** | 技术方案评审后 | 方案作者 |
| **技术分享** | 内部分享后整理 | 分享人 |
| **新人反馈** | 新成员入职 2 周后 | 导师 |
| **客户问题** | 高频客户问题归类后 | CSM / 技术支持 |

---

## 二、沉淀流程

```
源头事件
  → 快速记录（飞书文档，允许不完美）
  → 归类入库（按模块/领域分类）
  → 审核完善（技术负责人 Review）
  → 索引打标（添加标签、关联文档）
  → 定期推广（周会、新人培训、知识库首页）
```

---

## 三、知识库结构

```
📁 知识库根目录
├── 📁 架构设计/        ← ADR、架构图、技术方案
├── 📁 服务文档/        ← 各服务 README、API 文档
├── 📁 运维手册/        ← Runbook、部署手册、环境信息
├── 📁 故障复盘/        ← 所有 P0/P1 复盘记录
├── 📁 最佳实践/        ← 编码规范、流程指南、经验总结
└── 📁 新人指南/        ← 上手手册、工具配置、FAQ
```

---

## 四、知识更新机制

| 检查频率 | 内容 | 动作 |
|----------|------|------|
| 月度 | 新知识入库统计 | 更新知识库目录和索引 |
| 季度 | 文档健康度检查 | 标记过时、补充缺失、清理重复 |
| 事件驱动 | 服务/架构变更 | 更新相关文档 |
