# 知识管理 体系建设总览

> 知识管理是把散落在个人头脑、飞书文档、Wiki、代码注释里的知识，沉淀成可以被团队检索、复用、传承的组织资产。当团队规模增长、人员流动加快时，知识管理的价值会指数级放大——新人上手速度、经验复用效率、组织记忆连续性都依赖于此。

---

## 一、现状与痛点

### 1.1 核心子域评估

| 子域 | 当前状态 | 主要痛点 | 影响 |
|------|---------|---------|------|
| **文档标准** | 🟡 有规范但执行不一致 | 文档结构混乱，没有模板约束，质量参差不齐 | 文档可读性差，无法有效复用 |
| **知识沉淀** | 🔴 经验散落在人 | 故障处置经验、踩坑记录靠个人记忆，人员离职即消失 | 相同问题反复出现，新人上手慢 |
| **知识检索** | 🔴 无 RAG 系统 | 找文档靠手动搜索，耗时 5-15 分钟；跨文件夹检索困难 | 知识利用率低 |
| **文档质量** | 🔴 无 CI 检查 | 文档更新不及时，链接失效，内容过时 | 错误文档比没有文档更危险 |
| **Onboarding** | 🟡 有入职文档 | 新人入职需要 1-2 周熟悉环境，全靠老员工带 | 关键人依赖，效率低 |

### 1.2 典型问题场景

**场景一：找文档找了 30 分钟**
运维需要查 Kafka 消息堆积的处置步骤，在飞书搜索"Kafka 堆积"，找到 3 个不同版本的文档，不确定哪个是最新的。最终找到对应 Runbook 花了 30 分钟。如果有 RAG 问答，10 秒内就能得到答案。

**场景二：老员工离职带走核心经验**
某资深 SRE 离职，他熟知的"私有化环境 GPU 驱动兼容性问题排查"流程从未被文档化。后来实施团队遇到相同问题，重新摸索花了 3 天。

**场景三：文档内容过时导致事故**
某个配置变更文档更新了配置项名称，但忘记更新文档中的操作步骤。新来的工程师按文档操作，使用了旧的配置项名称，导致服务配置错误重启失败。

## 二、各子域建设方案

### 2.1 文档标准化（Markdown + Git）

**文档规范要点**：

```
文档目录结构（统一约定）：
  每个领域目录下必须包含：
  - 体系建设总览.md（领域全局视图）
  - AI提效计划.md（AI 赋能规划）
  - FAQ.md（常见问题）
  - 工具分析/（工具对比与选型）

文档格式规范：
  - 必须有一级标题（# 标题）
  - 所有外部链接 + 内部跨领域链接用相对路径
  - 代码块指定语言（```bash / ```yaml / ```python）
  - 表格中的状态统一用 emoji（✅🟡🔴）
```

**GitLab CI 文档质量检查**：

```yaml
# .gitlab-ci.yml 新增 docs-lint stage
docs-lint:
  stage: lint
  image: harbor.internal/tools/markdownlint:latest
  script:
    # 检查 Markdown 格式
    - markdownlint '**/*.md' --ignore node_modules
    # 检查内部链接是否有效
    - markdown-link-check '**/*.md' --config .mlc-config.json
    # AI 检查文档质量（可选，调用 Claude API）
    - python3 scripts/doc-quality-check.py
  rules:
    - changes:
        - "**/*.md"
```

### 2.2 RAG 知识库（Dify + 本工程文档）

**Dify 知识库架构**：

```
数据源（自动同步）：
  - 本工程所有 .md 文件（GitLab Webhook 触发更新）
  - 飞书知识库（通过飞书 API 接入）
  - Confluence（如有）

Dify 处理：
  - 文档分块（按章节切分，保留上下文）
  - 向量化（BGE-M3 / text-embedding-ada-002）
  - 存入 Milvus 向量数据库

查询入口：
  - 飞书 Bot（@机器人 + 问题）
  - Dify Web UI（内部工具地址）
  - 飞书群内 Slash Command

示例查询：
  "Kafka 消息堆积怎么排查？"
    → 检索相关 Runbook + FAQ
    → Claude 基于上下文生成精准答案
    → 附带原始文档链接
```

**效果预期**：
- 知识检索时间：5-15min → < 1min
- Onboarding 时间：1-2 周 → < 1 周（RAG 辅助答疑）

### 2.3 知识沉淀机制

**强制沉淀触发点**：

| 触发场景 | 沉淀方式 | 沉淀目标 |
|---------|---------|---------|
| P0/P1 故障复盘完成 | 复盘报告 → 归档到 10-SRE/流程模板/postmortems/ | Runbook 更新 |
| 私有化交付踩坑 | 交付复盘 → 归档到 13-私有化交付/playbooks/ | Playbook 更新 |
| 新工具/技术选型 | ADR 文档 → 归档到 02-架构治理/01-架构评审/ | 选型记录 |
| 高频 FAQ 发现 | 飞书 Bot 高频查询 → 运营同步到对应领域 FAQ.md | FAQ 更新 |

**每月知识统计（Dify 数据）**：
- 查询次数 Top 10 问题（高频问题 → 优先完善文档）
- 未命中查询（RAG 找不到答案 → 文档补充待办）

### 2.4 会议纪要自动化（飞书妙记 + Claude）

**会议知识沉淀流程**：

```
飞书会议（录音 + 自动转录）
  ↓ 飞书妙记生成转录文本
  ↓ Claude API 处理：
    - 提取关键决策（WHAT + 为什么）
    - 提取行动项（WHO + WHAT + 截止日期）
    - 生成结构化纪要（100-300 字摘要）
  ↓ 自动发送到飞书群 + 创建飞书文档
  ↓ 重要决策同步到对应领域文档

效率：30-60 分钟手工整理 → 5-10 分钟 AI 辅助 review
```

### 2.5 Onboarding 加速

**新人 Onboarding 知识地图**：

```
Week 1：环境 + 基础
  - 工具账号开通清单（Jira/GitLab/飞书/Grafana）
  - 本工程阅读路径（按角色定制：研发/运维/实施）
  - 飞书 Bot 答疑（不懂的直接问，RAG 7x24h 响应）

Week 2：实践
  - 完成第一个真实任务（有 Mentor 带）
  - 参加一次 OnCall 值班（影子模式）
  - 阅读 Top 3 故障复盘报告

验收标准：能独立完成领域内日常任务
```

## 五、建设路径

### Phase 0（第 1-2 周）：规范化

| 任务 | 具体行动 | 验收标准 | Owner |
|------|---------|---------|-------|
| 文档规范发布 | 发布文档格式规范 + 模板，建立团队共识 | 规范文档经团队 review 通过 | 知识管理 Lead |
| CI 文档检查 | 接入 markdownlint + link-check 到 GitLab CI | MR 中文档格式错误时 CI 提示 | DevOps |
| 知识沉淀触发点 | 将故障复盘/交付复盘的文档归档写入流程 | 下一次复盘有文档归档记录 | 各 Lead |

### Phase 1（第 3-6 周）：RAG 上线

| 任务 | 具体行动 | 验收标准 | Owner |
|------|---------|---------|-------|
| Dify 知识库建立 | Dify 部署后，接入本工程所有 .md 文档 | 飞书 Bot 可回答工程文档相关问题 | AI Infra |
| 飞书 Bot 问答 | 配置飞书 Bot 接入 Dify API | 团队成员测试满意度 ≥ 70% | AI Infra |
| 会议纪要 AI 整理 | 飞书妙记 → Claude API → 结构化纪要模板 | 下次重要会议后 10 分钟内出纪要 | 知识管理 Lead |

### Phase 2（第 7-10 周）：运营持续化

| 任务 | 具体行动 | 验收标准 | Owner |
|------|---------|---------|-------|
| 月度知识健康报告 | Dify 查询统计 → 未命中率 → 文档补充待办 | 每月推送飞书，Top 10 高频问题有对应文档 | 知识管理 Lead |
| Onboarding 路径优化 | 基于新人反馈优化阅读路径 + RAG 答疑 | 新人 1 周完成 Onboarding 验收 | HR + 各 Lead |
| 知识贡献激励 | 每月评选"最佳文档贡献者"，飞书公示 | 月度新增知识条目 ≥ 5 | 知识管理 Lead |
